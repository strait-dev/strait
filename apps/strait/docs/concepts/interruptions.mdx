# Interruptions & Compensation

Strait supports graceful workflow interruption with automatic resource cleanup using the **Saga compensation pattern**. When a workflow is canceled or a step fails, completed steps can trigger compensation actions to undo their effects.

## The Problem

Consider a workflow that:
1. **Allocates a GPU instance** (step 1)
2. **Runs AI inference** (step 2)
3. **Releases the GPU** (step 3)

If step 2 fails or the workflow is canceled, step 3 never runs — you leak the GPU. Compensation solves this.

## How It Works

### Compensation Steps

Each workflow step can define a `compensate_step_ref` — a reference to another step that "undoes" the original step's work:

```yaml
steps:
  - step_ref: allocate-gpu
    job_id: job-allocate-gpu
    compensate_step_ref: release-gpu    # ← runs on cancel/failure

  - step_ref: run-inference
    job_id: job-run-inference
    depends_on: [allocate-gpu]

  - step_ref: release-gpu
    job_id: job-release-gpu
    # This step is triggered automatically as compensation
```

### Cancel Flow

When a workflow is canceled:

1. **Active steps are canceled** — running and waiting steps are marked `canceled`
2. **In-flight runs are interrupted** — job runs transition to `canceling` → `canceled`
3. **Compensation runs in reverse** — completed steps with compensation refs are triggered in reverse chronological order (most recently completed first)
4. **Workflow is marked canceled** — after compensation completes

```
Timeline:
  allocate-gpu ✓  →  run-inference (running)  →  CANCEL
                                                    │
                                                    ├── cancel run-inference
                                                    └── compensate: trigger release-gpu
                                                         (with allocate-gpu's output as payload)
```

### Compensation Payload

Compensation steps receive the **original step's output** as their payload. This lets the compensation step know what to clean up:

```json
// allocate-gpu output (becomes release-gpu input):
{"gpu_id": "gpu-42", "region": "us-east-1"}
```

## Run Status: `canceling`

A new transitional status `canceling` is used to track the cleanup phase:

```
executing → canceling → canceled
waiting   → canceling → canceled
```

This lets you distinguish between:
- A run that was **instantly canceled** (queued → canceled)
- A run that is **cleaning up** (executing → canceling → canceled)

## HTTP Job Cancellation

For HTTP-mode jobs, you can define a `cancel_endpoint_url` on the job. When a run is canceled mid-flight, Strait sends a cancellation webhook to this URL:

```json
{
  "name": "Long Running Task",
  "endpoint_url": "https://api.example.com/run",
  "cancel_endpoint_url": "https://api.example.com/cancel",
  "execution_mode": "http"
}
```

The cancellation webhook includes the run ID and job ID so your service can clean up.

## Sandbox Cancellation

For sandbox-mode jobs, cancellation is automatic. Canceling the gRPC context terminates the Forge sandbox process immediately. The BEAM supervisor handles cleanup.

## Failure Policies + Compensation

Compensation integrates with existing failure policies:

| Policy | Behavior |
|--------|----------|
| `fail_workflow` | Workflow fails → compensation runs for all completed steps with compensation refs |
| `skip_dependents` | Only downstream steps are skipped, no compensation |
| `continue` | Workflow continues, no compensation |

## Compensation Status Tracking

Strait tracks compensation progress on the workflow run with three fields:

| Field | Description |
|-------|-------------|
| `compensation_status` | Current status: `none`, `pending`, `running`, `completed`, `partial`, `failed` |
| `compensation_steps_total` | Number of steps that need compensation |
| `compensation_steps_completed` | Number of steps successfully compensated |

### Status Lifecycle

```
none → pending → running → completed
                        ↘ partial (some steps compensated, some failed)
                        ↘ failed  (all compensation steps failed)
```

- **`partial`** — Some compensation steps succeeded but others failed. Check `compensation_steps_completed` vs `compensation_steps_total` for progress.
- **`failed`** — All compensation steps failed. Requires manual intervention.

### Handling Partial Failures

When compensation partially fails, you can retry only the failed steps:

```bash
# Retry only steps that haven't been compensated yet
POST /v1/workflow-runs/{id}/compensate
```

`RetryFailedCompensation` is **idempotent** — it skips already-completed compensation steps and only retries failed or unprocessed ones.

### CompensationResult

Both cancel and retry endpoints return a `CompensationResult`:

```json
{
  "compensation_result": {
    "compensated": ["step-a", "step-b"],
    "failed": ["step-c"],
    "status": "partial"
  }
}
```

## API

### Cancel a Workflow Run

```
DELETE /v1/workflow-runs/{id}
```

Triggers the full cancel + compensation flow. Returns the updated workflow run and compensation result.

### Retry Failed Compensation

```
POST /v1/workflow-runs/{id}/compensate
```

Retries compensation for steps that failed during a previous cancel attempt. Only available when `compensation_status` is `partial` or `failed`.
