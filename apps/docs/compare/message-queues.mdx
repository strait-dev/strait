A message queue moves messages from producers to consumers. Strait coordinates work from trigger to result.

Both are useful. The question is how much of the job lifecycle you want the system to own.

Strait's internal queue uses `pgque`, with a vendored and modified SQL snapshot of [PgQue](https://github.com/NikolayS/pgque) as its PostgreSQL ready-event log. PgQue supplies the bloat-resistant event-log primitive. Strait adds the job orchestration layer around it: run state, retries, workflows, workers, APIs, and observability.

## Use a Queue When

A queue is usually enough when:

- Each message maps to one simple unit of work.
- You already own retry, timeout, and `dead_letter` handling.
- You do not need workflow steps, approvals, or event waits.
- You do not need run history for customers or your team.
- You want a low-level building block for your own job system.

## Use Strait When

Strait is a better fit when:

- You want every job execution tracked as a run.
- You need retries, timeouts, replay, and `dead_letter` handling.
- You need scheduled jobs.
- You need workflows with dependencies between steps.
- You need approvals, event waits, or long-running processes.
- You want job operations exposed through an API, SDKs, and dashboard.

## Practical Difference

With a queue, your application usually owns:

- retry rules
- attempt history
- idempotency
- replay tooling
- dashboards
- workflow dependencies
- scheduling
- access control

With Strait, those are part of the product. Your code focuses on the work itself.

## Example

For a single welcome email, a queue may be enough.

For customer onboarding, Strait is usually a better fit:

1. Create account.
2. Send welcome email.
3. Wait for billing setup.
4. Provision workspace.
5. Notify the team.
6. Retry only the failed step.

That process has state, dependencies, retries, and recovery, which is more than message delivery on its own.

## How to Decide

Ask one question:

> If this fails at 2 AM, what do I need to know and what do I need to retry?

If the answer is just "process the message again", a queue may be enough.

If the answer includes "which step failed, with what payload, after how many attempts, and what can I safely replay", use Strait.
