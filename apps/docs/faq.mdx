Questions about pricing and plans live in the [Billing FAQ](/billing/faq). This page covers running and operating Strait.

<AccordionGroup>
  <Accordion title="Does Strait run my job code?">
    No. Your code runs on your own infrastructure. Strait orchestrates the work and reaches your code in one of two ways: it POSTs to an HTTP endpoint you expose, or it streams runs to a long-lived worker process that connects over gRPC. See [Execution Modes](/concepts/execution-modes).
  </Accordion>
  <Accordion title="What infrastructure does Strait require?">
    Strait requires PostgreSQL, Redis, and Sequin. PostgreSQL is the source of truth and hosts Strait's PgQue-backed ready-event log, Redis powers rate limiting and the worker transport plane, and Sequin streams database changes through CDC. Startup fails if any of the three is missing or unreachable. See [Self-Hosting](/guides/self-hosting).
  </Accordion>
  <Accordion title="Do I need a message broker like RabbitMQ, Kafka, or SQS?">
    No. Strait uses a vendored and modified SQL snapshot of [PgQue](https://github.com/NikolayS/pgque) in PostgreSQL as its ready-event log, so no separate broker is needed. See [Architecture](/architecture).
  </Accordion>
  <Accordion title="HTTP mode or worker mode?">
    Use HTTP mode when your code can expose an endpoint Strait can reach. Use worker mode when your code runs somewhere Strait cannot call directly, such as a private network, or when you want workers to pull work over a single outbound gRPC connection. See [Execution Modes](/concepts/execution-modes) and [Run Private Workers](/guides/private-workers).
  </Accordion>
  <Accordion title="How do retries interact with idempotency keys?">
    Retries re-run the same run, so your endpoint should be safe to call more than once. Idempotency keys apply to trigger requests: they stop a retried trigger from creating duplicate runs. The two solve different problems, and production jobs usually want both. See [Idempotency](/guides/idempotency) and [Retry Strategies](/concepts/retry-strategies).
  </Accordion>
  <Accordion title="What is the difference between the community and cloud editions?">
    The community edition is open source and self-hosted, with the full orchestration feature set. The cloud edition is the hosted service at strait.dev, adding managed infrastructure, usage metering, and billing. Your job code runs on your own infrastructure either way. The edition is selected at compile time, not by an environment variable.
  </Accordion>
  <Accordion title="Can I run only the API or only workers?">
    Yes. The binary runs in `api`, `worker`, or `all` mode. Start with `all` for small deployments, then split into separate `api` and `worker` processes as traffic grows. See [Deployment](/guides/deployment).
  </Accordion>
  <Accordion title="What versions of PostgreSQL, Redis, and Sequin are supported?">
    PostgreSQL 18, Redis 8, and Sequin v0.14.6 are the tested versions and what the self-host stack ships. PostgreSQL must have logical replication enabled for Sequin. See [Self-Hosting](/guides/self-hosting).
  </Accordion>
  <Accordion title="How are secrets stored?">
    Secrets are encrypted at rest using the keys you configure (`ENCRYPTION_KEY`, `SECRET_ENCRYPTION_KEY`). Use unique values per environment and store them in a secret manager. See [Environment Variables](/configuration/environment-variables).
  </Accordion>
  <Accordion title="How do I upgrade?">
    Pull the new image and restart. Migrations run automatically on startup. See [Self-Hosting](/guides/self-hosting) for the self-host flow and [Deployment](/guides/deployment) for production.
  </Accordion>
  <Accordion title="Is there an SDK?">
    Yes, for TypeScript, Python, Go, Ruby, and Rust. You can also call the REST API directly. See the [SDKs overview](/sdks/overview).
  </Accordion>
  <Accordion title="What happens to a run if a worker dies mid-execution?">
    The run is reaped once the worker stops heartbeating and returned to the queue, then retried according to its retry policy. See [Troubleshooting](/guides/troubleshooting) and [Retry Strategies](/concepts/retry-strategies).
  </Accordion>
</AccordionGroup>
