Webhook subscriptions let you receive notifications for specific event types across your project, not just individual jobs. Instead of setting a `webhook_url` on each job, you register a subscription once and get notified for every matching event project-wide.

## How It Works

You register subscriptions per project via the API. When a matching event occurs (a run completes, fails, or times out), Strait sends a signed HTTP POST to each active subscription URL.

Deliveries are persisted before dispatch, so notifications survive restarts. Strait retries with exponential backoff and keeps permanently failing deliveries available for review and replay.

## Creating a Subscription

Cloud plans cap active webhook endpoints across the organization. Free includes zero webhook endpoints, Starter includes three, Pro includes ten, Scale includes twenty-five, and Business is unlimited. Webhook endpoint add-ons are not sold at launch.

The signing secret is generated by Strait and returned in the response **exactly once**. Capture it during creation. There is no way to retrieve the plaintext value later. To replace it, use the rotate-secret endpoint.

```bash
curl -X POST https://your-strait-instance/v1/webhooks/subscriptions \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_abc123",
    "webhook_url": "https://your-service.com/webhooks/strait",
    "event_types": ["run.completed", "run.failed", "workflow.completed"]
  }'
```

Response:
```json
{
  "subscription": {
    "id": "wsub_123",
    "project_id": "proj_abc123",
    "webhook_url": "https://your-service.com/webhooks/strait",
    "event_types": ["run.completed", "run.failed", "workflow.completed"],
    "active": true,
    "created_at": "2026-03-15T10:00:00Z"
  },
  "signing_secret": "whsec_abc123..."
}
```

A `secret` field in the request body is ignored. Server-side generation guarantees a sufficiently strong, randomly distributed signing key. Existing subscriptions are unaffected; rotate them with `POST /v1/webhooks/subscriptions/{id}/rotate-secret` if you want to migrate to a freshly generated secret.

## Event Types

| Event Type | Description |
| :--- | :--- |
| `run.completed` | A job run finishes successfully. |
| `run.failed` | A job run fails with an error. |
| `run.timed_out` | A job run exceeds its timeout limit. |
| `run.canceled` | A job run is manually canceled. |
| `workflow.completed` | A workflow finishes successfully. |
| `workflow.failed` | A workflow fails. |
| `slo.budget_warning` | A project approaches an SLO error-budget threshold. |
| `billing.cap_warning` | An organization reaches 80 percent of its monthly spending cap. |
| `billing.cap_reached` | An organization reaches 100 percent of its monthly spending cap. |
| `billing.cap_disabled` | An organization removes its monthly spending cap. |
| `billing.overage_disabled` | An organization disables paid-plan overage. |
| `billing.suspended` | An organization is suspended after failed-payment dunning. |
| `billing.delinquent` | A suspended organization moves to delinquent state. |
| `billing.payment_succeeded` | A payment succeeds after an invoice or dunning event. |
| `schedule.suspended` | A schedule is paused because a billing cap or plan change blocks new dispatch. |
| `workflow.registration_rejected` | A workflow registration is rejected by a launch plan gate. |
| `sla.credit_issued` | An SLA service credit is recorded for an eligible organization. |

## Circuit Breaker

If your endpoint returns 5xx errors on 5 consecutive deliveries, Strait opens the circuit and stops sending for 60 seconds. After the cooldown, Strait retries one delivery. If it succeeds, normal delivery resumes. If it fails again, the circuit stays open for another cooldown period.

This prevents Strait from overwhelming a failing endpoint with retries.

## Security

All deliveries are signed with HMAC-SHA256 using the subscription's secret, so your service can verify that the request came from Strait.

The signature is included in the `X-Strait-Signature` header. Receivers should also validate the `X-Strait-Timestamp` to protect against replay attacks. See [Webhooks](/concepts/webhooks) for signature verification steps and code examples.
