Webhook subscriptions let you register URLs that receive notifications when runs complete, fail, or time out. Unlike per-job `webhook_url` configuration, subscriptions apply project-wide and support filtering by event type.

## How It Works

You register subscriptions per project via the API. When a matching event occurs (e.g., a run completes or fails), Strait sends a signed HTTP POST to each active subscription URL.

Deliveries are persisted to the database before dispatch, so notifications survive worker restarts. Strait retries with exponential backoff and stores permanently failing deliveries in a dead letter queue (a holding area for undeliverable messages you can investigate later).

## Event Types

| Event Type | Description |
| :--- | :--- |
| `run.completed` | Triggered when a job run finishes successfully. |
| `run.failed` | Triggered when a job run fails with an error. |
| `run.timed_out` | Triggered when a job run exceeds its timeout limit. |
| `run.canceled` | Triggered when a job run is manually canceled. |
| `workflow.completed` | A workflow finishes successfully. |
| `workflow.failed` | A workflow fails. |

## Configuration


Strait uses a circuit breaker (a pattern that temporarily stops calling a failing endpoint to let it recover) to protect subscription endpoints. If a target URL consistently returns errors, Strait temporarily disables delivery to that endpoint.

## Security

All deliveries are signed with HMAC-SHA256 using the subscription's secret, so your service can verify that the request came from Strait.

The signature is included in the `X-Strait-Signature` header. Receivers should also validate the `X-Strait-Timestamp` to protect against replay attacks. See the webhooks concept doc for detailed signature verification steps and implementation examples.
