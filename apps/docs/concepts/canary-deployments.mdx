Canary deployments let you roll out a new job version to a small percentage of traffic before promoting it to 100%. If the new version has problems, you catch them before they affect all your users.

## When to Use It

You've updated your order-processing job to use a new API. Instead of switching all traffic at once and hoping it works, you send 10% of new runs to the updated version. If error rates stay low, you promote to 25%, then 50%, then 100%. If errors spike at any step, Strait rolls back automatically.

## How It Works

When a canary deployment is active, Strait routes each new run to either the baseline or canary version based on the traffic split. Runs that are already in progress always finish on the version they started with.

At each check interval, Strait evaluates the canary version's health:

- **Promote**: Error rate and P99 latency are both below thresholds → advance to the next traffic step.
- **Rollback**: Either metric exceeds its threshold → revert to 0% canary traffic.
- **Hold**: Fewer than 5 runs on the canary version → wait for more data before deciding.

## Creating a Canary

```bash
curl -X POST https://your-strait-instance/api/v1/jobs/{jobId}/canary \
  -H "Authorization: Bearer $API_KEY" \
  -d '{
    "target_version": 2,
    "traffic_percentage": 10,
    "auto_promote": {
      "enabled": true,
      "steps": [10, 25, 50, 100],
      "interval": "15m",
      "failure_rate_threshold": 5.0,
      "latency_p99_threshold": "10s"
    }
  }'
```

This creates a canary that starts at 10% traffic. Every 15 minutes, Strait checks health and promotes to the next step if the canary version has less than 5% failure rate and P99 latency under 10 seconds.

## Promoting Manually

If `auto_promote` is disabled, promote the canary by updating the traffic percentage:

```bash
curl -X PATCH https://your-strait-instance/api/v1/jobs/{jobId}/canary \
  -H "Authorization: Bearer $API_KEY" \
  -d '{ "traffic_percentage": 100 }'
```

Setting traffic to 100% completes the canary and makes the target version the new default.

## Rolling Back

If the canary version is causing problems, roll back:

```bash
curl -X PATCH https://your-strait-instance/api/v1/jobs/{jobId}/canary \
  -H "Authorization: Bearer $API_KEY" \
  -d '{ "traffic_percentage": 0 }'
```

All new runs go back to the baseline version. In-flight canary runs finish on their version.

## Health Thresholds

| Setting | What it controls |
| :--- | :--- |
| `failure_rate_threshold` | Maximum error rate (%) before rollback triggers. |
| `latency_p99_threshold` | Maximum P99 latency (the worst-case response time for 99% of runs) before rollback. |
| `interval` | How often Strait checks canary health and decides to promote, hold, or rollback. |
| `steps` | Traffic percentages to walk through during auto-promotion. |
