Canary deployments let you gradually shift new workflow runs from version N to version N+1, with automatic rollback if failure rates spike.

## Traffic Routing

When a canary deployment is active, new workflow runs are routed to the source or target version using weighted random selection. In-flight runs always complete on the version they started with (guaranteed by workflow snapshots).

| Traffic % | Behavior |
|-----------|----------|
| 0% | All runs use source version |
| 10% | ~10% of new runs use target version |
| 50% | Even split between versions |
| 100% | All runs use target version |

## Auto-Promote Configuration

```json
{
  "target_version": 2,
  "traffic_percentage": 10,
  "auto_promote": {
    "enabled": true,
    "steps": [10, 25, 50, 100],
    "interval": "15m",
    "failure_rate_threshold": 5.0,
    "latency_p99_threshold": "10s"
  }
}
```

At each interval, the health monitor compares the target version's metrics against thresholds:

- **Promote**: If failure rate and P99 latency (the worst-case response time for 99% of runs) are both below thresholds, advance to the next traffic step
- **Rollback**: If either metric exceeds its threshold, roll back to 0% target traffic
- **Hold**: If fewer than 5 runs exist for the target version, wait for more data

## Health Evaluation

At each interval, the health monitor makes a three-way decision:

1. **Insufficient data** (fewer than 5 target runs) -- hold, wait for more data
2. **Failure rate or P99 latency exceeds threshold** -- roll back
3. **All checks pass** -- promote to next traffic step

## Canary States

| State | Meaning |
|-------|---------|
| `active` | Traffic is split between versions |
| `promoting` | Advancing to the next traffic step |
| `rolling_back` | Reverting to the source version |
| `completed` | Target version at 100%, canary finished |
| `rolled_back` | Fully reverted to source version |
