Canary deployments let you roll out a new workflow version to a small percentage of traffic before promoting it to 100%. If the new version has problems, you catch them before they affect every run.

A canary always compares two versions of the same workflow: a `source_version` that currently serves traffic and a `target_version` you want to roll out. Both are the integer [version](/concepts/versioning) counters Strait assigns each time you publish a workflow.

## When to Use It

You've updated your order-fulfillment workflow to call a new payment provider. Instead of switching all traffic at once, you send 10% of new runs to the updated version. If error rates stay low, you promote to 25%, then 50%, then 100%. If errors spike at any step, Strait rolls back automatically.

## How It Works

When a canary deployment is active, Strait routes each new workflow run to either the source or target version based on the traffic split. Runs already in progress always finish on the version they started with.

At each check interval, Strait evaluates the target version's health:

- **Promote**: Error rate and P99 latency are both below thresholds, so it advances to the next traffic step.
- **Rollback**: Either metric exceeds its threshold, so it reverts to 0% canary traffic.
- **Hold**: Fewer than 5 runs on the target version, so it waits for more data before deciding.

## Creating a Canary

```bash
curl -X POST https://api.strait.dev/v1/canary-deployments \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "workflow_id": "order-fulfillment",
    "source_version": 1,
    "target_version": 2,
    "traffic_pct": 10,
    "auto_promote": {
      "enabled": true,
      "steps": [10, 25, 50, 100],
      "interval": 900000000000,
      "failure_rate_threshold": 5.0,
      "latency_p99_threshold": 10000000000
    }
  }'
```

This creates a canary that starts at 10% traffic. Duration fields are encoded as nanoseconds. Every 15 minutes, Strait checks health and promotes to the next step when the target version has less than 5% failure rate and P99 latency under 10 seconds.

## Promoting Manually

If `auto_promote` is disabled, promote the canary by updating the traffic percentage on the workflow:

```bash
curl -X PATCH https://api.strait.dev/v1/workflows/order-fulfillment/canary \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "traffic_pct": 100 }'
```

Setting traffic to 100% completes the canary and makes the target version the new default.

## Rolling Back

If the target version is causing problems, roll back:

```bash
curl -X POST https://api.strait.dev/v1/workflows/order-fulfillment/canary/rollback \
  -H "Authorization: Bearer $API_KEY"
```

All new runs go back to the source version. In-flight target runs finish on their version. Check the current split at any time with `GET /v1/workflows/{workflowID}/canary`.

## Health Thresholds

| Setting | What it controls |
| :--- | :--- |
| `failure_rate_threshold` | Maximum error rate (%) before rollback triggers. |
| `latency_p99_threshold` | Maximum P99 latency in nanoseconds before rollback. |
| `interval` | How often Strait checks canary health and decides to promote, hold, or rollback, encoded as nanoseconds. |
| `steps` | Traffic percentages to walk through during auto-promotion. |
