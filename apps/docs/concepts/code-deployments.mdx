Code deployments let you ship source code to Strait instead of pointing a job at an HTTP endpoint or pre-built image. Upload a tarball of your code, Strait builds a container image with BuildKit, pushes it to your registry, and runs it.

## When to Use It

You want Strait to own the full build-and-run lifecycle. Instead of building Docker images in your own CI pipeline and configuring a managed job to pull them, you upload source code and Strait handles the rest.

| Scenario | Why code deployments? |
| :--- | :--- |
| No CI pipeline | You don't have Docker builds set up. Ship code, Strait builds it. |
| Rapid iteration | Push code changes without rebuilding images locally. |
| Standardized builds | Every deployment uses the same base images and build process. |

## How It Works

1. **Upload code.** You send a tarball (`.tar.gz`) of your source code. Strait stores it in the configured object store (S3 or R2).
2. **Build image.** BuildKit builds a container image from the tarball using a base image matched to your runtime.
3. **Push to registry.** The built image is pushed to your container registry with a digest for pinning.
4. **Run.** The job uses the built image for execution, just like a managed job with a pre-built image.

## Supported Runtimes

| Runtime | Base Image | Value |
| :--- | :--- | :--- |
| Python | `python:3.12-slim` | `python` |
| TypeScript | `node:22-slim` | `typescript` |
| Ruby | `ruby:3.3-slim` | `ruby` |
| Rust | `rust:1.83-slim` | `rust` |
| Go | `golang:1.24-alpine` | `go` |

Set the `runtime` field when creating a deployment. Strait selects the matching base image and knows how to build your code.

## Deploying Code

### Create a deployment

Upload your tarball and create a code deployment:

```bash
curl -X POST https://your-strait-instance/api/v1/code-deployments \
  -H "Authorization: Bearer $API_KEY" \
  -d '{
    "project_id": "proj_abc123",
    "job_id": "job_xyz",
    "source_type": "code",
    "runtime": "python"
  }'
```

The response includes an `id` and a pre-signed URL for uploading your tarball.

### Confirm the deployment

After the tarball is uploaded, confirm the deployment to start the build:

```bash
curl -X POST https://your-strait-instance/api/v1/code-deployments/{id}/confirm \
  -H "Authorization: Bearer $API_KEY"
```

Strait kicks off a BuildKit build. The deployment status moves through `pending` → `building` → `ready` (or `failed` / `timed_out`).

### Check status

```bash
curl https://your-strait-instance/api/v1/code-deployments/{id} \
  -H "Authorization: Bearer $API_KEY"
```

When `status` is `ready`, the image is built and the job can run it. The response includes `image_uri`, `image_digest`, and `build_logs`.

## Deployment Versions

Deployment versions control how traffic shifts when you deploy new code. Each version wraps a code deployment and adds a rollout strategy.

### Create a version

```bash
curl -X POST https://your-strait-instance/api/v1/deployment-versions \
  -H "Authorization: Bearer $API_KEY" \
  -d '{
    "deployment_id": "dep_abc123",
    "job_id": "job_xyz",
    "project_id": "proj_abc123",
    "strategy": "canary",
    "canary_percent": 10
  }'
```

| Strategy | Behavior |
| :--- | :--- |
| `all_at_once` | All traffic moves to the new version immediately. |
| `canary` | Only `canary_percent` of traffic goes to the new version until promoted. |

### Finalize and promote

Finalize the version after the deployment is ready, then promote it to take full traffic:

```bash
# Finalize
curl -X POST https://your-strait-instance/api/v1/deployment-versions/{id}/finalize \
  -H "Authorization: Bearer $API_KEY"

# Promote to 100% traffic
curl -X POST https://your-strait-instance/api/v1/deployment-versions/{id}/promote \
  -H "Authorization: Bearer $API_KEY"
```

A promoted version moves to `active` status. For canary strategies, promotion shifts all traffic to the new version.

## Rollback

Roll back a deployment version to revert traffic to the previous version:

```bash
curl -X POST https://your-strait-instance/api/v1/deployment-versions/{id}/rollback \
  -H "Authorization: Bearer $API_KEY"
```

The version status changes to `rolled_back`. All new runs use the previous active version. You can also roll back at the deployment level:

```bash
curl -X POST https://your-strait-instance/api/v1/code-deployments/{id}/rollback \
  -H "Authorization: Bearer $API_KEY"
```

## Infrastructure Configuration

Code deployments require BuildKit, a container registry, and an object store. Configure these with environment variables:

| Variable | Description |
| :--- | :--- |
| `BUILDKIT_ADDRESS` | BuildKit daemon address for image builds. |
| `OBJECT_STORE_TYPE` | Object store backend (`s3` or `r2`). |
| `OBJECT_STORE_BUCKET` | Bucket name for tarball storage. |
| `OBJECT_STORE_ENDPOINT` | Object store endpoint URL. |
| `OBJECT_STORE_REGION` | Object store region. |
| `CONTAINER_REGISTRY_TYPE` | Registry type (e.g., `docker`). |
| `CONTAINER_REGISTRY_URL` | Registry URL for pushing built images. |
| `CONTAINER_REGISTRY_USER` | Registry auth username. |
| `CONTAINER_REGISTRY_PASS` | Registry auth password. |
