Managed execution runs your jobs inside containers that Strait provisions and manages. Instead of calling an HTTP endpoint, Strait creates a Docker or Kubernetes container, runs your code, and collects the result.

## When to Use It

| Use Case | Why Managed? |
| :--- | :--- |
| Long-running jobs (minutes to hours) | No HTTP timeout constraints; the container runs until the process exits. |
| Custom runtimes | Bring any Docker image — Python ML pipelines, Rust binaries, Node scripts. |
| Isolated execution | Each run gets its own ephemeral container with no shared state. |
| Checkpoint / resume | Pause a running container and resume it later with preserved state. |

### Comparison with HTTP Execution

| Aspect | HTTP (`http`) | Managed (`managed`) |
| :--- | :--- | :--- |
| Dispatch mechanism | POST to `endpoint_url` | Provision and start container |
| Result delivery | HTTP response body | SDK callback or exit code |
| Timeout model | HTTP request timeout | Container process lifetime |
| Cold start | None (endpoint already running) | 5–15s (cold) or 1–2s (warm pool) |
| Infrastructure cost | You manage servers | Per-second compute billing by preset |
| Pause / resume | Not supported | Supported (container stop / start) |

## Container Runtimes

Strait supports two container runtimes, set via the `COMPUTE_RUNTIME` environment variable:

| Runtime | Value | Best For |
| :--- | :--- | :--- |
| Docker | `docker` | Single-server deployments and local development |
| Kubernetes | `k8s` | Production workloads with orchestration and scaling |

```bash
# Local development
COMPUTE_RUNTIME=docker

# Production
COMPUTE_RUNTIME=k8s
K8S_NAMESPACE=strait-jobs
```

## How It Works

When Strait dequeues a run for a managed job:

1. **Create container.** Strait provisions a new container with your image, or acquires one from the warm pool.
2. **Inject environment variables.** Run context — payload, API URL, SDK token, project secrets — is injected as env vars.
3. **Wait for exit.** The container runs until the process exits. Exit code `0` means success; non-zero triggers a retry.
4. **Record result.** Wall-clock duration and the SDK completion callback (or exit code) are recorded.

```json
{
  "name": "train-model",
  "slug": "train-model",
  "execution_mode": "managed",
  "machine_preset": "medium-1x",
  "image": "my-registry.example.com/trainer:latest"
}
```

## Warm Pool

The warm pool reduces cold start latency from 5–15 seconds down to 1–2 seconds by reusing stopped containers.

After a clean exit (exit code `0`), the container is returned to the pool instead of being destroyed. The pool is keyed by `image:region`, so containers are only reused for the same image. When a container is reused, Strait injects fresh environment variables — each reuse gets a clean run context.

| Parameter | Default | Description |
| :--- | :--- | :--- |
| `WARM_POOL_ENABLED` | `true` | Enable or disable the warm pool. |
| `WARM_POOL_MAX_PER_JOB` | `3` | Maximum stopped containers per `image:region` key. |
| `WARM_POOL_TTL` | `5m` | How long idle containers are kept before cleanup. |

## Pause and Resume

Managed runs can be paused and resumed. Pausing stops the container gracefully and preserves its reference. Resuming restarts the same container with fresh environment variables. If the container has been cleaned up, Strait falls back to a new one.

Workflows can also pause and resume containers across step boundaries.

## Compute Presets

Compute presets define the CPU and memory allocation for managed runs. Set the preset on the job via the `machine_preset` field.

| Preset | CPUs | Memory |
| :--- | :--- | :--- |
| `micro` | 1 shared | 256 MB |
| `small-1x` | 1 shared | 512 MB |
| `medium-1x` | 2 dedicated | 4096 MB |
| `large-1x` | 4 dedicated | 8192 MB |
| `large-2x` | 8 dedicated | 16384 MB |

If a container runs out of memory, Strait automatically upgrades the preset and retries, stepping up through the sizes above.

## Cost Tracking

Managed execution is billed per second based on the compute preset. Cost is calculated from when the container starts to when it exits.

Projects can set a daily compute cost limit. A warning fires at 80% of the budget. Once exceeded, new runs are rejected until the next day.

## Configuration Reference

| Variable | Default | Description |
| :--- | :--- | :--- |
| `COMPUTE_RUNTIME` | `none` | Compute backend: `none`, `docker`, or `k8s`. |
| `K8S_NAMESPACE` | — | Kubernetes namespace (required for `k8s`). |
| `MAX_CONCURRENT_MACHINES` | `10` | Maximum containers running simultaneously. |
| `DEFAULT_REGION` | — | Default region for container placement. |
| `REQUIRE_IMAGE_DIGEST` | `false` | Require image digests instead of tags. |
| `EXTERNAL_API_URL` | — | Public API URL passed to containers for SDK callbacks. |
| `WARM_POOL_ENABLED` | `true` | Enable the warm container pool. |
| `WARM_POOL_MAX_PER_JOB` | `3` | Maximum pooled containers per `image:region` key. |
| `WARM_POOL_TTL` | `5m` | TTL for idle warm containers. |
