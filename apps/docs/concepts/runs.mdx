A run is one execution of a [job](/concepts/jobs). It records the payload, status, attempts, timing, result, and error details.

Runs are what you watch while work is happening: a job defines what can run, and the run shows what actually happened.

Cloud run history is retained according to the organization's plan: Free retains 7 days, Starter 14 days, Pro 30 days, Scale 60 days, Business 90 days, and Enterprise is contract-defined. The extended run history add-on adds 30 days on eligible plans.

## Run States

Most runs follow a simple path:

```text
queued -> executing -> completed
```

When something fails, Strait retries according to the job policy. If the run still cannot complete, it moves to `dead_letter`.

| State | Meaning |
|---|---|
| `delayed` | The run is scheduled for a future time. |
| `queued` | The run is waiting to start. |
| `dequeued` | A worker claimed the run and is preparing execution. |
| `executing` | Strait is calling your endpoint or connected worker. |
| `waiting` | The run is paused on a durable wait, such as an external event or workflow dependency. |
| `completed` | The work finished successfully. |
| `failed` | The endpoint returned an error and no retry remains. |
| `timed_out` | The endpoint did not finish before `timeout_secs`. |
| `crashed` | The worker stopped while executing the run. |
| `system_failed` | Strait failed the run because of an internal system error. |
| `canceled` | The run was canceled before completion. |
| `expired` | The run was not picked up before its TTL. |
| `dead_letter` | Retries were exhausted. The run can be inspected and replayed. |
| `replay_staged` | A dead-letter replay is being staged before a new queued attempt is created. |
| `paused` | The run is paused by an operator or policy. |

## Trigger a Run

```bash
curl -X POST https://api.strait.dev/v1/jobs/send-welcome-email/trigger \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "payload": { "user_id": "user_abc123" },
    "metadata": { "source": "signup" }
  }'
```

Response:

```json
{
  "id": "run_abc123",
  "job_id": "job_abc123",
  "status": "queued",
  "attempt": 1
}
```

## Check Status

```bash
curl https://api.strait.dev/v1/runs/run_abc123 \
  -H "Authorization: Bearer $STRAIT_API_KEY"
```

Completed runs include a `result`. Failed and timed-out runs include error details.

## Watch Live Updates

Use server-sent events when you want to wait for a run in a script or update a dashboard in real time.

```bash
curl -N https://api.strait.dev/v1/runs/run_abc123/stream \
  -H "Authorization: Bearer $STRAIT_API_KEY"
```

See [Real-Time Streaming](/concepts/real-time) for browser authentication and event formats.

## Failed Runs

When a run fails, check:

| Field | What to look for |
|---|---|
| `status` | Whether the run is retrying, terminal, or in `dead_letter`. |
| `attempt` | How many attempts have already run. |
| `error` | The last error returned by your endpoint or worker. |
| `execution_trace` | Timing information for the attempt. |
| `metadata` and `tags` | Your own correlation fields. |

If the run is in `dead_letter`, fix the underlying problem first, then replay the run.

See [Handle Failed Runs](/guides/handle-failed-runs) for a practical recovery flow.

## Replay

Replay creates another attempt at the same work.

```bash
curl -X POST https://api.strait.dev/v1/runs/run_abc123/replay \
  -H "Authorization: Bearer $STRAIT_API_KEY"
```

Use replay after you have fixed the endpoint, restored a dependency, or corrected bad configuration.

## Metadata and Tags

Use metadata for correlation fields that help you understand where a run came from.

```json
{
  "metadata": {
    "source": "signup",
    "request_id": "req_123"
  }
}
```

Use tags when you want to filter and group runs.

```json
{
  "tags": {
    "tenant": "acme",
    "region": "us-east"
  }
}
```

## Per-Run Overrides

You can override selected job defaults when triggering a run.

```bash
curl -X POST https://api.strait.dev/v1/jobs/import-users/trigger \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "payload": { "batch_id": "batch_123" },
    "timeout_secs_override": 600,
    "max_attempts_override": 5
  }'
```

Overrides apply only to that run. The job definition is unchanged.
