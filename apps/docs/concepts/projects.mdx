## What is a Project?

A project is the top-level container in Strait. Every job, workflow, run, environment, secret, and API key belongs to exactly one project. Use projects to separate concerns: one per service, team, or deployment stage.

Projects provide:

- **Isolation**: jobs and workflows in one project cannot reference resources in another.
- **Quotas**: per-project limits on concurrency, cost, and rate of API calls.
- **Scoped API keys**: each key grants access to a single project with fine-grained permission scopes.

---

## Creating a Project

Creating a project is an org-level operation, so it authenticates with the internal management secret rather than a project-scoped API key. An API key only exists once a project does. The `id` is caller-supplied and must be unique within the org.

```bash
curl -X POST https://api.strait.dev/v1/projects \
  -H "X-Internal-Secret: $INTERNAL_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "payments",
    "org_id": "org_acme",
    "name": "Payments Service"
  }'
```

Response:

```json
{
  "id": "payments",
  "org_id": "org_acme",
  "name": "Payments Service",
  "created_at": "2026-06-01T12:00:00Z",
  "updated_at": "2026-06-01T12:00:00Z"
}
```

A project has five fields: `id`, `org_id`, `name`, `created_at`, and `updated_at`.

On Strait Cloud, the dashboard creates projects for you. Self-hosted operators use the internal secret directly, as shown above.

Cloud plans cap projects per organization: Free includes one project, Starter includes three, Pro includes ten, Scale includes fifty, and Business is unlimited. Project add-ons are not sold at launch.

## Organizations

Projects can belong to an organization via the `org_id` field. An organization is a string identifier that groups projects for billing and cross-project queries. Set `org_id` when creating a project to associate it with an organization.

API keys can also carry an `org_id`, granting access across all projects in that organization instead of a single project.

## Quotas and Limits

Every project has a quota that caps concurrency, throughput, and cost. View your project's current limits:

```bash
curl https://api.strait.dev/v1/projects/payments/settings \
  -H "Authorization: Bearer YOUR_API_KEY"
```

Key quota fields:

| Field | Description |
|---|---|
| `max_queued_runs` | Maximum runs waiting in the queue. |
| `max_executing_runs` | Maximum runs executing concurrently. |
| `max_jobs` | Maximum job definitions in the project. |
| `max_workflows` | Maximum workflow definitions in the project. |
| `rate_limit_requests` | Maximum API requests per second. |

Cost ceiling fields are internal and not configurable in the launch product. Quotas are tied to your `plan_tier`. When a limit is reached, Strait rejects the operation with a clear error rather than silently dropping work. A trigger that would exceed the queued or executing-run quota returns `429` with the `enqueue_throttled` code:

```json
{
  "error": {
    "code": "enqueue_throttled",
    "message": "project queued quota exceeded"
  },
  "request_id": "req_abc123"
}
```

Back off and retry once capacity frees up, or raise the limit. See the [API error reference](/api-reference/overview) for the full code list.

## API Key Scopes

Each API key is scoped to a project and carries a set of permission scopes. Assign only the scopes a service needs. A worker that triggers jobs does not need `secrets:write`.

| Category | Scopes |
|---|---|
| **Jobs** | `jobs:read`, `jobs:write`, `jobs:trigger` |
| **Runs** | `runs:read`, `runs:write` |
| **Workflows** | `workflows:read`, `workflows:write`, `workflows:trigger` |
| **Secrets** | `secrets:read`, `secrets:write` |
| **API Keys** | `api-keys:manage` |
| **Webhooks** | `webhooks:read`, `webhooks:write` |
| **Dead-letter runs** | `dlq:read`, `dlq:replay`, `dlq:purge` |
| **Outbox** | `outbox:read`, `outbox:retry`, `outbox:purge` |
| **Stats** | `stats:read` |
| **RBAC** | `rbac:manage` |
| **Workers** | `workers:connect` |
| **Projects** | `projects:read`, `projects:write`, `projects:manage` |

### Creating and Rotating Keys

Create a key scoped to a project. `scopes` is required and must list at least one permission. The plaintext `key` is returned once, at creation, and never again, so store it immediately.

```bash
curl -X POST https://api.strait.dev/v1/api-keys \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "payments",
    "name": "ci-deploy",
    "scopes": ["jobs:trigger", "runs:read"]
  }'
```

The very first key for a fresh project is minted with the internal secret instead, since no key exists yet. Swap `-H "Authorization: Bearer ..."` for `-H "X-Internal-Secret: $INTERNAL_SECRET"`.

Rotate a key when it leaks or on a schedule. Rotation issues a new key and keeps the old one valid for a grace period (default 60 minutes, max 10080) so callers can cut over without downtime:

```bash
curl -X POST https://api.strait.dev/v1/api-keys/{keyID}/rotate \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "grace_period_minutes": 120 }'
```

Revoke a key immediately with `DELETE /v1/api-keys/{keyID}`. See [Authentication](/guides/authentication) for the full key lifecycle.
