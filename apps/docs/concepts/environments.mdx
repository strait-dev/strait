Environments let you run the same job against different endpoints (staging vs production, US vs EU, or any variation your deployment needs). Instead of duplicating job definitions, you create environments with different configuration and point jobs at them.

Cloud plans cap the number of active environments in an organization. Free and Starter include one active environment, Pro includes three, Scale includes ten, and Business is unlimited. The additional environments add-on is available on Pro and Scale.

## When to Use It

You have an `order-processing` job that calls your API. In development, it should hit `staging.api.myapp.com`. In production, it should hit `api.myapp.com`. Without environments, you'd need two separate job definitions. With environments, you create one job and bind it to different environments.

## How It Works

An environment is a named set of key-value variables attached to a project. When you set `environment_id` on a job, Strait injects that environment's variables at execution time. If the environment defines an `ENDPOINT_URL` variable, Strait routes the job to that URL instead of the job's default `endpoint_url`.

## Example: Staging vs Production

On plans with room for more than one active environment, create two environments:

```bash
# Create staging environment
curl -X POST https://api.strait.dev/v1/environments \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "my-project",
    "name": "Staging",
    "slug": "staging",
    "variables": {
      "ENDPOINT_URL": "https://staging.api.myapp.com/jobs"
    }
  }'

# Create production environment
curl -X POST https://api.strait.dev/v1/environments \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "my-project",
    "name": "Production",
    "slug": "production",
    "variables": {
      "ENDPOINT_URL": "https://api.myapp.com/jobs"
    }
  }'
```

Then bind a job to an environment:

```bash
curl -X PATCH https://api.strait.dev/v1/jobs/{jobId} \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "environment_id": "env_staging_id" }'
```

The job now dispatches to the staging URL. Switch it to production by changing `environment_id`.

## Variable Inheritance

Environments can have parents. A child environment inherits all variables from its parent, and can override specific values.

```text
Production (parent)
  ENDPOINT_URL=https://api.myapp.com/jobs
  API_KEY=prod-key-123
  TIMEOUT=30

Production-EU (child, parent_id = Production)
  ENDPOINT_URL=https://eu.api.myapp.com/jobs  ← overrides parent
  # API_KEY and TIMEOUT inherited from parent
```

Strait resolves variables by walking up the parent chain and flattening the result. Child values win on conflicts.

## Plan Limits

Environments are created explicitly through the API or dashboard. New projects do not auto-provision development, staging, or production environments at launch.

When an organization reaches its plan's active environment cap, new environment creation is rejected with a structured billing error. Existing environments are not renamed or deleted during normal use. If a plan downgrade leaves the organization above its new cap, Strait keeps the newest environments and removes the oldest excess environments during downgrade cleanup.

## URL Validation

When an environment defines `ENDPOINT_URL`, Strait validates the URL to prevent requests to private networks. Private IP ranges (`10.0.0.0/8`, `192.168.0.0/16`) and loopback addresses (`127.0.0.1`) are blocked. Validation runs both when you update the environment and when the job dispatches.

Environment variables are encrypted at rest when `SECRET_ENCRYPTION_KEY` is configured.
