---
title: "Environments"
description: "Managing project-specific configurations and routing."
---

Environments let you run the same jobs and workflows against different configurations -- for example, routing to a staging API during development and a production API when you ship.

## Environment Model

Each environment has the following fields:

| Field | Type | Description |
| :--- | :--- | :--- |
| `id` | `string` | Unique identifier (UUIDv7). |
| `project_id` | `string` | The project this environment belongs to. |
| `name` | `string` | Human-readable name (e.g., "Production"). |
| `slug` | `string` | URL-friendly identifier (e.g., "prod"). |
| `parent_id` | `string` | Optional. Links to a parent environment for variable inheritance. |
| `is_standard` | `bool` | Whether this is a built-in environment. Standard environments cannot be deleted or renamed. |
| `variables` | `map[string]string` | Key-value configuration. Encrypted at rest when you set `SECRET_ENCRYPTION_KEY`. |

## Standard Environments

Every project is created with three standard environments:

- **Development** (`development`)
- **Staging** (`staging`)
- **Production** (`production`)

You cannot delete or rename standard environments. To add more granular configuration, create custom environments as children of standard ones.

## Variable Inheritance

You can nest environments by setting `parent_id`. A child environment inherits all variables from its parent. If the child defines the same key, the child's value wins. Strait resolves variables by walking up the parent chain and flattening the result at execution time.

## Endpoint URL Overrides

If an environment defines an `ENDPOINT_URL` variable, Strait uses that URL instead of the job's default `endpoint_url` at dispatch time. This lets you route the same job to different services without editing the job definition.

## SSRF Validation

To prevent Server-Side Request Forgery (SSRF) -- where an attacker tricks your system into making requests to private networks -- Strait validates all overridden URLs:
- **Blocked addresses**: Private IP ranges (`10.0.0.0/8`, `192.168.0.0/16`) and loopback addresses (`127.0.0.1`).
- **Enforcement points**: Strait validates URLs both in the API layer (when you update an environment or job) and in the worker layer (when resolving the override at dispatch time).

## Use Case: Routing

You can maintain a single job definition (e.g., `process-order`) and route it to different services per environment:
- **Staging**: `ENDPOINT_URL=https://staging.api.myapp.com/jobs`
- **Production**: `ENDPOINT_URL=https://api.myapp.com/jobs`
