Event triggers let Strait wait for something outside the current job: a webhook, an approval, a payment status change, or a signal from another system.

The wait can last minutes, hours, or days without keeping a worker busy.

## How It Works

1. A job or workflow creates a wait with an `event_key`.
2. The run moves to `waiting`.
3. Your system sends an event with the same key.
4. Strait resumes the run or workflow with the event payload.

```text
check-order -> wait-for-approval -> fulfill-order
                    ^
                    |
          POST /v1/events/approval:order_123/send
```

## Event Keys

An `event_key` identifies the event Strait is waiting for.

Use a key that is unique to the real-world thing you are waiting on:

```text
approval:order_123
payment:invoice_456
aml-check:user_abc123
```

Keys must be unique while they are waiting. Prefixing keys by type and entity ID is the easiest way to avoid collisions.

## Workflow Waits

Use a `wait_for_event` step when a workflow needs an external signal before moving on.

```json
{
  "step_ref": "approval",
  "type": "wait_for_event",
  "event_key": "approval:{{payload.order_id}}",
  "timeout_secs": 86400,
  "depends_on": ["create-order"]
}
```

When the event arrives, Strait completes the wait step and continues the workflow.

## Durable Sleep

Use a `sleep` step when the workflow needs to pause for a known amount of time.

```json
{
  "step_ref": "cooldown",
  "type": "sleep",
  "sleep_duration": "30m",
  "depends_on": ["notify"]
}
```

Sleep steps are useful for cooling-off periods, reminders, retry-after windows, and delayed follow-ups.

## Waiting from a Job

A job can also pause itself with the SDK run endpoint. This endpoint uses the run token Strait sends to the active job handler, not a project API key.

```bash
curl -X POST https://api.strait.dev/sdk/v1/runs/{runID}/wait-for-event \
  -H "Authorization: Bearer $RUN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "event_key": "approval:order_123",
    "timeout_secs": 7200
  }'
```

When the event arrives, Strait re-queues the run with the event payload available to the handler.

## Sending an Event

Send an event to resolve one waiting key:

```bash
curl -X POST https://api.strait.dev/v1/events/approval:order_123/send \
  -H "Authorization: Bearer strait_..." \
  -H "Content-Type: application/json" \
  -d '{
    "payload": {
      "approved": true,
      "reviewer": "alice@example.com"
    }
  }'
```

If the same event is sent again with the same payload, Strait treats it as the same result. If the payload is different, Strait returns a conflict.

## Timeouts

Set `timeout_secs` so a run does not wait forever.

When the timeout expires:

| Source | Result |
| :--- | :--- |
| Workflow step | The step fails and the workflow follows its failure policy. |
| Job run | The run moves to `timed_out`. |

Use longer timeouts for human approvals and shorter timeouts for machine-to-machine callbacks.

## Canceling a Wait

Cancel a wait when the external work is no longer needed:

```bash
curl -X DELETE https://api.strait.dev/v1/events/approval:order_123 \
  -H "Authorization: Bearer strait_..."
```

## Watching Status

List waiting events:

```bash
curl "https://api.strait.dev/v1/events?status=waiting" \
  -H "Authorization: Bearer strait_..."
```

Stream updates for one event:

```bash
curl -N https://api.strait.dev/v1/events/approval:order_123/stream \
  -H "Authorization: Bearer strait_..."
```
