When a [Job Run](/concepts/runs) reaches a terminal state (`completed`, `failed`, `timed_out`) or an [Event Trigger](/concepts/event-triggers) resolves, Strait can notify your external services automatically via webhook.

## Delivery Models

Strait has two webhook delivery models:

### Job Run Webhooks

When a run reaches a terminal state, the executor checks whether the parent Job has a `webhook_url` configured and sends a signed HTTP POST. Strait retries delivery in-memory with blocking backoff (up to 3 attempts). If the worker process crashes during delivery, in-flight retries are lost.

### Event Trigger Webhooks

Event trigger webhooks (configured via `notify_url` on wait-for-event steps) use a **durable delivery** model. Strait persists deliveries to the `webhook_deliveries` table and a background worker processes them. This survives process restarts -- Strait picks up pending deliveries on the next poll cycle.

## Delivery Lifecycle

1. **Terminal state reached**: When a run finishes, the executor checks whether the parent Job has a `webhook_url`.
2. **Payload construction**: Strait assembles a `WebhookPayload` containing the run's final state, result, and error (if any).
3. **Signing**: If you configured a `webhook_secret`, Strait signs the payload with HMAC-SHA256 and includes a timestamp for replay protection.
4. **Dispatch**: Strait sends an HTTP POST to the `webhook_url`.
5. **Retry**: If delivery fails, Strait retries based on the response status code.
6. **Dead letter**: After exhausting retries, Strait marks the delivery as `dead` (event trigger) or silently drops it (job run). Dead-lettered deliveries -- ones that failed all retry attempts -- stay in the database so you can inspect and replay them.

## Webhook Payload

The payload Strait sends to your endpoint includes:

| Field | Type | Description |
| :--- | :--- | :--- |
| `run_id` | `string` | Identifies which run triggered this webhook. |
| `job_id` | `string` | The job definition this run belongs to. |
| `project_id` | `string` | The project that owns the job. |
| `status` | `string` | The terminal status (`completed`, `failed`, `timed_out`, etc.). |
| `attempt` | `int` | How many attempts the run made before finishing. |
| `result` | `json` | The output data from a successful run (null on failure). |
| `error` | `string` | The error message from a failed run (null on success). |
| `timestamp` | `time.Time` | When Strait generated the webhook. |

## HMAC-SHA256 Signing

Webhook signing lets you verify that a delivery actually came from Strait and hasn't been tampered with. Signed deliveries include:

- `X-Strait-Timestamp`: Unix timestamp used in signature generation
- `X-Strait-Signature`: `sha256=<signature>`
- `X-Webhook-Signature`: compatibility header for existing consumers

Strait signs the payload with HMAC-SHA256 using your `webhook_secret` as the key and `{timestamp}.{raw-json-body}` as the message. The timestamp is included so you can reject deliveries older than 5 minutes, preventing replay attacks.

**Signed payload format:**
```
signed_payload = "{unix_timestamp}.{raw_json_body}"
signature      = HMAC-SHA256(webhook_secret, signed_payload)
```

### Verifying Signatures (Go Example)

```go
func verifySignature(payload []byte, timestamp string, signatureHeader string, secret string) bool {
    if !strings.HasPrefix(signatureHeader, "sha256=") || timestamp == "" {
        return false
    }
    actualSig := signatureHeader[7:]

    mac := hmac.New(sha256.New, []byte(secret))
    mac.Write([]byte(timestamp + "."))
    mac.Write(payload)
    expectedSig := hex.EncodeToString(mac.Sum(nil))

    return hmac.Equal([]byte(actualSig), []byte(expectedSig))
}
```

## Retry Strategy

### Job Run Webhooks

Strait attempts to deliver job run webhooks up to **3 times** with exponential backoff:
- **Attempt 1**: Immediate
- **Attempt 2**: 1 second delay
- **Attempt 3**: 5 seconds delay

### Event Trigger Webhooks

Event trigger webhooks (configured via `notify_url` on wait-for-event) use a persistent delivery queue that survives process restarts:
- **Max attempts**: 5
- **Backoff**: Exponential — 5s, 25s, 125s, 625s (capped at 30 minutes)
- **Delivery worker**: Polls the database every 5 seconds for pending deliveries

### Retry Rules
- **Client errors (4xx)**: Strait treats these as permanent failures and dead-letters the delivery immediately -- no retries.
- **Server errors (5xx) / Timeouts**: Strait retries up to the maximum attempt limit.
- **Context cancellation**: If the system is shutting down, Strait may abort delivery with a "context canceled" error.

## Configuration

### Job Webhooks
Webhooks are configured per job:
- `webhook_url`: The destination for the POST request.
- `webhook_secret`: (Optional) The key used for HMAC signing.

### Event Trigger Webhooks
Event trigger webhooks are configured per trigger:
- `notify_url`: Set when creating a wait-for-event step or SDK call.
- Strait tracks delivery status in the `webhook_deliveries` table using the `event_trigger_id`.

### Dead Letter Queue (DLQ) Management

Strait stores failed webhook deliveries in the database. You can inspect and retry them:

```bash
# List all failed deliveries for a project
curl "https://strait.dev/v1/webhook-deliveries?project_id=proj_1&status=failed" \
  -H "Authorization: Bearer strait_..."

# Retry a specific failed delivery
curl -X POST https://strait.dev/v1/webhook-deliveries/{deliveryID}/retry \
  -H "Authorization: Bearer strait_..."
```

The retry endpoint resets the delivery to `pending` status with zero attempts and immediate next retry time. You can only retry deliveries in `failed` status -- retrying a `pending` or `delivered` delivery returns `409 Conflict`.

<Warning>
Always use HTTPS for your webhook URLs to protect the payload and signature in transit.
</Warning>
