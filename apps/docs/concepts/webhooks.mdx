Webhooks notify your services when something happens in Strait: a job completes, a run fails, or an event triggers. Strait signs every webhook so you can verify it came from us.

## Two Types of Webhooks

Strait has two webhook models with different delivery guarantees:

**Job webhooks** are configured on the job itself (`webhook_url`). When a run reaches a terminal state (`completed`, `failed`, `timed_out`), Strait sends a signed POST to that URL. Delivery is retried in-memory up to 3 times.

**Event trigger webhooks** are configured on wait-for-event steps (`notify_url`). These use durable delivery: Strait persists the delivery to the database before dispatch, so notifications survive process restarts.

## Setting Up a Job Webhook

Set `webhook_url` and optionally `webhook_secret` on the job:

```json
{
  "name": "process-order",
  "slug": "process-order",
  "webhook_url": "https://api.example.com/webhooks/strait",
  "webhook_secret": "whsec_your_secret_here"
}
```

When a run finishes, Strait sends a POST with this payload:

| Field | Type | Description |
| :--- | :--- | :--- |
| `run_id` | `string` | Which run triggered this webhook. |
| `job_id` | `string` | The job this run belongs to. |
| `project_id` | `string` | The project that owns the job. |
| `status` | `string` | Terminal status: `completed`, `failed`, `timed_out`. |
| `attempt` | `int` | How many attempts the run made. |
| `result` | `json` | Output data from a successful run (null on failure). |
| `error` | `string` | Error message from a failed run (null on success). |
| `timestamp` | `time` | When Strait generated the webhook. |

## Verifying Signatures

If you set a `webhook_secret`, Strait signs the payload with HMAC-SHA256. The signature covers a timestamp and the raw JSON body, so you can reject replays older than 5 minutes.

Headers included on signed deliveries:

- `X-Strait-Timestamp`: Unix timestamp used in signing
- `X-Strait-Signature`: `v1=<hex signature>`

**Signed payload format:**
```text
signed_payload = "{unix_timestamp}.{raw_json_body}"
signature      = HMAC-SHA256(webhook_secret, signed_payload)
```

### Verification Example (Go)

```go
func verifySignature(payload []byte, timestamp string, signatureHeader string, secret string) bool {
    if !strings.HasPrefix(signatureHeader, "v1=") || timestamp == "" {
        return false
    }
    actualSig := signatureHeader[3:]

    mac := hmac.New(sha256.New, []byte(secret))
    mac.Write([]byte(timestamp + "."))
    mac.Write(payload)
    expectedSig := hex.EncodeToString(mac.Sum(nil))

    return hmac.Equal([]byte(actualSig), []byte(expectedSig))
}
```

## Retry Behavior

**Job webhooks** retry up to 3 times with short backoff (immediate, 1s, 5s). Retries are in-memory: if the process restarts, in-flight retries are lost.

**Event trigger webhooks** retry up to 5 times with exponential backoff (5s, 25s, 125s, 625s, capped at 30 minutes). Retries are durable and survive process restarts.

For both types, 4xx responses are treated as permanent failures (no retries). 5xx responses and timeouts are retried.

## Failed Deliveries

Failed event trigger deliveries are stored in the database. You can inspect and replay them:

```bash
# List failed deliveries
curl "https://your-strait-instance/v1/webhooks/deliveries?status=failed" \
  -H "Authorization: Bearer strait_..."

# Retry a specific delivery
curl -X POST "https://your-strait-instance/v1/webhooks/deliveries/{deliveryID}/retry" \
  -H "Authorization: Bearer strait_..."
```

The legacy `/v1/webhook-deliveries` path remains available. New integrations should use `/v1/webhooks/deliveries` and `/v1/webhooks/deliveries/{id}/retry`.

<Warning>
Always use HTTPS for your webhook URLs to protect the payload and signature in transit.
</Warning>
