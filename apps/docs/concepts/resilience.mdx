Strait protects your endpoints from cascading failures. When a service is overloaded, unhealthy, or receiving bad data, these patterns keep the rest of your system stable.

## Circuit Breaker

When an endpoint fails repeatedly, Strait stops sending it traffic — like a fuse that trips to protect the circuit.

**The problem:** Your payment API goes down. Without protection, Strait keeps dispatching runs to it, piling up timeouts and wasting retries.

**How it works:** Strait tracks consecutive failures per endpoint. When failures cross the threshold, the circuit "opens" and Strait stops dispatching to that endpoint. After a cooldown, Strait sends a single test request. If it succeeds, the circuit closes and normal traffic resumes. If it fails, the circuit stays open.

| State | What happens |
| :--- | :--- |
| **Closed** (default) | Runs dispatch normally. Failures are counted. |
| **Open** | All dispatches skip this endpoint. Runs requeue with backoff. |
| **Half-open** | One test request fires. Success closes the circuit; failure reopens it. |

### Endpoint Health Scoring

Beyond binary open/closed, Strait computes a health score (0–100) for each endpoint based on three signals:

- **Success rate** (50% weight): Fraction of recent dispatches that succeeded.
- **Failure/timeout rate** (30% weight): Fraction that failed or timed out.
- **Latency score** (20% weight): How close P95 latency is to the configured timeout.

| Score | Level | Effect |
| :--- | :--- | :--- |
| > 60 | Healthy | Full concurrency allowed |
| 30–60 | Degraded | Concurrency throttled to 25% of max |
| < 30 | Unhealthy | Endpoint blocked, runs snoozed with backoff |

### Fallback Endpoint

Set `fallback_endpoint_url` on a job to route runs to a backup URL when the primary fails. The fallback fires after the primary returns a non-retryable error or exhausts retries within the current dispatch cycle. Fallback attempts count against the run's `max_attempts` budget.

## Rate Limiting

Control how fast Strait calls your endpoint by setting rate limits on the job definition.

**The problem:** Your downstream API allows 100 requests per minute. Without rate limiting, a burst of triggered runs could exceed that and get you throttled or blocked.

Set `rate_limit_max` and `rate_limit_window_secs` on a job:

```json
{
  "rate_limit_max": 100,
  "rate_limit_window_secs": 60
}
```

When a job exceeds its rate limit, runs stay in `queued` status until the window resets.

### API-Level Rate Limits

Strait also applies rate limits at the API layer:

- **Global**: `RATE_LIMIT_REQUESTS` per `RATE_LIMIT_WINDOW` per IP
- **Trigger**: `TRIGGER_RATE_LIMIT_REQUESTS` for `/trigger` endpoints

## Concurrency Limiting

Cap how many runs of a single job execute at the same time.

**The problem:** A batch trigger creates 500 runs of the same job. Without a cap, all 500 try to run simultaneously and overwhelm the downstream service.

Set `max_concurrency` on the job definition:

```json
{
  "max_concurrency": 10
}
```

Strait checks the count of currently executing runs when dequeuing. If the job is at its limit, the run stays queued until a slot opens.

For dynamic scaling based on system load, see [Adaptive Concurrency](/concepts/adaptive-concurrency).

## Poison Pill Detection

A poison pill is a run that crashes with the same error every time — for example, a malformed payload that always triggers a 500. Without detection, it burns through all retry attempts, wastes capacity, and can trip the circuit breaker for healthy runs on the same endpoint.

Set `poison_pill_threshold` on the job definition:

```json
{
  "poison_pill_threshold": 3
}
```

When consecutive failures produce the same error (matched by hashing the first 200 characters), Strait sends the run to `dead_letter` instead of retrying.

- **`null` or `0`**: Disabled (default).
- **`1`**: First failure goes straight to `dead_letter`.
- **`3`** (recommended): Catches persistent failures before wasting retry budget.

## Retry Jitter

All retry strategies add ±20% random jitter to prevent thundering herd when many runs retry simultaneously. For full retry configuration, see [Retry Strategies](/concepts/retry-strategies).

## Graceful Shutdown

When Strait receives `SIGINT`/`SIGTERM`, it shuts down in stages:

1. Stop accepting new work (polling loop exits).
2. Drain in-progress runs (30-second timeout).
3. Stop scheduler, cron, and reaper.
4. Drain HTTP server (10-second grace period).
5. Close database and Redis connections, flush telemetry.

In-progress runs finish and record results before the process exits.

## Stale Run Recovery

If a worker crashes, runs that stopped heartbeating are recovered automatically:

- Runs in `dequeued` with no heartbeat for 5+ minutes move to `system_failed`.
- Runs in `waiting` with no heartbeat for 1+ hour move to `system_failed`.
