---
title: "Resilience"
description: "Protect downstream services with retries, rate limits, concurrency limits, and dead_letter handling."
---

Strait gives each job clear limits for how it should retry, how fast it should run, and what should happen when a payload keeps failing.

Use these settings to protect the services your jobs call.

## Retries

Retries handle temporary failures such as network errors, timeouts, and short service outages.

```json
{
  "max_attempts": 5,
  "retry_strategy": "exponential",
  "retry_backoff_secs": 30
}
```

Strait adds jitter to retries so a large batch does not retry all at once.

See [Retry Strategies](/concepts/retry-strategies) for the full retry model.

## Rate Limits

Use rate limits when a downstream API only allows a certain number of requests in a time window.

```json
{
  "rate_limit_max": 100,
  "rate_limit_window_secs": 60
}
```

When the job reaches its limit, new runs wait in the queue until capacity is available again.

## Concurrency Limits

Use `max_concurrency` to cap how many runs of one job can execute at the same time.

```json
{
  "max_concurrency": 10
}
```

This is usually the first setting to add for jobs that call databases, model providers, third-party APIs, or any service with limited capacity.

## Circuit Breakers

When an endpoint fails repeatedly, Strait can pause dispatches to that endpoint for a short period. Runs are retried later instead of continuing to hit an unhealthy service.

| State | What happens |
| :--- | :--- |
| `closed` | Runs dispatch normally. |
| `open` | Runs wait and retry later. |
| `half_open` | Strait sends a test request before resuming normal traffic. |

Circuit breakers are useful when a dependency is down and repeated calls would only create more failures.

## Fallback Endpoints

Set `fallback_endpoint_url` when a job can safely use a backup endpoint.

```json
{
  "endpoint_url": "https://api.example.com/jobs/send-email",
  "fallback_endpoint_url": "https://backup.example.com/jobs/send-email"
}
```

The fallback should perform the same job and accept the same payload shape as the primary endpoint.

## dead_letter Handling

Some runs should not keep retrying. A malformed payload, missing customer record, or unsupported state can fail the same way every time.

Set `poison_pill_threshold` to move repeated identical failures to `dead_letter`.

```json
{
  "poison_pill_threshold": 3
}
```

Use the failed-runs guide to inspect and replay `dead_letter` runs after the underlying issue is fixed.

## Recommended Baseline

For most production jobs, start with:

```json
{
  "max_attempts": 5,
  "retry_strategy": "exponential",
  "retry_backoff_secs": 30,
  "max_concurrency": 10,
  "poison_pill_threshold": 3
}
```

Then tune rate limits and concurrency based on the capacity of the service you call.
