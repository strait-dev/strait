When your jobs call external endpoints, failures in one service can cascade and take down others. Strait includes several resilience patterns that detect failures early, limit blast radius, and recover automatically.

## Circuit Breaker

A circuit breaker stops calling an endpoint that keeps failing -- like a fuse that trips to protect the rest of the system. When consecutive failures cross a threshold, the circuit "opens" and Strait skips dispatches to that endpoint until a cooldown period elapses.

### Job Dispatch Circuit Breaker

Strait persists circuit breaker state in the `endpoint_circuit_state` table so it survives process restarts.

**States:**
- **Closed** (default): Dispatches proceed normally. Strait tracks consecutive failures.
- **Open**: Strait skips all dispatches to this endpoint and requeues the run with backoff. After a cooldown period, the circuit moves to half-open.
- **Half-open**: Strait sends a single test request to see if the endpoint has recovered. If it succeeds, the circuit closes. If it fails, the circuit re-opens.


### Endpoint Health Scoring

Beyond the binary open/closed circuit breaker, Strait computes a continuous health score (0--100) for each endpoint. The score adjusts limits based on recent performance, weighing recent results more heavily than older ones, across three signals:

- **Success rate** (50% weight): Fraction of dispatches that return a successful response.
- **Failure/timeout rate** (30% weight): Fraction of dispatches that fail or time out.
- **Latency score** (20% weight): How close the P95 latency is to the job's configured timeout.

**Health levels:**
- **Healthy (score > 60)**: Full concurrency allowed.
- **Degraded (score 30-60)**: Concurrency is proportionally throttled (down to 25% of max).
- **Unhealthy (score < 30)**: Endpoint is blocked (equivalent to circuit open). Runs are snoozed with backoff.

Strait stores the health score in the `endpoint_health_scores` table and updates it after every dispatch. The score recovers naturally as successful dispatches push it back above the thresholds.

### Fallback Endpoint

When you configure a `fallback_endpoint_url` on a job, Strait automatically retries against the fallback URL if the primary endpoint fails. The fallback fires when:

- The primary endpoint returns a non-retryable error
- The primary endpoint has exhausted retry attempts within the current dispatch cycle

Fallback attempts are counted against the run's `max_attempts` budget. Both primary and fallback endpoints receive the same SSRF validation at dispatch time.

## Concurrency Limiting

If one job triggers a burst of concurrent runs, it can starve other jobs of worker capacity. Per-job concurrency caps prevent that.

### max_concurrency

Set `max_concurrency` on a job definition to limit how many runs of that job can execute at the same time. Strait enforces this at dequeue time -- when a run is pulled from the queue, Strait checks how many runs of that job are currently executing. If the count is at or above `max_concurrency`, the run stays in the queue until a slot opens up.

## Rate Limiting

### Per-Job Rate Limits

Set `rate_limit_max` and `rate_limit_window_secs` on a job definition to control dispatch throughput. When a job exceeds its rate limit, Strait holds runs in `queued` status until the window expires.

### API Rate Limiting

Strait applies rate limits at the API layer via `httprate` middleware:
- **Global rate limit**: `RATE_LIMIT_REQUESTS` per `RATE_LIMIT_WINDOW` per IP
- **Trigger rate limit**: `TRIGGER_RATE_LIMIT_REQUESTS` specifically for `/trigger` endpoints
- **RBAC control-plane limits**: Stricter per-route limits on mutating authorization endpoints

## Retry Strategies

You can configure one of four retry strategies per job:

| Strategy | Behavior |
| :--- | :--- |
| `exponential` | Delay doubles each attempt: 1s, 2s, 4s, 8s, ... |
| `linear` | Fixed increment per attempt: 5s, 10s, 15s, 20s, ... |
| `fixed` | Same delay every attempt |
| `custom` | Per-attempt delays specified via `retry_delays_secs` array |

All strategies add +/-20% random jitter to prevent a thundering herd when many runs retry at the same time.

When `attempt >= max_attempts`, the run moves to the review queue -- a holding area for runs that exhausted all retries. You can inspect and replay these runs via the API.

For detailed retry configuration, see [Retry Strategies](/concepts/retry-strategies).

## Poison Pill Detection

A poison pill is a run that consistently crashes with the same error across retries -- for example, a malformed payload that always triggers a 500. Without detection, these runs exhaust all retry attempts, waste worker capacity, and can trip the circuit breaker, blocking healthy runs to the same endpoint.

### How It Works

When you set `poison_pill_threshold` on a job, Strait tracks consecutive identical errors by hashing the error message:

1. On each failure, Strait hashes the first 200 characters of the error message.
2. If the hash matches the previous attempt's hash, Strait increments a counter. If it differs, the counter resets to 1.
3. When the counter reaches `poison_pill_threshold`, Strait sends the run directly to the review queue with a `poison pill detected` error instead of retrying.

### Configuration

Set `poison_pill_threshold` on the job definition:

```json
{
  "poison_pill_threshold": 3
}
```

- **`null` or `0`**: Detection disabled (default). Runs follow normal retry behavior.
- **`1`**: Aggressive -- the first failure immediately routes to the review queue.
- **`3`** (recommended): Tolerates transient variations but catches persistent failures before wasting retry budget.

### Interaction with Other Resilience Features

- **Circuit breaker**: A poison-pill review queue entry still counts as a circuit breaker failure, but it stops the run from hammering the endpoint across remaining retries.
- **Error classification**: Non-retryable error classes (client errors, auth failures) bypass poison pill detection entirely -- they go directly to the review queue.
- **Retry strategies**: When below the threshold, normal retry backoff applies. The counter persists across retries so the count stays accurate even with delays between attempts.

## Graceful Shutdown

Strait shuts down in stages to protect currently running work:

1. **Signal capture**: `SIGINT`/`SIGTERM` triggers context cancellation.
2. **Stop accepting work**: The worker polling loop exits; no new runs are dequeued.
3. **Drain running jobs**: All in-progress jobs finish. A 30-second timeout prevents indefinite blocking.
4. **Stop scheduler**: Cron ticker, delayed poller, and reaper stop.
5. **Drain HTTP server**: 10-second grace period for in-flight API requests.
6. **Cleanup**: Database and Redis connections close, telemetry exporters flush.

Running jobs continue executing even after shutdown starts, so they can complete and record results before the process exits.

## Stale Run Recovery

If a worker crashes or loses network connectivity, some runs may stop heartbeating. The stale reaper detects these and transitions them to a recoverable state:

- Runs in `dequeued` with no heartbeat for 5+ minutes move to `system_failed`.
- Runs in `waiting` with no heartbeat for 1+ hour move to `system_failed`.
- Database-level coordination ensures only one instance runs the reaper at a time.

This is a safety net for worker crashes, network partitions, and other failures where a run is claimed but never completed.

## OOM Recovery

When the OOM killer terminates a managed container (exit code `137`), Strait automatically recovers:

1. Strait classifies the exit code as an OOM kill.
2. The executor selects the next machine preset in the upgrade chain (`micro` -> `small-1x` -> ... -> `large-2x`).
3. Strait stores the recommendation for 24 hours, so future runs of the same job start at the upgraded preset.
4. If the job is already at `large-2x`, the run moves to the review queue.

This prevents OOM loops from burning through retry budget on an undersized preset.

## Region Failover

When a region is unavailable during container provisioning, the executor fails over to alternate regions. Strait always tries the primary region first, then fallback regions in order. The run is only snoozed if all configured regions are unavailable.

## Orphaned Container Cleanup

The reaper also detects orphaned containers from managed execution:

- Containers in a running state with no associated active run are flagged as orphaned.
- Strait destroys orphaned containers to prevent resource leakage and cost accumulation.
- This covers edge cases like executor crashes mid-dispatch, network partitions during result recording, and containers that outlive their run.
## Budget Protection

Strait enforces project compute budgets with a two-phase reservation model:

- **Reservation**: Before provisioning a container, Strait reserves the estimated cost against the daily budget. Concurrent dispatches cannot over-commit because each reservation is checked before proceeding.
- **Soft-limit warning**: At 80% of the daily budget, Strait emits a structured log and metric alert so you get advance notice.
- **Commit on completion**: After the run finishes, Strait replaces the reservation with actual cost. Unused budget from over-estimates is released immediately.
