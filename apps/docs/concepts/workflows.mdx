A workflow is a set of steps with dependencies between them. You define which steps run first, which wait for others to finish, and what happens when something fails. Strait runs the steps in the right order, passing data between them.

## DAG Concepts

A workflow is structured as a directed acyclic graph (DAG) — a dependency graph where steps can depend on other steps, but circular dependencies are not allowed:
- **Steps** are the individual units of work (the nodes).
- **Dependencies** connect steps via the `depends_on` field (the edges).
- **Fan-out** means one step triggers multiple downstream steps. **Fan-in** means one step waits for multiple upstream steps to finish.

A workflow can include up to **1000** steps.

<Note>
For implementation-level behavior (scheduler decisions, explainability APIs, critical-path estimates, policy enforcement, and replay controls), see [DAG Runtime](/concepts/dag-runtime).
</Note>

## Step Types

| Type | Description |
| :--- | :--- |
| `job` | Runs a [Job](/concepts/jobs). Use for any work that calls an HTTP endpoint. |
| `approval` | Pauses the workflow until a person approves or rejects. Use for deploy gates, compliance checks, or anything requiring human sign-off. |
| `sub_workflow` | Runs another workflow as a nested step. Use to reuse an existing workflow as part of a larger pipeline. |
| `wait_for_event` | Pauses until an external event arrives via API. Use for third-party callbacks or human-in-the-loop steps. See [Event Triggers](/concepts/event-triggers). |
| `sleep` | Pauses for a specified duration without consuming resources. Use for cooldown periods or scheduled delays. |

## Workflow Versioning

Workflows use a snapshotting mechanism. Every time a workflow's steps are updated, the `version` increments. The entire DAG definition is copied to a versioned table, ensuring that running workflows are not disrupted by changes to the definition.

When you update a workflow that has active runs on a previous version, the update response includes `active_runs_on_previous_version` and `previous_version_id` so you can assess the impact. Set `breaking_change: true` in the update request to record a `workflow.updated_breaking` audit event. See [Versioning](/concepts/versioning#breaking-change-detection) for details.

## Step Conditions

Steps can define a `condition` (JSON) that must evaluate to true before the step starts.
- **step_status**: Check the status of a dependency (e.g., `completed`, `failed`).
- **step_status_in**: Check a dependency against a set of statuses.
- **not**: Invert a nested condition.
- **Composite Logic**: Use `all_of` or `any_of` for complex boolean logic.

If a condition is false, the step is marked as `skipped`.

## Data Flow and Templates

Workflows support dynamic data flow between steps using template variables.

### Template Variable Rendering
Use `{{var}}` syntax with dot notation to access data:
- `{{payload.key}}`: Access the initial workflow trigger payload.
- `{{parent_outputs.step_ref.key}}`: Access the output of a specific parent step.

### Output Transforms
Steps can define an `output_transform` using JSONPath expressions. This lets a step export only a subset of its result to downstream steps.

### Payload Merging
The payload for a step is constructed by merging three layers:
1. **Trigger Payload**: The data provided when the workflow was started.
2. **Step Payload**: Static data defined in the workflow step configuration.
3. **Parent Outputs**: The results (optionally transformed) of all direct dependencies.

## Failure Policies

The `on_failure` field determines the workflow's behavior when a step fails:
- `fail_workflow` (default): The entire workflow run transitions to `failed`.
- `skip_dependents`: The failed step's dependents are marked as `skipped`, but other branches continue.
- `continue`: The failure is ignored, and dependents proceed as if it succeeded.

## Concurrency Control

- `max_concurrent_runs`: Limits how many instances of the workflow can run simultaneously.
- `max_parallel_steps`: Limits the number of steps that can execute in parallel within a single workflow run.
- `concurrency_key` (step-level): Serializes steps that share the same key, even if they are otherwise ready to run.
- `resource_class` (step-level): Assigns a step to a capacity bucket (`small`, `medium`, or `large`). The scheduler respects per-class capacity limits when determining which steps to start.

## Sub-workflow Nesting

Workflows can trigger other workflows using the `sub_workflow` step type.
- **Depth Limit**: The engine enforces a default maximum nesting depth of 10 to prevent infinite recursion.
- **Propagation**: When a sub-workflow completes, its outputs are aggregated and returned to the parent step.

## Workflow Policies

Project-level governance controls that apply to all workflows in a project:

| Policy | Description |
| :--- | :--- |
| `max_fan_out` | Maximum steps a single step can trigger. Prevents one step from spawning too many parallel branches. |
| `max_depth` | Maximum sequential dependencies from start to finish. Prevents deeply nested workflows that are hard to debug. |
| `forbidden_step_types` | List of step types that cannot be used in workflows for this project. |
| `require_approval_for_deploy` | When true, requires an approval step before any deploy-type step. |

Policies are managed via `GET /v1/workflow-policies/{projectID}` and `PUT /v1/workflow-policies/{projectID}`. They are enforced at workflow create, update, and trigger time.

## Step Overrides

At trigger time, you can provide `step_overrides` to selectively enable or disable specific steps for that specific run.

For previewing the resulting execution order (after overrides), use `POST /v1/workflows/{workflowID}/plan`. This endpoint accepts optional `step_overrides`, applies them to the current workflow version, validates the resulting DAG, and returns the topological execution order, root steps, and step count.

For validating a DAG structure without persisting changes, use `POST /v1/workflows/{workflowID}/dry-run`. This accepts an optional `steps` array and returns whether the DAG is valid.

## Runtime Introspection and Recovery

Beyond definition and execution, workflow runs expose runtime control APIs:

- `GET /v1/workflow-runs/{workflowRunID}/graph`: execution graph, runnable set, and critical-path estimate.
- `GET /v1/workflow-runs/{workflowRunID}/explain`: paginated scheduler/condition decision log.
- `POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/retry`: retry a single terminal step.
- `POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/replay-subtree`: replay a branch from a selected step.

Version analysis APIs:

- `GET /v1/workflows/{workflowID}/versions/{fromVersionID}/diff/{toVersionID}`
- `GET /v1/workflows/{workflowID}/versions/{versionID}/impact`
- `POST /v1/workflows/{workflowID}/simulate`

These endpoints are intended for operations and incident response, not only workflow authoring.
For detailed endpoint payloads and operational runbooks, see [DAG Runtime](/concepts/dag-runtime) and [DAG Operations Playbook](/guides/dag-operations-playbook).

## Dynamic DAG Expansion

Workflow runs can expand at runtime without mutating the authored workflow version or snapshot. A completed step may return a top-level `dynamic_steps` array in its output, and Strait will materialize those steps for that workflow run only.

Each dynamic step uses the same authoring shape as a normal workflow step, including:
- `step_ref`
- `depends_on`
- `job_id` or `agent_id`
- `payload`
- `condition`
- the usual step-type fields such as `approval_*`, `sub_workflow_id`, `event_*`, and `sleep_duration_secs`

Example:

```json
{
  "summary": "planned 2 worker steps",
  "dynamic_steps": [
    {
      "step_ref": "draft",
      "agent_id": "agent_writer",
      "depends_on": ["plan"],
      "payload": { "section": "draft" }
    },
    {
      "step_ref": "review",
      "agent_id": "agent_reviewer",
      "depends_on": ["draft"]
    }
  ]
}
```

TypeScript runtime helpers in `@strait/agents` can build this envelope directly:

```ts
const steps = fanOutSteps({
  dependsOn: ["planner"],
  workers: [
    { agentId: "agent_logs" },
    { agentId: "agent_metrics" },
  ],
  synthesizer: {
    stepRef: "synthesis",
    agentId: "agent_synthesizer",
  },
});

return ctx.createDynamicSteps(steps, { knownStepRefs: ["planner"] });
```

Guardrails:
- Dynamic steps are scoped to the current workflow run only.
- Runtime expansion still enforces duplicate `step_ref` rejection, DAG validation, and the overall 1000-step workflow limit.
- Dynamic steps participate in normal scheduling, retries, conditions, fan-in, graph/debug views, replay, and compensation planning once materialized.

## Event-Driven Steps

### wait_for_event

Pauses the workflow until an external event arrives via API. The step creates an [Event Trigger](/concepts/event-triggers) with a globally unique `event_key` and optional timeout. The wait consumes no compute resources.

```json
{
  "step_ref": "aml-check",
  "type": "wait_for_event",
  "event_key": "aml:{{payload.user_id}}",
  "timeout_secs": 86400,
  "depends_on": ["extract"]
}
```

When the event arrives (via `POST /v1/events/{eventKey}/send`), the step completes with the event payload as output, and downstream steps can access it via `{{parent_outputs.aml-check.*}}`.

### sleep

Durable sleep steps pause a workflow for a specified duration without consuming resources:

```json
{
  "step_ref": "cooldown",
  "type": "sleep",
  "sleep_duration": "1h",
  "depends_on": ["notify"]
}
```

The step completes automatically when the duration expires.

### Event Chaining

Steps can auto-emit events on completion using `event_emit_key`:

```json
{
  "step_ref": "process",
  "job_id": "job_process",
  "event_emit_key": "done:{{payload.batch_id}}"
}
```

When this step completes, it auto-resolves any waiting trigger with key `done:{batch_id}`, enabling cross-workflow coordination.

## Agent-Backed Job Steps

Job steps can target either a raw `job_id` or an `agent_id`. When you provide `agent_id`, Strait resolves it to the agent's hidden backing job before persisting or executing the workflow, and workflow reads include both identifiers so the API keeps the agent abstraction visible.

The TypeScript agents SDK also exposes workflow helpers that compile directly to workflow authoring payloads:

- `strait.agentStep(stepRef, agentId, options)`
- `strait.approvalStep(stepRef, options)`
- `strait.agentWorkflow({...})`
- `strait.pipelinePattern(...)`
- `strait.debatePattern(...)`
- `strait.orchestratorPattern(...)`

These helpers stay job-native under the hood by emitting workflow steps with `agent_id`.

## Shared Workflow State For Agent Steps

Workflow-backed agent runs can share state through the SDK without introducing a second persistence system. Strait reuses the existing run state store and scopes it to the parent workflow run:

- `ctx.run.state.*` stays local to the current job run
- `ctx.workflow.state.*` reads and writes shared state for the parent workflow run

SDK endpoints:

- `POST /sdk/v1/runs/{runID}/workflow-state`
- `GET /sdk/v1/runs/{runID}/workflow-state`
- `GET /sdk/v1/runs/{runID}/workflow-state/{key}`
- `DELETE /sdk/v1/runs/{runID}/workflow-state/{key}`

These endpoints are only available when the current run belongs to a workflow step. Non-workflow runs continue using per-run state only.

```json
{
  "step_ref": "triage",
  "agent_id": "agent_support_triage",
  "depends_on": ["extract"]
}
```

This keeps the workflow engine job-native while letting authored workflows stay agent-centric.

## Step-Level Retries

Each step can have its own retry configuration (`retry_max_attempts`, `retry_backoff`), allowing transient failures in one part of the DAG to be resolved without restarting the entire workflow.
