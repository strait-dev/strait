A workflow is a set of steps that run in a defined order. Use workflows when one job depends on another, when several jobs can run in parallel, or when a process needs approvals, waits, or recovery steps.

Each workflow trigger creates a workflow run. Each job step creates its own [run](/concepts/runs), so you can inspect every step separately.

## Create a Workflow

This workflow validates an order, processes payment, then sends a confirmation.

```bash
curl -X POST https://api.strait.dev/v1/workflows \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_abc123",
    "name": "Order Pipeline",
    "slug": "order-pipeline",
    "steps": [
      {
        "step_ref": "validate",
        "step_type": "job",
        "job_id": "job_validate_order"
      },
      {
        "step_ref": "charge",
        "step_type": "job",
        "job_id": "job_charge_card",
        "depends_on": ["validate"]
      },
      {
        "step_ref": "notify",
        "step_type": "job",
        "job_id": "job_send_confirmation",
        "depends_on": ["charge"]
      }
    ]
  }'
```

Trigger it:

```bash
curl -X POST https://api.strait.dev/v1/workflows/order-pipeline/trigger \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "payload": { "order_id": "ord_42" } }'
```

## Dependencies

Steps with no dependencies start first. A step with `depends_on` waits until its dependencies finish.

```json
{
  "step_ref": "ship",
  "step_type": "job",
  "job_id": "job_ship_order",
  "depends_on": ["charge", "reserve_inventory"]
}
```

This lets you model fan-out and fan-in without writing polling code.

## Step Types

| Type | Use it for |
|---|---|
| `job` | Run a Strait job. |
| `approval` | Wait for a person to approve or reject. |
| `wait_for_event` | Wait for an external callback or event. |
| `sleep` | Wait for a duration without keeping a worker busy. |
| `sub_workflow` | Reuse another workflow as a step. |

## Passing Data Between Steps

Each step receives the workflow payload, its own configured payload, and outputs from parent steps.

Use templates when a step needs data from an earlier step:

```json
{
  "step_ref": "notify",
  "step_type": "job",
  "job_id": "job_send_confirmation",
  "depends_on": ["charge"],
  "payload": {
    "order_id": "{{payload.order_id}}",
    "receipt_id": "{{parent_outputs.charge.receipt_id}}"
  }
}
```

## Conditions

Use conditions when a step should run only in some cases.

```json
{
  "step_ref": "manual-review",
  "step_type": "approval",
  "depends_on": ["risk-check"],
  "condition": {
    "type": "gt",
    "left": "{{parent_outputs.risk-check.score}}",
    "right": 80
  }
}
```

If the condition is false, the step is skipped.

## Failure Behavior

By default, a failed step fails the workflow. You can change this per step with `on_failure`.

| Policy | Behavior |
|---|---|
| `fail_workflow` | Stop the workflow and mark it failed. |
| `skip_dependents` | Skip steps that depend on the failed step. |
| `continue` | Continue as if the step succeeded. |
| `compensate` | Run configured compensation steps. |

Use [Compensating Transactions](/concepts/compensating-transactions) when your workflow performs side effects that may need to be undone.

## Approvals and Event Waits

Approval steps pause the workflow until a person approves or rejects.

Event waits pause until your application sends an event to Strait.

```bash
curl -X POST https://api.strait.dev/v1/events/kyc:user_abc123/send \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "payload": { "status": "approved" } }'
```

See [Workflow Approvals](/guides/workflow-approvals) and [Event Triggers](/concepts/event-triggers).

## Versioning

Workflows are versioned. A workflow run keeps using the version it started with, even if you edit the workflow while that run is active.

Use `breaking_change: true` when an update intentionally changes the workflow contract.

See [Versioning](/concepts/versioning) for details.

## Inspect and Recover

Workflow runs expose APIs to inspect the graph, retry a step, or replay a branch.

Common operations:

| Operation | Endpoint | Use it when |
|---|---|---|
| View graph | `GET /v1/workflow-runs/{workflowRunID}/graph` | You need to see which steps ran, failed, or are waiting. |
| Explain a run | `GET /v1/workflow-runs/{workflowRunID}/explain` | You need to understand why a step did or did not run. |
| Retry a step | `POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/retry` | One step failed because of a transient issue. |
| Replay a subtree | `POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/replay-subtree` | A step and its downstream work should run again. |

For request and response schemas, use the API reference.
