Durable workflows handle processes that span days or weeks -- waiting for external events, human approvals, or scheduled pauses -- while surviving restarts and infrastructure changes. Use them when your workflow outlives a single request or process lifetime.

## Expected Completion Tracking

Each workflow step can declare an `expected_duration_secs` estimate. When a run starts, Strait calculates the expected completion time by finding the critical path -- the longest chain of dependent steps -- through the workflow graph and summing their durations.

```json
{
  "steps": [
    {"step_ref": "validate", "expected_duration_secs": 5},
    {"step_ref": "charge", "depends_on": ["validate"], "expected_duration_secs": 10},
    {"step_ref": "notify", "depends_on": ["charge"], "expected_duration_secs": 3}
  ]
}
```

This workflow has an expected duration of 18 seconds (5 + 10 + 3 along the critical path). The `expected_completion_at` field on the workflow run is set to `started_at + 18s`.

For parallel branches, only the longest branch contributes to the critical path:

```
A(5s) -> B(10s)
A(5s) -> C(20s)
```

Critical path: A + C = 25 seconds (not A + B + C).

As steps complete, the expected completion is recalculated based on remaining steps and actual progress.

## Stage Notifications

Each workflow step can configure notifications for specific state transitions:

```json
{
  "step_ref": "charge",
  "stage_notifications": {
    "on_complete": true,
    "on_failure": true,
    "on_skipped": true
  }
}
```

When the step transitions to a matching state, Strait enqueues notification deliveries to all enabled project notification channels (webhook, Slack, email, Discord).

The notification payload includes: workflow_id, workflow_run_id, step_ref, step_run_id, status, and timestamp.

## Durability Guarantees

Sleep steps, wait-for-event steps, and approval steps all use database-scheduled triggers, not in-memory timers. This means they survive process restarts:

- **Sleep steps**: Strait creates a trigger with an expiration time. When the trigger expires, the step completes automatically.
- **Wait-for-event steps**: Strait creates a trigger keyed to a specific event. External systems POST events to resolve the trigger.
- **Approval steps**: Strait creates both an approval record and a trigger for unified tracking.

The stale run reaper distinguishes legitimately waiting workflows from stuck ones by checking whether active triggers exist with future expiration timestamps.
