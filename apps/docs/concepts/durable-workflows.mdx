Some workflows take hours or days -- an order that waits for payment confirmation, an approval that waits for a human, a pipeline that sleeps between stages. Durable workflows survive process restarts and keep running until every step is done.

## When to Use Durable Workflows

You need a durable workflow when your process outlives a single request:

- **Multi-day order fulfillment**: Create order, wait for payment, reserve inventory, wait for shipping label, send confirmation.
- **Human-in-the-loop approvals**: Run automated checks, pause for a manager's approval, then deploy.
- **Scheduled pipelines**: Process data, sleep 6 hours, check results, sleep until next business day.

## How Durable Waits Work

Three step types create waits that consume no compute resources. Each is backed by a database record, not an in-memory timer, so they survive process restarts.

- **Sleep steps**: Strait creates a trigger with an expiration time. When the time arrives, the step completes automatically.
- **Wait-for-event steps**: Strait creates a trigger keyed to a specific event. External systems call `POST /v1/events/{eventKey}/send` to resolve the trigger and continue the workflow.
- **Approval steps**: Strait creates an approval record. The step waits until a human approves or rejects it via the API, or the approval timeout expires.

## Example: E-Commerce Order Flow

```json
{
  "steps": [
    { "step_ref": "create-order", "job_id": "order-create" },
    {
      "step_ref": "wait-for-payment",
      "type": "wait_for_event",
      "event_key": "payment.confirmed.{{create-order.order_id}}",
      "timeout_secs": 86400,
      "depends_on": ["create-order"]
    },
    {
      "step_ref": "approve-high-value",
      "type": "approval",
      "timeout_secs": 3600,
      "depends_on": ["wait-for-payment"]
    },
    {
      "step_ref": "reserve-inventory",
      "job_id": "inventory-reserve",
      "depends_on": ["approve-high-value"]
    },
    {
      "step_ref": "send-confirmation",
      "job_id": "notifications-send",
      "depends_on": ["reserve-inventory"]
    }
  ]
}
```

This workflow creates an order, waits up to 24 hours for a payment event, pauses for manual approval on high-value orders, reserves inventory, and sends a confirmation. If the process restarts at any point, every step picks up where it left off.

## Expected Completion Tracking

Each step can declare an `expected_duration_secs` estimate. Strait calculates the expected completion time by finding the longest chain of dependent steps (the critical path) and summing their durations.

```json
{
  "steps": [
    { "step_ref": "validate", "expected_duration_secs": 5 },
    { "step_ref": "charge", "depends_on": ["validate"], "expected_duration_secs": 10 },
    { "step_ref": "notify", "depends_on": ["charge"], "expected_duration_secs": 3 }
  ]
}
```

Expected duration: 18 seconds (5 + 10 + 3 along the critical path). For parallel branches, only the longest branch counts. As steps complete, the estimate recalculates based on remaining work.

## Stage Notifications

Steps can notify your team when they reach specific states:

```json
{
  "step_ref": "charge",
  "stage_notifications": {
    "on_complete": true,
    "on_failure": true,
    "on_skipped": true
  }
}
```

When a step transitions to a matching state, Strait sends notifications to all enabled project channels (webhook, Slack, email, Discord). The payload includes the workflow ID, run ID, step ref, status, and timestamp.
