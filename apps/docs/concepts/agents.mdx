Strait agents are managed, deployable execution units that sit on top of the existing job and run substrate.

They give you a higher-level control plane for long-running model-driven tasks without introducing a second run system. Agent runs reuse the same core telemetry pipeline as jobs:

- `job_runs`
- `run_usage`
- `run_checkpoints`
- `run_tool_calls`

This gives agents a higher-level authoring and deployment model without forking the execution substrate used by jobs and workflows.

## Mental Model

There are four layers in the current system:

1. control plane in Go
2. runtime package in `apps/agents`
3. public runtime-facing SDK in `packages/agents-sdk`
4. dashboard and API surface in `apps/app` and `apps/strait/internal/api`

In practice, that means:

- the Go service owns agent records, deployments, run creation, and persistence
- `apps/agents` executes the runtime contract locally or on Cloudflare as internal runtime infrastructure
- `@strait/agents` is the public runtime SDK that reports runtime events back to Strait
- the dashboard reads the same run and telemetry tables used elsewhere in the platform

## Resource Model

An agent tracks:

- identity (`id`, `name`, `slug`, `description`)
- execution model (`model`, config blob, runtime entrypoint metadata)
- deployment state
- a hidden backing `job_id`

The hidden backing job is an implementation detail that preserves the existing run lifecycle, analytics, queueing, and workflow contracts.

This is the key architectural decision behind the current platform. Agents are not a second run system. They are a higher-level control plane layered on top of the existing job system.

## Deployments

Each agent can have one or more deployments recorded in `agent_deployments`.

The initial implementation is local-first:

- deployment is provider-neutral
- execution targets `apps/agents`
- no Cloudflare or remote worker dependency is required

This lets the control plane, callback contract, and telemetry paths stabilize before a remote runtime is introduced.

The production Cloudflare path uses the same control-plane contract with provider-specific deployment metadata. Cloudflare deployments record:

- dispatch namespace
- user Worker script name
- dispatch Worker URL
- compatibility date
- outbound-worker sandbox policy metadata

The runtime package in `apps/agents` now has three production entrypoints:

- runtime Worker (`wrangler.jsonc`)
- dynamic dispatch Worker (`wrangler.dispatch.jsonc`)
- outbound Worker (`wrangler.outbound.jsonc`)

The Go deploy provider uploads the built runtime bundle from `apps/agents/dist/runtime/worker.js` when it exists and falls back to an embedded snapshot bundled into the Go binary. That keeps Cloudflare deploys functional in CI and local development before the full platform account is available.

Local and Cloudflare deployments share the same `agent_deployments` table, while `provider_metadata` stores the provider-specific contract required to redeploy, dispatch runs, and enforce sandbox policy.

## Run Lifecycle

An agent run follows this sequence:

1. an API caller triggers `/v1/agents/{id}/run`
2. the Go service creates a normal run against the hidden backing job
3. the provider-specific runtime path receives a dispatch envelope
4. the runtime emits checkpoints, usage, tool calls, stream chunks, and terminal state
5. those callbacks are persisted into the standard run telemetry tables
6. the dashboard and API expose the run like any other managed execution

This keeps:

- retries
- run status transitions
- telemetry collection
- analytics
- workflow dependencies

aligned with the rest of Strait.

## Runtime Contract

The local agent runtime receives a dispatch envelope and reports lifecycle state back through the SDK callback endpoints. The contract includes:

- run dispatch input
- stream chunks
- checkpoints
- tool-call reports
- usage reports
- terminal success or failure

These callbacks are mapped directly onto the existing run FSM and persisted using the same run tables used by jobs.

The runtime contract is shared across:

- the local CLI runtime
- the Cloudflare runtime Worker
- the Cloudflare dispatch Worker callback-forwarding path

That shared contract is what makes the local-first development loop meaningful instead of being a separate fake environment.

## SDK and Adapters

The local runtime uses `@strait/agents` to report state back to Strait.

The SDK provides:

- `StraitContext`
- pricing and budget enforcement
- checkpoint, progress, and usage helpers
- workflow state helpers
- adapter wrappers for Vercel AI SDK, OpenAI, and Anthropic

See [Agents SDK](/docs/sdks/agents) for the runtime-side API.

## Package Boundaries

Use these boundaries when you need to navigate the code:

- `apps/strait/internal/agents`
  - control plane, provider implementations, deploy and dispatch logic
- `apps/agents`
  - runtime package and Cloudflare Worker entrypoints
- `packages/agents-sdk`
  - public `@strait/agents` package: context, adapters, budgets, workflow helpers, and eval helpers
- `apps/app`
  - dashboard UX for creating, deploying, running, and inspecting agents

## Cloudflare Production Path

The Cloudflare production path now covers the local-first acceptance scope for:

- `STR-289`: Go control plane integration for deploy/delete/update and remote run dispatch
- `STR-297`: Worker runtime and dynamic dispatch Worker entrypoints in `apps/agents`
- `STR-294`: outbound-worker sandbox policy enforcement for runtime egress

What remains outside this scope:

- live staging smoke against a purchased Workers for Platforms account
- future dynamic isolate / `worker_loaders` sandboxing

## Outbound Sandbox

The first production sandbox implementation uses an outbound Worker instead of dynamic isolates.

That outbound layer is responsible for:

- blocking private-network destinations
- enforcing explicit allowlists
- annotating allowed and blocked requests
- making policy decisions visible in telemetry instead of hiding them in runtime failures

This is intentionally narrower than full code sandboxing, but it is deployable today and matches the current productionization scope.

## Workflow Orchestration

Agents can participate in workflows through agent-backed job steps. The workflow engine continues to execute on the existing workflow substrate, while agent-specific behavior is layered on top.

Current orchestration support includes:

- agent-backed workflow steps
- reusable pipeline, debate, and orchestrator helpers in the agents SDK
- runtime DAG expansion via a completed-step output envelope

Dynamic expansion uses this output shape:

```json
{
  "dynamic_steps": [
    {
      "step_ref": "research-2",
      "agent_id": "agent_123",
      "depends_on": ["planner"],
      "payload": { "topic": "billing regressions" }
    }
  ]
}
```

Each dynamic step is validated against the current runtime DAG before it is materialized. Cycles, duplicate `step_ref` values, and invalid dependencies fail the parent step instead of mutating workflow state partially.
