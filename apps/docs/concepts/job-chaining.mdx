Job chaining lets you connect jobs into simple pipelines without building a full workflow DAG. When Job A completes (or fails), Strait automatically triggers Job B with mapped output as input.

## Configuration

Jobs support four chaining fields:

- `on_complete_trigger_job` -- slug of the job to trigger when this job completes successfully
- `on_complete_trigger_workflow` -- slug of the workflow to trigger on completion (existing feature)
- `on_failure_trigger_job` -- slug of the job to trigger on terminal failure (failed, timed_out, crashed, system_failed)
- `on_failure_trigger_workflow` -- slug of the workflow to trigger on failure

Both `on_complete` and `on_failure` support a `payload_mapping` field that extracts specific fields from the result using dot-notation paths.

## Payload Mapping

The mapping is a JSON object where keys are output field names and values are dot-notation paths into the result:

```json
{
  "on_complete_trigger_job": "process-result",
  "on_complete_payload_mapping": {
    "user_id": "user.id",
    "amount": "payment.amount"
  }
}
```

If the result is `{"user": {"id": "u-123"}, "payment": {"amount": 49.99}}`, the downstream job receives `{"user_id": "u-123", "amount": 49.99}`.

If mapping fails (invalid paths, non-object result), the full result is passed through as fallback.

## Failure Context

When `on_failure_trigger_job` fires, the downstream job receives a structured failure payload:

```json
{
  "source_job_id": "job-123",
  "source_run_id": "run-456",
  "error": "connection refused",
  "error_class": "connection",
  "status": "failed",
  "attempt": 3,
  "original_input": { ... }
}
```

## Chain Depth Enforcement

To prevent infinite loops (Job A triggers Job B triggers Job A), Strait enforces a maximum chain depth of 10. Each downstream run increments the `lineage_depth` counter. When the limit is reached, the trigger is logged and skipped.

## Job Dependencies

Job dependencies control execution order between jobs -- one job only runs after another job reaches a specific state.

A dependency consists of:

- **Job ID** -- the job that is waiting
- **Depends On Job ID** -- the job that must reach a certain state first
- **Condition** -- the required state

### Dependency Conditions

- `completed` (default) -- the dependency job must finish successfully
- `failed` -- the dependency job must fail
- `any` -- the dependency job must reach any terminal state (completed, failed, canceled, etc.)

### Use Cases

- **Sequential processing** -- Job B ("Generate Invoice") only runs after Job A ("Process Payment") completes
- **Error handling** -- Job C ("Notify Support") triggers only if Job A fails
- **Cleanup** -- Job D ("Delete Temporary Files") runs regardless of outcome

### Create a Dependency

```bash
curl -X POST https://strait.dev/v1/jobs/{jobB_ID}/dependencies \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "depends_on_job_id": "{jobA_ID}",
    "condition": "completed"
  }'
```

### List Dependencies

```bash
curl -H "Authorization: Bearer $API_KEY" \
  https://strait.dev/v1/jobs/{jobID}/dependencies
```

### Delete a Dependency

```bash
curl -X DELETE https://strait.dev/v1/jobs/{jobID}/dependencies/{dependencyID} \
  -H "Authorization: Bearer $API_KEY"
```

### Behavior

When a job with dependencies is triggered:

1. Strait checks if all dependencies are met.
2. If not met, the job run enters a `waiting` state or is delayed until conditions are satisfied.
3. Once all dependency conditions are met, the job moves to `queued` for execution.

<Warning>
  Circular dependencies (e.g., Job A depends on Job B, and Job B depends on Job A) are not allowed and will return a validation error.
</Warning>

## Job Groups

Job groups organize jobs into logical sets. Use them when a project has many jobs serving different purposes or belonging to different services.

### Create a Group

```bash
curl -X POST https://strait.dev/v1/job-groups \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_123",
    "name": "Email Notifications",
    "slug": "email-notifications",
    "description": "Jobs related to sending various email notifications."
  }'
```

### List Groups

```bash
curl -H "Authorization: Bearer $API_KEY" \
  https://strait.dev/v1/job-groups?project_id=proj_123
```

### Update a Group

```bash
curl -X PATCH https://strait.dev/v1/job-groups/{groupID} \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Updated Group Name"
  }'
```

### Delete a Group

```bash
curl -X DELETE https://strait.dev/v1/job-groups/{groupID} \
  -H "Authorization: Bearer $API_KEY"
```

### Assigning Jobs to Groups

When creating or updating a job, provide the `group_id`:

```bash
curl -X POST https://strait.dev/v1/jobs \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_123",
    "group_id": "{groupID}",
    "name": "Welcome Email",
    "slug": "welcome-email",
    "endpoint_url": "https://api.myapp.com/jobs/welcome-email"
  }'
```

### Listing Jobs by Group

```bash
curl -H "Authorization: Bearer $API_KEY" \
  https://strait.dev/v1/job-groups/{groupID}/jobs
```

<Tip>
  Use job groups to organize by service (e.g., "Auth Service", "Billing Service") or by function (e.g., "Data Processing", "Webhooks").
</Tip>
