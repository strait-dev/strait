---
title: "Job Chaining"
description: "Connect jobs into sequential pipelines where each job's output triggers the next."
---

Job chaining lets the output of one job automatically trigger the next. Use it for simple sequential pipelines where each job depends on the result of the previous one.

## When to Use Chaining vs Workflows

**Use chaining** for straightforward A → B → C sequences where each step depends on the previous result. Chaining is configured directly on the job — no separate definition needed.

**Use [workflows](/concepts/workflows)** when you need parallel steps, conditional branches, approval gates, or complex dependency graphs. Workflows give you a full DAG with visual debugging and simulation.

If your chain grows beyond 3-4 steps or needs branching logic, it's time to move to a workflow.

## Setting Up a Chain

Configure chaining fields on a job:

- `on_complete_trigger_job` — job slug to trigger when this job completes
- `on_complete_trigger_workflow` — workflow slug to trigger on completion
- `on_failure_trigger_job` — job slug to trigger on terminal failure
- `on_failure_trigger_workflow` — workflow slug to trigger on failure

A job can have both success and failure triggers. Failure triggers fire on any terminal failure state: failed, timed out, crashed, or system failure.

## Payload Mapping

By default, the completed job's full result becomes the next job's payload. Use `on_complete_payload_mapping` to extract specific fields:

```json
{
  "on_complete_trigger_job": "process-result",
  "on_complete_payload_mapping": {
    "user_id": "user.id",
    "amount": "payment.amount"
  }
}
```

If the result is `{"user": {"id": "u-123"}, "payment": {"amount": 49.99}}`, the downstream job receives `{"user_id": "u-123", "amount": 49.99}`.

If mapping fails (invalid paths or non-object result), the full result is passed through as a fallback.

## Failure Behavior

When `on_failure_trigger_job` fires, the downstream job receives a structured failure payload:

```json
{
  "source_job_id": "job-123",
  "source_run_id": "run-456",
  "error": "connection refused",
  "error_class": "connection",
  "status": "failed",
  "attempt": 3,
  "original_input": { }
}
```

Use this to build error-handling jobs that log failures, send alerts, or attempt recovery.

## Chain Depth

Strait enforces a maximum chain depth of 10 to prevent infinite loops (e.g., Job A triggers Job B triggers Job A). Each downstream run increments a `lineage_depth` counter. When the limit is reached, the trigger is logged and skipped.

The default limit of 10 is configurable per project.
