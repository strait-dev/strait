Strait supports two execution modes that control where and how your job code runs. The mode is set on the job definition via the `execution_mode` field.

## Comparison

| | HTTP (default) | Worker |
| :--- | :--- | :--- |
| How it works | Strait POSTs to your endpoint | Strait streams runs to a connected worker over gRPC |
| Your code runs on | Your servers | Any host running the Strait worker SDK |
| Setup | Point a job at an endpoint URL | Run the worker SDK and register handlers |
| Scaling | You scale your endpoint | Add or remove workers; Strait load-balances across them |
| Best for | Existing APIs, microservices, serverless | Long-lived processes, on-prem fleets, persistent connections |

---

## HTTP Mode

HTTP is the default execution mode. Strait sends an HTTP POST to your job's `endpoint_url` with the run payload, headers (`X-Run-ID`, `X-Job-ID`, `X-Attempt`), and an SDK token. Your server processes the request and returns an HTTP response.

- **2xx** — run succeeds.
- **408, 429, 5xx** — transient failure, Strait retries.
- **Other 4xx** — permanent failure, no retry.

### When to use HTTP

- You already have an API or microservice that does the work.
- Your infrastructure handles scaling, and you want Strait only for orchestration, retries, and scheduling.
- You run on serverless platforms (Lambda, Cloud Run, Vercel) where the platform manages compute.

### Example

```json
{
  "name": "Send Welcome Email",
  "slug": "send-welcome-email",
  "endpoint_url": "https://your-api.com/jobs/send-welcome-email"
}
```

No `execution_mode` field is needed — HTTP is the default. See [Jobs](/concepts/jobs) for the full configuration reference.

## Worker Mode

In worker mode, Strait dispatches runs to a long-lived worker process that connects to the API over gRPC. Workers register at startup, hold a streaming connection, and receive runs as Strait dequeues them.

When Strait dequeues a worker run:

1. **Match a worker.** Strait selects a connected worker whose registered job slugs include the run's job.
2. **Stream the run.** The run payload and an SDK token are streamed to the worker over the gRPC channel.
3. **Wait for the result.** The worker executes your handler and streams the result back; Strait records duration and the final state on the run.
4. **Heartbeat.** Disconnected workers are detected by the heartbeat sweep and their in-flight runs are reclaimed.

### When to use worker mode

- You want a persistent gRPC connection rather than exposing a public HTTP endpoint.
- You run on networks (on-prem, private VPC) where Strait cannot reach your servers directly.
- You need a single process to handle many runs concurrently with shared in-memory state (caches, connection pools).

### Example

```json
{
  "name": "Train Model",
  "slug": "train-model",
  "execution_mode": "worker"
}
```

Start a worker with the SDK and it will pick up runs for any registered slug. See the SDK integration guide for the connection and registration flow.

## Choosing a Mode

```
Do you already have an HTTP endpoint that does the work?
├── Yes → Use HTTP mode.
└── No  → Run a worker process and use worker mode.
```

**Start with HTTP** if you have existing infrastructure. It requires no changes to your deployment — just point a job at your endpoint. Switch to worker mode when you need a persistent connection, run on a network Strait can't reach, or want to keep heavy state in-process.
