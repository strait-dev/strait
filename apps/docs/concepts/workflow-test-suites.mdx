Test suites let you define expected outcomes for your workflows and run them automatically. Use them in CI to catch regressions before deploying.

## When to Use

- Before deploying workflow changes to production
- In CI pipelines as an automated gate
- When refactoring workflows that other teams depend on

## Defining a Test Suite

Tests are defined in JSON. Each test specifies a workflow, a sample payload, mock endpoint responses, and assertions about what should happen:

```json
{
  "tests": [
    {
      "name": "happy path: order processed",
      "workflow": "order-processing",
      "payload": {"order_id": "test-123", "amount": 99.99},
      "mocks": {
        "payment-service": {
          "response": {"status": "charged", "transaction_id": "txn-456"},
          "status_code": 200
        }
      },
      "assertions": [
        {"step": "validate-order", "status": "completed"},
        {"step": "charge-payment", "status": "completed", "output_contains": "txn-456"},
        {"workflow_status": "completed"},
        {"step": "total", "duration_under": "30s"}
      ]
    }
  ]
}
```

## Assertion Types

| Assertion | Fields | What it checks |
|-----------|--------|----------------|
| Step status | `step` + `status` | A step reached the expected terminal status |
| Output contains | `step` + `output_contains` | Step output includes a substring |
| Workflow status | `workflow_status` | The overall workflow reached the expected outcome |
| Duration bound | `step` + `duration_under` | Step completed within a time limit |

## Mock Endpoints

Override job endpoints with mock responses during test execution. Each mock accepts:

- `response` — JSON response body returned to the step
- `status_code` — HTTP status code (default 200)
- `headers` — response headers
- `latency_ms` — simulated response delay in milliseconds

Mocks prevent tests from hitting real services, keeping results deterministic.

## Running in CI

Test results output as JUnit XML for CI systems:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<testsuite name="order-processing" tests="2" failures="1">
  <testcase name="happy path" classname="order-processing" time="0.150"/>
  <testcase name="payment failure" classname="order-processing" time="0.080">
    <failure message="step charge status: expected=completed actual=failed" type="AssertionFailure"/>
  </testcase>
</testsuite>
```

Exit code 0 when all tests pass, 1 when any test fails. Run a subset of tests by name pattern:

```
strait test --filter "payment failure"
```
