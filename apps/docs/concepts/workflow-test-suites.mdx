Workflow test suites let platform teams write tests for their workflows the same way they write tests for application code: defined fixtures, mocked external services, assertions on outcomes, and deterministic results.

## Test Definition Format

Tests are defined in JSON (YAML support planned):

```json
{
  "tests": [
    {
      "name": "happy path: order processed",
      "workflow": "order-processing",
      "payload": {"order_id": "test-123", "amount": 99.99},
      "mocks": {
        "payment-service": {
          "response": {"status": "charged", "transaction_id": "txn-456"},
          "status_code": 200
        }
      },
      "assertions": [
        {"step": "validate-order", "status": "completed"},
        {"step": "charge-payment", "status": "completed", "output_contains": "txn-456"},
        {"workflow_status": "completed"},
        {"step": "total", "duration_under": "30s"}
      ]
    }
  ]
}
```

## Assertion Types

| Assertion | Field | Description |
|-----------|-------|-------------|
| Step status | `step` + `status` | Verify a step reached the expected status |
| Output contains | `step` + `output_contains` | Check step output contains a substring |
| Workflow status | `workflow_status` | Verify the overall workflow outcome |
| Duration bound | `step` + `duration_under` | Verify execution completed within time limit |

## Mock Endpoints

The mock configuration supports:

- `response` -- JSON response body
- `status_code` -- HTTP status code (default 200)
- `headers` -- Response headers
- `latency_ms` -- Simulated response delay

## CI Integration

Test results can be output as JUnit XML for CI systems:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<testsuite name="order-processing" tests="2" failures="1">
  <testcase name="happy path" classname="order-processing" time="0.150"/>
  <testcase name="payment failure" classname="order-processing" time="0.080">
    <failure message="step &quot;charge&quot; status == completed: expected=completed actual=failed" type="AssertionFailure"/>
  </testcase>
</testsuite>
```

Exit code 0 for all pass, 1 for any failure.

## Test Filtering

Run specific tests by name pattern:

```
strait test --filter "payment failure"
```

## Validation

Test suites are validated at parse time:
- Every test must have a `name` and `workflow`
- At least one assertion is required per test
- Each assertion must specify either `step` or `workflow_status`

## Implementation

- Package: `apps/strait/internal/workflow/testing/`
- Types: `types.go` (TestSuite, TestCase, Assertion, MockEndpoint, TestResult)
- Parser: `parser.go` (JSON parsing with validation)
- Runner: `runner.go` (assertion evaluation)
- Output: `output.go` (JUnit XML generation)
- Tests: `testing_test.go` (31 tests including fuzz and adversarial)
