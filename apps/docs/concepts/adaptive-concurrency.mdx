Adaptive concurrency lets Strait raise or lower execution capacity between a minimum and maximum value.

Use it when workload changes during the day and a fixed concurrency value is either too low for peaks or too high during quiet periods.

## When to Use It

Use adaptive concurrency when:

- Traffic varies by hour, day, or upstream event.
- A downstream service starts failing when too many requests run at once.
- You want a guardrail without constantly tuning one fixed worker count.

For simple jobs, start with `max_concurrency` first. Add adaptive concurrency when total workload changes enough that a fixed value becomes hard to manage.

## How It Behaves

Strait re-evaluates capacity on a fixed interval using two signals: how deep the queue is and how busy the current executors are. It does not look at success or failure rates; per-job retries and circuit breakers handle failing endpoints separately.

**Scaling up.** When the queue is backing up (more than twice the current limit) and executors are busy (over 70% utilization), Strait raises the limit. The increase is tiered by how deep the backlog is:

| Queue depth | Increase |
| :--- | :--- |
| Over 1000 | Double the current limit |
| Over 100 | Raise by 50% |
| Otherwise | Raise by 25% |

**Scaling down.** When the queue is empty and executors are mostly idle (under 20% utilization) for two consecutive checks, Strait lowers the limit by 33%.

Every adjustment is clamped to the configured minimum and maximum, so capacity rises quickly under load and eases back down during quiet periods without ever leaving the bounds you set.

## Configuration

Set these values on your Strait instance:

| Variable | Default | Purpose |
| :--- | :--- | :--- |
| `ADAPTIVE_CONCURRENCY_MIN` | `5` | Lowest execution capacity Strait can use. |
| `ADAPTIVE_CONCURRENCY_MAX` | `100` | Highest execution capacity Strait can use. |

Example:

```bash
ADAPTIVE_CONCURRENCY_MIN=5
ADAPTIVE_CONCURRENCY_MAX=50
```

Choose the minimum based on your normal quiet-period workload. Choose the maximum based on what your downstream services can safely handle.

## Per-Job Limits Still Apply

Adaptive concurrency controls overall execution capacity.

Per-job limits such as `max_concurrency` and `rate_limit_max` still apply. If adaptive concurrency allows 50 active runs but a job has `max_concurrency: 5`, that job still runs at most 5 at a time.
