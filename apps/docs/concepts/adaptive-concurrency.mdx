Adaptive concurrency automatically scales the number of workers between a min and max bound based on observed throughput and error rates. Instead of picking a fixed worker count, Strait adjusts execution capacity in real time so you don't have to.

## When to Use It

Your API handles 100 requests/second normally but spikes to 500 during peak hours. With a fixed worker pool sized for the peak, you waste resources 90% of the time. Sized for the average, you drop work during spikes.

Adaptive concurrency scales the pool up during the spike and back down after, so you get both efficiency and capacity.

Use it when:

- **Load is unpredictable**: Traffic patterns vary by time of day, events, or upstream triggers.
- **Downstream services have limits**: Strait backs off when error rates increase, protecting APIs you call from being overwhelmed.
- **You want efficiency without tuning**: Only the workers needed for current demand are active.

## How It Works

Strait uses AIMD (Additive Increase, Multiplicative Decrease) — the same algorithm TCP uses for congestion control.

### Scaling Up

When throughput is high and errors stay low, Strait increases the worker count by 1 each monitoring interval. Gradual increases avoid overshooting — the pool climbs steadily until it hits the ceiling or errors appear.

### Scaling Down

When errors spike or throughput drops, Strait cuts concurrency by 50%. Aggressive reduction stops a bad situation from getting worse. If your downstream API starts returning 429s at 40 concurrent workers, Strait drops to 20 and climbs back gradually.

### Convergence

The combination of slow increase and fast decrease converges on a stable operating point: the highest throughput your system can sustain without errors. The pool oscillates slightly around this point, which is expected and healthy.

## Configuration

Set these environment variables on your Strait instance:

| Variable | Default | What it controls |
| :--- | :--- | :--- |
| `ADAPTIVE_CONCURRENCY_MIN` | `1` | Floor for the worker count. Strait never scales below this, even during error spikes. Set this to your minimum acceptable throughput. |
| `ADAPTIVE_CONCURRENCY_MAX` | `WORKER_CONCURRENCY` | Ceiling for the worker count. Strait never scales above this, even during traffic bursts. Defaults to your configured worker concurrency. |

### Example

For a system that needs at least 5 workers to keep up with baseline load and can handle up to 50 before saturating downstream services:

```bash
ADAPTIVE_CONCURRENCY_MIN=5
ADAPTIVE_CONCURRENCY_MAX=50
```

During normal load, the pool might settle at 15 workers. During a spike, it climbs toward 50. If errors start appearing at 40, AIMD cuts back to 20 and gradually climbs again.

### Choosing Values

- **Min**: Set this to the worker count that handles your quietest period. Too low and you'll see queuing during normal operation. Too high and you waste resources during off-hours.
- **Max**: Set this to the highest concurrency your downstream services can handle. Check your API provider's rate limits, database connection pool size, or external service SLAs.

If you're unsure, start with `MIN=1` and `MAX` at your current `WORKER_CONCURRENCY`. Monitor queue depth and error rates, then adjust.

## Interaction with Per-Job Limits

Adaptive concurrency controls the total worker pool size. Per-job limits (`max_concurrency`, `rate_limit_max`) still apply on top of it. If adaptive concurrency scales the pool to 50 workers but a specific job has `max_concurrency: 5`, that job still only runs 5 at a time — other workers pick up different jobs.
