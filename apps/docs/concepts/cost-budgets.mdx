Cost budgets prevent runaway spending on workloads with variable cost. Set per-run and daily limits to control how much your jobs can spend on API calls â€” LLM tokens, external services, or any usage your code reports.

## When to Use It

You run a summarization job that calls GPT-4 on user-uploaded documents. Most documents cost $0.05 per run, but a 200-page PDF could cost $15. Without a per-run budget, one bad input burns your daily spend. Without a daily budget, a spike in traffic does the same at scale.

## How It Works

Your job code reports cost as it runs. Strait tracks cumulative cost per run and per project per day. When a limit is hit, Strait stops the work before more money is spent.

1. **Your code reports usage**: After each API call, report what it cost via the SDK or API.

<CodeGroup>
```typescript TypeScript
await ctx.reportUsage({
  provider: "openai",
  model: "gpt-4o",
  prompt_tokens: 500,
  completion_tokens: 200,
  cost_microusd: 25000, // $0.025
});
```

```python Python
await ctx.report_usage(
    provider="openai",
    model="gpt-4o",
    prompt_tokens=500,
    completion_tokens=200,
    cost_microusd=25000,  # $0.025
)
```
</CodeGroup>

2. **Per-run check**: Each time usage is reported, Strait checks whether the run has exceeded its `max_cost_per_run_microusd` limit. If exceeded, the run is failed.

3. **Daily check**: Before a new run is enqueued, Strait checks total cost for the current day (in the project's configured timezone). If the daily limit is reached, the trigger is rejected.

All costs use **micro-USD** (1/1,000,000 of a dollar) with integer precision. This avoids floating-point rounding errors â€” $0.01 is `10000` micro-USD.

## Setting Budgets

Set budget limits on the project's quota:

```bash
curl -X PUT https://your-strait-instance/api/v1/projects/{projectId}/quota \
  -H "Authorization: Bearer $API_KEY" \
  -d '{
    "max_cost_per_run_microusd": 5000000,
    "max_daily_cost_microusd": 100000000,
    "timezone": "America/New_York"
  }'
```

This sets a $5 per-run limit and $100 daily limit. Runs that would exceed either limit are stopped before execution.

## Monitoring

At 80% of the daily budget, Strait emits a structured log and metric alert so you get advance notice before runs start being rejected.

Query current usage through the API to track spending trends and adjust limits.

## Configuration Reference

### Quota Settings

| Setting | Default | Description |
| :--- | :--- | :--- |
| `max_cost_per_run_microusd` | `0` (unlimited) | Maximum cost a single run can accumulate. |
| `max_daily_cost_microusd` | `0` (unlimited) | Maximum total cost across all runs per day. |
| `timezone` | `UTC` | Timezone for the daily budget window. |

### Usage Event Fields

| Field | Type | Description |
| :--- | :--- | :--- |
| `provider` | `string` | AI provider (e.g., `openai`, `anthropic`). |
| `model` | `string` | Model name (e.g., `gpt-4o`). |
| `prompt_tokens` | `int` | Number of input tokens. |
| `completion_tokens` | `int` | Number of output tokens. |
| `total_tokens` | `int` | Sum of prompt and completion tokens. |
| `cost_microusd` | `int64` | Cost in micro-USD. |

## Plan-quota enforcement

Project budgets are one of several gates Strait runs before accepting work. Plan-quota gates fire at the API boundary; when one trips, the response is a `402 Payment Required` with a canonical body:

```json
{
  "code": "quota_exceeded",
  "kind": "plan_cap_reached",
  "message": "monthly run cap reached",
  "limit": 10000,
  "current": 10001,
  "plan": "starter",
  "upgrade_url": "https://strait.dev/upgrade"
}
```

Top-level `code` is always `quota_exceeded` so SDKs can branch on a single string. The granular reason lives in `kind` for finer-grained UX and telemetry.

The one exception is fail-open exhaustion: if the billing layer cannot reach its dependencies safely, the response is `503 Service Unavailable` with `code: "service_degraded"` so clients treat it as a transient availability event, not a plan rejection.

### Gates and their codes

| Gate | When it fires | HTTP | `kind` |
| :--- | :--- | :--- | :--- |
| `CheckMonthlyRunLimit` | Run trigger / enqueue | `402` | `plan_cap_reached` |
| `CheckDailyRunLimit` | Run trigger / enqueue | `402` | `org_daily_run_limit_exceeded` |
| `CheckConcurrentRunLimit` | Run dispatch | `402` | `org_concurrent_run_limit_exceeded` |
| `CheckDailyAIModelCallLimit` | AI usage report | `402` | `org_daily_ai_call_limit_exceeded` |
| `CheckMaxDispatchPriority` | Run trigger with custom priority | `402` | `dispatch_priority_exceeded` |
| `CheckProjectLimit` | `POST /v1/projects/` | `402` | `project_limit_reached` |
| `CheckMemberLimit` | Adding a member to an org | `402` | `member_limit_reached` |
| `CheckOrgCreationLimit` | Creating an additional org | `402` | `org_limit_reached` |
| `CheckWorkerConnectionLimit` | gRPC worker connect | `402` | `worker_connections_reached` |
| `CheckSpendingLimit` | Run enqueue (org spend cap) | `402` | `spending_limit_reached` |
| `CheckProjectBudgetLimit` | Run enqueue (per-project budget) | `402` | `project_budget_reached` |
| `CheckProjectSuspended` | Any request on a project suspended by downgrade | `402` | `project_suspended` |
| Dunning gate | Any plan-restricted operation while in arrears | `402` | `payment_restricted` / `grace_period_expired` |
| Fail-open fallback | Billing dependency unreachable | `503` | `service_degraded` |

Codes are stable: SDKs and dashboards can rely on these strings. New reasons are added as new `kind` values; the top-level `code: "quota_exceeded"` envelope never changes.
