The workflow simulator returns the predicted execution order of a workflow's steps without running anything. It resolves each step's dependencies and reports the order they would run in, so you can sanity-check the wiring before triggering a real run.

This is a static plan based on step dependencies. The simulator does not call your endpoints, execute steps, evaluate conditions, or use a payload.

## When to Use

- After adding or reordering steps, to confirm dependencies resolve into the order you expect.
- Before triggering the first run of a newly published version.
- To catch a step that runs earlier or later than intended because of a missing or extra `depends_on`.

## Simulating a Workflow

```bash
curl -X POST https://api.strait.dev/v1/workflows/order-processing/simulate \
  -H "Authorization: Bearer $STRAIT_API_KEY"
```

The simulator uses the workflow's current published version. The response lists the resolved order:

```json
{
  "workflow_id": "order-processing",
  "version": 3,
  "predicted_order": [
    "validate-order",
    "charge-payment",
    "reserve-inventory",
    "send-confirmation"
  ],
  "step_count": 4
}
```

| Field | Description |
| :--- | :--- |
| `workflow_id` | The workflow that was simulated. |
| `version` | The published version the order was computed from. |
| `predicted_order` | Step refs in the order they would run, resolved from `depends_on`. |
| `step_count` | Number of steps in the workflow. |

## How the Order Is Computed

Strait performs a topological sort of the steps' `depends_on` relationships, so a step always appears after the steps it depends on. The order is deterministic: independent steps are returned in a stable, sorted sequence rather than at random.

Steps with no dependency relationship between them can run in parallel at runtime even though `predicted_order` lists them in sequence. To see the full dependency structure, including which steps are independent, use the workflow graph at `GET /v1/workflows/{workflowID}/graph`.
