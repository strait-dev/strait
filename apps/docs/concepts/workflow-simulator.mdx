The workflow simulator lets you test how a workflow will execute with sample data without touching production. It supports three modes: dry run, sandbox, and failure injection.

## Simulation Modes

### Dry Run

Walks through the DAG, evaluates conditions, resolves step dependencies, and returns the execution plan without calling any endpoints.

```
POST /v1/workflows/{id}/simulate
{
  "mode": "dry_run",
  "payload": {"order_id": "test-123", "amount": 99.99}
}
```

### Sandbox

Executes against real endpoints but with a sandbox flag. Results are not persisted to production run history.

### Failure Injection

Injects failures at specified steps to test fallback paths and compensation behavior:

```json
{
  "mode": "failure_injection",
  "payload": {"order_id": "test-123"},
  "failure_injection": {
    "charge-payment": "insufficient_funds"
  }
}
```

## Simulation Result

The response includes:

- **Execution plan**: Ordered list of steps with parallel groups, estimated durations, and cost estimates
- **DAG visualization**: Nodes and edges for rendering the execution graph
- **Estimated duration**: Critical path calculation (same algorithm as `expected_completion_at`)
- **Estimated cost**: Sum of per-step job cost estimates from historical data
- **Condition results**: Which conditional branches were taken (dry run assumes conditions pass)
- **Failure paths**: Steps where failures were injected, with compensation indicators
- **Compensation indicators**: Steps marked `would_compensate: true` if they have `compensation_job_id`

## Simulation History

Past simulations are saved per-user in the `workflow_simulations` table for re-running and comparing against actual executions.
