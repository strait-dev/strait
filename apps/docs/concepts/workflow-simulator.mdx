The workflow simulator lets you preview workflow behavior before running production jobs.

Use it to check step order, conditions, fallback paths, and compensation behavior.

## When to Use

- Before deploying a workflow with several dependent steps.
- After changing step order or adding conditional branches.
- Before testing fallback or compensation paths against real services.

## Simulation Modes

### Dry Run

Checks conditions, resolves step dependencies, and returns the planned order. No endpoints are called.

```
POST /v1/workflows/{id}/simulate
{
  "mode": "dry_run",
  "payload": {"order_id": "test-123", "amount": 99.99}
}
```

Dry run assumes all conditions pass. Use it to check step order and parallel branches.

### Sandbox

Calls real endpoints with a sandbox flag. Results do not appear in production run history.

Use sandbox mode when you need to check endpoint responses without creating production runs.

### Failure Injection

Inject failures at specific steps to test fallback paths and compensation behavior:

```json
{
  "mode": "failure_injection",
  "payload": {"order_id": "test-123"},
  "failure_injection": {
    "charge-payment": "insufficient_funds"
  }
}
```

The response shows which steps would trigger compensation and which fallback branches activate.

## Simulation Response

The response includes:

- **Execution plan**: ordered steps and parallel groups.
- **Condition results**: branches the workflow would take.
- **Estimated duration**: expected completion time.
- **Failure paths**: injected failures and resulting recovery steps.

## Simulation History

Past simulations are saved per user. You can run one again or compare it with a real workflow run later.
