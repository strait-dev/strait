The workflow simulator lets you test a workflow without executing real jobs. Validate your DAG structure, test conditions, and preview execution paths before deploying.

## When to Use

- Before deploying a workflow with complex step dependencies
- After refactoring step ordering or adding conditional branches
- When you want to verify fallback and compensation paths without triggering real side effects

## Simulation Modes

### Dry Run

Walks through the DAG, evaluates conditions, resolves step dependencies, and returns the planned execution order. No endpoints are called.

```
POST /v1/workflows/{id}/simulate
{
  "mode": "dry_run",
  "payload": {"order_id": "test-123", "amount": 99.99}
}
```

Dry run assumes all conditions pass. Use it to verify step ordering and parallelism.

### Sandbox

Executes against real endpoints but marks all calls with a sandbox flag. Results are not persisted to production run history.

Use sandbox mode when you need to verify actual endpoint responses but don't want runs to appear in production history.

### Failure Injection

Injects failures at specific steps to test fallback paths and compensation behavior:

```json
{
  "mode": "failure_injection",
  "payload": {"order_id": "test-123"},
  "failure_injection": {
    "charge-payment": "insufficient_funds"
  }
}
```

The response shows which steps would trigger compensation and which fallback branches activate.

## Simulation Response

The response includes:

- **Execution plan** — ordered list of steps with parallel groups and estimated durations
- **Condition results** — which conditional branches were taken
- **Estimated duration** — critical path calculation
- **Failure paths** — steps where failures were injected, with compensation indicators

## Simulation History

Past simulations are saved per user. You can re-run a previous simulation or compare it against actual workflow executions to verify your assumptions held up in production.
