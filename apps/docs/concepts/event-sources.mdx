title: "Event Sources"
description: "Named inbound event channels that route external events to jobs and workflows."
---

Event sources define where external events come from — connecting third-party services, APIs, or internal systems to Strait's event trigger system. Each source is a named inbound channel that routes events to one or more jobs or workflows via subscriptions.

## Event Sources vs Event Triggers

- **Event Sources** are inbound channels for receiving external events and routing them to jobs or workflows. They are always-on endpoints that evaluate subscriptions on every dispatch.
- **Event Triggers** are durable waits used by workflow steps and runs to pause execution until a matching event arrives. They are created by in-flight runs and consumed once.

<Note>
Event Sources push events into the system. Event Triggers wait for events from within the system.
</Note>

## Subscriptions

A subscription connects a source to a target. Two target types are supported:

- **`job`** — when matched, a new job run is enqueued with `triggered_by: "event"` and the dispatched payload.
- **`workflow`** — when matched, a new workflow run is triggered with the dispatched payload.

A single event source can have multiple subscriptions, each pointing to a different target with its own filter expression. This lets one inbound event trigger several jobs and workflows at once.

## Filter Expressions

The optional `filter_expr` field on a subscription controls whether a dispatched event triggers that subscription:

- If the expression evaluates to `true`, the subscription fires.
- If it evaluates to `false`, the subscription is skipped for that event.
- If `filter_expr` is omitted, the subscription fires on every dispatch.

<Tip>
Use filter expressions to route different event shapes through a single source. For example, a `webhooks` source could route `"action": "created"` events to one job and `"action": "deleted"` events to another.
</Tip>

### Examples

Match a specific action:
```json
{
  "filter_expr": "payload.action == 'created'"
}
```

Match a numeric threshold:
```json
{
  "filter_expr": "payload.amount > 100"
}
```

Combine conditions:
```json
{
  "filter_expr": "payload.event_type == 'order.completed' && payload.total > 0"
}
```

## Event Dispatch

Send events into the system via `POST /v1/events/dispatch`:

1. The source is looked up by `name` from the request body.
2. If the source is disabled, the dispatch is rejected.
3. Each enabled subscription is evaluated against the event payload.
4. Matching subscriptions trigger their target (enqueue a job run or start a workflow run).

```json
{
  "dispatched": 2,
  "source": "webhook-events"
}
```

<Warning>
A disabled event source rejects all dispatches. Subscriptions with `enabled: false` are also skipped, even if their filter expression would match.
</Warning>

## Schema Validation

The optional `schema` field on an event source accepts a JSON Schema definition. When set, inbound event payloads are validated against this schema before subscriptions are evaluated, catching malformed payloads early.

## API Reference

| Operation | Method | Path |
| :--- | :--- | :--- |
| Create source | `POST` | `/v1/event-sources` |
| List sources | `GET` | `/v1/event-sources` |
| Get source | `GET` | `/v1/event-sources/{sourceID}` |
| Update source | `PATCH` | `/v1/event-sources/{sourceID}` |
| Delete source | `DELETE` | `/v1/event-sources/{sourceID}` |
| Create subscription | `POST` | `/v1/event-sources/{sourceID}/subscriptions` |
| List subscriptions | `GET` | `/v1/event-sources/{sourceID}/subscriptions` |
| Delete subscription | `DELETE` | `/v1/event-sources/{sourceID}/subscriptions/{subscriptionID}` |
