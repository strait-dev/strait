You deploy a new version of a job while 50 runs are executing the old version. Versioning ensures those runs complete with the config they started with, while new runs pick up the latest config.

## When to Use Versioning

Versioning is automatic: every `UpdateJob` or workflow `PATCH` creates a new version. What you choose is the **version policy** that controls how in-flight runs interact with updates.

## How It Works

When you update a job or workflow, Strait snapshots the current configuration before applying changes. Each run records which version it was enqueued under:

```json
{
  "id": "run_abc123",
  "job_id": "job_abc123",
  "job_version": 2,
  "job_version_id": "ver_xyz789ghi012",
  "status": "completed"
}
```

When the executor picks up a run, it reads the job config at the version the run was enqueued with. If you update a job's endpoint URL while runs are queued, those queued runs still execute against the original endpoint.

## Version Policy

Each job and workflow has a `version_policy` that controls what happens to queued runs when a new version is deployed:

| Policy | Behavior | Best for |
| :--- | :--- | :--- |
| `pin` (default) | Runs execute with the version they were enqueued under. | Safety-first deployments. Most workflows. |
| `latest` | Queued runs upgrade to the latest version at dequeue time. | Fast rollouts, non-breaking changes (e.g., logging tweaks). |
| `minor` | Upgrade only if the new version is marked `backwards_compatible`. | Controlled rollouts where some changes are safe. |

### When to Use Each Policy

**Use `pin`** (the default) when:
- Your workflow has side effects tied to specific config (endpoints, schemas)
- You cannot guarantee backwards compatibility between versions
- You need exact reproducibility for auditing

**Use `latest`** when:
- Changes are non-breaking (log format, timeout adjustments)
- You want all runs to use the newest config immediately
- Fast iteration matters more than run-level consistency

**Use `minor`** when:
- Some updates are safe for in-flight runs and some are not
- You want explicit control over which updates propagate
- Set `backwards_compatible: true` on safe updates, `false` on breaking ones

## Setting Version Policy

```json
POST /v1/jobs
{
  "name": "my-job",
  "version_policy": "latest"
}
```

<Note>
Version policy is enforced at dequeue time. `pin` is the default and the safest choice. It applies automatically when you do not specify a policy.
</Note>

## Workflow Versioning

Workflows use the same versioning model. Every `PATCH /v1/workflows/{workflowID}` increments the version and snapshots the previous step graph. In-flight runs are never affected by later edits.

When you update a workflow with active runs, the response tells you:

```json
{
  "id": "wf_abc",
  "version": 4,
  "active_runs_on_previous_version": 3,
  "previous_version_id": "ver_xyz"
}
```

To acknowledge a breaking update, set `breaking_change: true` in the request body. Strait records a `workflow.updated_breaking` audit event.

## Browsing Versions

- `GET /v1/jobs/{jobID}/versions`: list all version snapshots
- `GET /v1/workflows/{workflowID}/versions`: list workflow version snapshots
- `GET /v1/workflows/{workflowID}/versions/{versionID}/steps`: step graph for a specific version
- `GET /v1/workflows/{workflowID}/active-versions`: versions with active runs and status breakdowns

## Dual Version Identifiers

Each version has two identifiers:

| Field | Type | Purpose |
| :--- | :--- | :--- |
| `version` | Integer | Auto-incrementing counter (1, 2, 3...). For ordering. |
| `version_id` | String | Globally unique ID (`ver_k8f2m9x1p3`). Safe for URLs and APIs. |
