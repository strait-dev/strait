When a job fails because of a timeout, network error, or retryable response, Strait uses the job's retry policy to decide when to try again. This page covers the four retry strategies, plus the surrounding controls that keep retries from making things worse: circuit breakers, fallback endpoints, and `dead_letter` handling.

## Core Strategies

Strait supports four retry strategies.

### Exponential

The delay increases exponentially with each attempt.

- **Formula**: `base * 2^(attempt-1)`
- **Example**: 1s, 2s, 4s, 8s
- **Use it for**: temporary outages, network failures, and rate-limited endpoints.

### Linear

The delay increases by a constant amount with each attempt.

- **Formula**: `base * attempt`
- **Example**: 1s, 2s, 3s
- **Use it for**: predictable backoff where exponential growth would wait too long.

### Fixed

The delay remains constant for every attempt.

- **Formula**: `base`
- **Example**: 30s, 30s, 30s
- **Use it for**: polling-style retries where the interval should not change.

### Custom

Use an explicit list of delays in seconds.

```json
{
  "retry_delays_secs": [1, 5, 30, 120]
}
```

If attempts exceed the list length, Strait reuses the last delay.

## Common Properties

### Jitter

Strait adds a small random variation to retry delays so many failed runs do not retry at the exact same moment.

### Delay Bounds

- Minimum delay: 1 second.
- Maximum delay: 1 hour.

### How Retries Are Scheduled

Strait records the next retry time on the run. The run stays queued until that time passes.

## Configuration

Retries can be configured at multiple levels:

- Job level: set `max_attempts`, `retry_strategy`, and `retry_delays_secs`.
- Workflow step level: override retry behavior for one step.
- Run override: trigger a specific run with a different retry policy.

## Circuit Breakers

When an endpoint fails repeatedly, Strait can pause dispatches to that endpoint for a short period. Runs are retried later instead of continuing to hit an unhealthy service.

| State | What happens |
| :--- | :--- |
| `closed` | Runs dispatch normally. |
| `open` | Runs wait and retry later. |
| `half_open` | Strait sends a test request before resuming normal traffic. |

Circuit breakers are useful when a dependency is down and repeated calls would only create more failures.

## Fallback Endpoints

Set `fallback_endpoint_url` when a job can safely use a backup endpoint.

```json
{
  "endpoint_url": "https://api.example.com/jobs/send-email",
  "fallback_endpoint_url": "https://backup.example.com/jobs/send-email"
}
```

The fallback should perform the same job and accept the same payload shape as the primary endpoint.

## dead_letter

`dead_letter` holds runs that exhausted all retries. Inspect the failure and replay the run after the issue is fixed.

Some runs should not keep retrying. A malformed payload, missing customer record, or unsupported state can fail the same way every time. Set `poison_pill_threshold` to move repeated identical failures to `dead_letter` early instead of running through the full retry budget:

```json
{
  "poison_pill_threshold": 3
}
```

See the [Handle Failed Runs](/guides/handle-failed-runs) guide for inspecting and replaying `dead_letter` runs.

## Recommended Baseline

For most production jobs, start with:

```json
{
  "max_attempts": 5,
  "retry_strategy": "exponential",
  "retry_backoff_secs": 30,
  "max_concurrency": 10,
  "poison_pill_threshold": 3
}
```

Then tune rate limits and concurrency based on the capacity of the service you call.
