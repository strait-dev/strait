When a run fails because of a timeout, network error, or retryable response, Strait schedules another attempt using backoff. This page covers the backoff policies Strait applies, where you can change them, and the surrounding controls that keep retries from making things worse: circuit breakers, fallback endpoints, and `dead_letter` handling.

## Backoff Policies

Strait applies one of two backoff policies when computing the next retry delay.

### Exponential

The delay doubles with each attempt, capped at the maximum delay.

- **Formula**: `initial * 2^(attempt-1)`, capped at the max delay.
- **Example** (initial 1s): 1s, 2s, 4s, 8s, ...
- **Use it for**: temporary outages, network failures, and rate-limited endpoints.

This is the default. Every job uses exponential backoff with a 1 second initial delay capped at 1 hour.

### Fixed

The delay stays constant for every attempt.

- **Formula**: `initial`, every attempt.
- **Example** (initial 30s): 30s, 30s, 30s, ...
- **Use it for**: polling-style retries where the interval should not change.

Fixed backoff is selectable per workflow step (see [Where Retries Are Configured](#where-retries-are-configured)).

## Common Properties

### Jitter

Strait applies +-20% random jitter to each computed delay so many runs that failed together do not retry at the same instant and overwhelm a recovering service.

### Delay Bounds

- **Initial delay**: 1 second by default.
- **Maximum delay**: 1 hour by default. Exponential growth never exceeds this.

### How Retries Are Scheduled

Strait records the next retry time on the run and leaves it queued until that time passes. There is no busy-waiting; the run is simply not eligible to dispatch until then.

## Where Retries Are Configured

### Job level

Set `max_attempts` to control how many times a run is retried before it stops. Job runs use exponential backoff with the default 1 second initial delay and 1 hour cap.

```json
{
  "max_attempts": 5
}
```

<Note>
A job's backoff shape is fixed to exponential today. To use fixed backoff or to change the initial and maximum delays, configure retries on a workflow step. The `retry_strategy` and `retry_delays_secs` job fields are accepted for forward compatibility but are not yet applied; runs use exponential backoff regardless of their value.
</Note>

### Workflow step level

A workflow step can override retry behavior for that step's runs:

| Field | Description |
| :--- | :--- |
| `retry_max_attempts` | Maximum attempts for this step. |
| `retry_backoff` | `exponential` or `fixed`. |
| `retry_initial_delay_secs` | Initial delay (and the constant delay for `fixed`). |
| `retry_max_delay_secs` | Cap on the delay for `exponential`. |

```json
{
  "step_ref": "charge-card",
  "job_id": "payment-charge",
  "retry_max_attempts": 4,
  "retry_backoff": "fixed",
  "retry_initial_delay_secs": 10
}
```

## Circuit Breakers

When an endpoint fails repeatedly, Strait can pause dispatches to that endpoint for a short period. Runs are retried later instead of continuing to hit an unhealthy service.

| State | What happens |
| :--- | :--- |
| `closed` | Runs dispatch normally. |
| `open` | Runs wait and retry later. |
| `half_open` | Strait sends a test request before resuming normal traffic. |

Circuit breakers are useful when a dependency is down and repeated calls would only create more failures.

## Fallback Endpoints

Set `fallback_endpoint_url` when a job can safely use a backup endpoint.

```json
{
  "endpoint_url": "https://api.example.com/jobs/send-email",
  "fallback_endpoint_url": "https://backup.example.com/jobs/send-email"
}
```

The fallback should perform the same job and accept the same payload shape as the primary endpoint.

## dead_letter

`dead_letter` holds runs that exhausted all retries. Inspect the failure and replay the run after the issue is fixed.

Some runs should not keep retrying. A malformed payload, missing customer record, or unsupported state can fail the same way every time. Set `poison_pill_threshold` to move repeated identical failures to `dead_letter` early instead of running through the full retry budget:

```json
{
  "poison_pill_threshold": 3
}
```

See the [Handle Failed Runs](/guides/handle-failed-runs) guide for inspecting and replaying `dead_letter` runs.

## Recommended Baseline

For most production jobs, start with:

```json
{
  "max_attempts": 5,
  "max_concurrency": 10,
  "poison_pill_threshold": 3
}
```

This gives exponential backoff (1s, 2s, 4s, ... up to 1 hour) with five attempts, bounded concurrency, and early dead-lettering of runs that fail identically. Tune rate limits and concurrency based on the capacity of the service you call.
