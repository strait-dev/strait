When a job execution fails (non-2xx response or timeout), Strait uses a retry strategy to determine when to attempt the execution again.

## Core Strategies

Strait supports four retry strategies.

### 1. Exponential (Default)
The delay increases exponentially with each attempt.
- **Formula**: `base * 2^(attempt-1)`
- **Example**:
    - Attempt 1: ~1s
    - Attempt 2: ~2s
    - Attempt 3: ~4s
    - Attempt 4: ~8s
- **Use Case**: Best for transient network issues or rate-limited endpoints.

### 2. Linear
The delay increases by a constant amount with each attempt.
- **Formula**: `base * attempt`
- **Example**:
    - Attempt 1: ~1s
    - Attempt 2: ~2s
    - Attempt 3: ~3s
- **Use Case**: Predictable, gradually increasing backoff.

### 3. Fixed
The delay remains constant for every attempt.
- **Formula**: `base`
- **Example**:
    - Attempt 1: ~1s
    - Attempt 2: ~1s
    - Attempt 3: ~1s
- **Use Case**: Polling-style retries where the interval should not change.

### 4. Custom
Uses a user-provided array of delays in seconds.
- **Behavior**: `[1, 5, 30, 120]`
    - Attempt 1: 1s
    - Attempt 2: 5s
    - Attempt 3: 30s
    - Attempt 4: 120s
- **Note**: If the number of attempts exceeds the array length, the last value in the array is repeated.
- **Use Case**: Full control over the retry sequence.

## Common Properties

### Jitter
A **plus or minus 20% random variation (jitter)** is applied to all calculated delays. This prevents all failed runs from retrying at the exact same moment, which could overwhelm your endpoint.

### Delay Bounds
- **Minimum delay**: All delays are at least **1 second**, preventing zero or near-zero delays.
- **Maximum delay**: All delays are capped at **1 hour**, no matter what the formula calculates.

### How Retries Are Scheduled
Strait adds the calculated delay to the current time and records it as the run's `next_retry_at`. The run stays in the queue and won't execute again until that time passes.

## Configuration

Retries can be configured at multiple levels:

- **Job Level**: Set `retry_strategy` and `retry_delays_secs` on the Job definition.
- **Workflow Step Level**: Steps can override the job's strategy using `retry_backoff`, `retry_initial_delay_secs`, and `retry_max_delay_secs`.
- **Run Override**: Individual runs can be triggered with overrides for `max_attempts`, `retry_backoff`, etc.

## Dead Letter Queue (DLQ)

The dead letter queue is a holding area for runs that have failed all their retry attempts. Instead of marking these runs as permanently `failed`, Strait moves them to `dead_letter` status so you can inspect the failure and manually replay the run once you've fixed the underlying issue.

Strait can also fast-track runs to the dead letter queue using **poison pill detection**: if you set `poison_pill_threshold` on a job and the same error repeats that many times in a row, Strait stops retrying early instead of burning through all attempts. See [Resilience: Poison Pill Detection](/concepts/resilience#poison-pill-detection) for details.
