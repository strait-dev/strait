When a job fails because of a timeout, network error, or retryable response, Strait uses the job's retry policy to decide when to try again.

## Core Strategies

Strait supports four retry strategies.

### Exponential

The delay increases exponentially with each attempt.

- **Formula**: `base * 2^(attempt-1)`
- **Example**: 1s, 2s, 4s, 8s
- **Use it for**: temporary outages, network failures, and rate-limited endpoints.

### Linear

The delay increases by a constant amount with each attempt.

- **Formula**: `base * attempt`
- **Example**: 1s, 2s, 3s
- **Use it for**: predictable backoff where exponential growth would wait too long.

### Fixed

The delay remains constant for every attempt.

- **Formula**: `base`
- **Example**: 30s, 30s, 30s
- **Use it for**: polling-style retries where the interval should not change.

### Custom

Use an explicit list of delays in seconds.

```json
{
  "retry_delays_secs": [1, 5, 30, 120]
}
```

If attempts exceed the list length, Strait reuses the last delay.

## Common Properties

### Jitter

Strait adds a small random variation to retry delays so many failed runs do not retry at the exact same moment.

### Delay Bounds

- Minimum delay: 1 second.
- Maximum delay: 1 hour.

### How Retries Are Scheduled

Strait records the next retry time on the run. The run stays queued until that time passes.

## Configuration

Retries can be configured at multiple levels:

- Job level: set `max_attempts`, `retry_strategy`, and `retry_delays_secs`.
- Workflow step level: override retry behavior for one step.
- Run override: trigger a specific run with a different retry policy.

## dead_letter

`dead_letter` holds runs that exhausted all retries. Inspect the failure and replay the run after the issue is fixed.

Strait can also move repeated identical failures to `dead_letter` early with `poison_pill_threshold`. See [Resilience](/concepts/resilience).
