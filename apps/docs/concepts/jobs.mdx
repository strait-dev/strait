A job is a reusable definition of work. It tells Strait where to send the payload, how long the work may run, and what to do if it fails.

Each time you trigger a job, Strait creates a [run](/concepts/runs). The job is the template. The run is one execution of that template.

## Create a Job

At minimum, a job needs a name, a slug, and an endpoint URL.

```bash
curl -X POST https://api.strait.dev/v1/jobs \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_abc123",
    "name": "Send Welcome Email",
    "slug": "send-welcome-email",
    "endpoint_url": "https://api.example.com/jobs/send-welcome-email"
  }'
```

Then trigger it:

```bash
curl -X POST https://api.strait.dev/v1/jobs/send-welcome-email/trigger \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "payload": { "user_id": "user_abc123" } }'
```

The response includes a run ID. Use that ID to check status, stream live updates, or inspect failures.

## What Your Endpoint Receives

Strait calls your endpoint with an HTTP POST.

```json
{
  "payload": { "user_id": "user_abc123" },
  "metadata": {},
  "job_id": "job_abc123",
  "run_id": "run_abc123",
  "attempt": 1
}
```

Common headers:

| Header | Description |
|---|---|
| `X-Run-ID` | The current run ID. |
| `X-Job-ID` | The job being executed. |
| `X-Attempt` | The current attempt number. |
| `X-SDK-Token` | Token used by SDK code running inside the job. |

Return any `2xx` status to mark the run as completed. Return a non-`2xx` status to fail the attempt.

## Retries and Timeouts

Jobs retry automatically when an attempt times out or returns a retryable error.

```json
{
  "max_attempts": 5,
  "timeout_secs": 60,
  "retry_strategy": "exponential"
}
```

Supported retry strategies:

| Strategy | Use it when |
|---|---|
| `exponential` | Most failures are transient. This is the default. |
| `linear` | You want a steady increase between attempts. |
| `fixed` | You want the same delay every time. |
| `custom` | You need exact retry delays. |

When a run exhausts its attempts, Strait moves it to `dead_letter` so you can inspect and replay it.

See [Retry Strategies](/concepts/retry-strategies) for the full retry model.

## Scheduling

Add `cron` to run a job on a schedule.

```json
{
  "cron": "0 9 * * 1-5",
  "timezone": "America/New_York"
}
```

This runs the job at 9:00 AM Eastern, Monday through Friday.

Use `cron_overlap_policy` to decide what happens if the previous scheduled run is still active:

| Policy | Behavior |
|---|---|
| `allow` | Start the new run anyway. |
| `skip` | Skip this tick. |
| `cancel_running` | Cancel the active run and start a new one. |

See [Scheduling](/concepts/scheduling) for more examples.

## Protect Your Endpoint

Use concurrency and rate limits when your endpoint cannot handle unlimited parallel work.

```json
{
  "max_concurrency": 10,
  "rate_limit_max": 100,
  "rate_limit_window_secs": 60
}
```

This allows up to 10 concurrent runs and at most 100 starts per minute.

For per-customer or per-resource limits, use `concurrency_key` when triggering a run.

## Avoid Duplicate Work

Use idempotency keys when the caller may retry the same request.

```bash
curl -X POST https://api.strait.dev/v1/jobs/send-welcome-email/trigger \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Idempotency-Key: user-signup-user_abc123" \
  -H "Content-Type: application/json" \
  -d '{ "payload": { "user_id": "user_abc123" } }'
```

Use `dedup_window_secs` or `debounce_window_secs` when duplicate triggers come from events rather than client retries.

## Environments

Bind a job to an environment when the same job should call different endpoints in development, staging, and production.

```json
{
  "environment_id": "env_prod"
}
```

See [Environments](/concepts/environments) for setup details.
