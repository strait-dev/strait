The DAG runtime handles execution of complex workflows -- parallel steps, conditional branching, durable waits, and recovery from failures. For workflow modeling (step types, conditions, data flow), see [Workflows](/concepts/workflows).

## Parallel Execution

Steps with no dependencies on each other run in parallel automatically. The engine groups steps by depth in the dependency graph -- steps at the same depth with no shared edges run concurrently.

```json
{
  "steps": [
    { "step_ref": "validate", "job_id": "validate-input" },
    { "step_ref": "check-inventory", "job_id": "inventory-check" },
    { "step_ref": "check-fraud", "job_id": "fraud-check" },
    { "step_ref": "charge", "job_id": "payment-charge", "depends_on": ["validate", "check-inventory", "check-fraud"] }
  ]
}
```

Here `validate`, `check-inventory`, and `check-fraud` run in parallel. `charge` starts only after all three complete.

## Conditional Branching

Steps can define a `condition` that is evaluated against the current status of other steps. If the condition is false, the step is immediately marked `skipped` and its dependents are notified.

```json
{
  "step_ref": "deploy-canary",
  "depends_on": ["build"],
  "condition": {
    "type": "all_of",
    "conditions": [
      { "type": "step_status", "step_ref": "lint", "status": "completed" },
      { "type": "not", "condition": { "type": "step_status", "step_ref": "canary", "status": "failed" } }
    ]
  }
}
```

Supported condition operators: `step_status`, `step_status_in`, `not`, `all_of`, `any_of`, `eq`, `ne`, `gt`, `gte`, `lt`, `lte`, `contains`, `in`, `regex`, `exists`.

## Concurrency Controls

Three mechanisms bound concurrent execution:

| Control | Scope | Effect |
| :--- | :--- | :--- |
| `max_parallel_steps` | Workflow | Caps total simultaneously running steps within one run. |
| `concurrency_key` | Step | Serializes steps sharing the same key, even if they are otherwise ready. Use for steps accessing the same external resource. |
| `resource_class` | Step | Assigns steps to capacity buckets: `small` (50 concurrent), `medium` (20), `large` (5). Defaults to `small`. |

## Durable Waits

Three step types create durable waits that consume no compute:

- **`sleep`**: Waits for a specified duration. The step completes automatically when the time expires.
- **`wait_for_event`**: Waits for an external event. Call `POST /v1/events/{eventKey}/send` to resolve.
- **`approval`**: Waits for a human to approve or reject via the API.

All durable waits are backed by database records, not in-memory timers. They survive process restarts. See [Durable Workflows](/concepts/durable-workflows) for details.

## Recovery

When a step or workflow fails, you have several recovery options:

**Retry a single step**
```
POST /v1/workflow-runs/{runID}/steps/{stepRef}/retry
```
Retries a failed step in place. The attempt counter increments and the step re-enters the execution pipeline.

**Replay a subtree**
```
POST /v1/workflow-runs/{runID}/steps/{stepRef}/replay-subtree
```
Re-executes a branch of the DAG from the specified step. All downstream steps re-run; steps outside the branch are untouched.

**Retry an entire run**

Creates a new workflow run that copies completed steps (with their original outputs) and re-executes from the first failed step onward. The new run records `triggered_by: retry` and links back via `retry_of_run_id`.

**Skip or force-complete a step**

Mark a stuck step as `skipped` to unblock dependents, or `completed` with an optional output payload when the external side effect succeeded but the callback was lost.

**Pause and resume**

Patch a running workflow to `paused`. Running steps finish, but no new steps start. On resume, the scheduler re-evaluates all runnable steps.

<Info>
Recovery operations create a clean audit trail -- the original run's data is never mutated.
</Info>

## Policy Enforcement

Policies constrain execution at three levels:

**Step-level**

| Policy | Field | Effect |
| :--- | :--- | :--- |
| Timeout | `timeout_secs_override` | Overrides the job's default timeout for this step. |
| Retries | `retry_max_attempts`, `retry_backoff` | Step-level retry with configurable backoff. |
| Failure | `on_failure` | `fail_workflow` (default), `skip_dependents`, `continue`, or `compensate`. |
| Cost gate | `cost_gate_threshold_microusd` | Pauses step for approval if estimated cost exceeds threshold. |

**Workflow-level**

| Policy | Effect |
| :--- | :--- |
| `max_concurrent_runs` | Rejects triggers when the limit is reached. |
| `max_parallel_steps` | Caps simultaneously running steps in a single run. |
| `timeout_secs` | Sets an expiration on the workflow run. |

**Project-level** (via `GET/PUT /v1/workflow-policies/{projectID}`)

| Policy | Effect |
| :--- | :--- |
| `max_fan_out` | Maximum steps a single step can trigger. |
| `max_depth` | Maximum sequential dependency chain length. |
| `forbidden_step_types` | Step types that cannot be used. |
| `require_approval_for_deploy` | Mandates an approval step before deploy steps. |
