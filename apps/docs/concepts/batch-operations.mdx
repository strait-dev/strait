Batch operations let you trigger thousands of jobs in a single API call, track their progress as a group, and handle failures individually without losing the entire batch.

## When to Use Batch Operations

- **Bulk notifications**: Send a message to 10,000 users in one call.
- **Data pipeline fan-out**: Process each row of an import file as a separate job run.
- **Periodic sweeps**: Re-process all failed orders from the last hour.

Any time you need to create many runs of the same job from a single trigger point.

## Triggering a Batch

Send an array of payloads to the bulk trigger endpoint:

```http
POST /v1/jobs/{jobID}/trigger/bulk
```

```json
{
  "items": [
    { "payload": { "user_id": "user_001", "message": "Your order shipped" } },
    { "payload": { "user_id": "user_002", "message": "Your order shipped" } },
    { "payload": { "user_id": "user_003", "message": "Your order shipped" } }
  ]
}
```

Strait creates a `BatchOperation` to track the group and individual runs for each item. Each run's `batch_id` links back to the batch operation.

## Tracking Progress

Query the batch to see how many items have been processed:

```http
GET /v1/batch-operations/{batchID}?project_id=...
```

```json
{
  "id": "batch_abc",
  "job_id": "job_abc123",
  "item_count": 3,
  "created_count": 3,
  "finished_at": null
}
```

- `item_count`: Number of items submitted.
- `created_count`: Number of runs actually created (may be less due to deduplication).
- `finished_at`: Set when all items have been processed. Null while in progress.

To list all runs in a batch:

```http
GET /v1/runs?batch_id={batchID}
```

## Idempotency

If your job has a [deduplication window](/concepts/jobs) configured, duplicate payloads within the same batch are filtered. The gap between `item_count` and `created_count` tells you how many items were deduplicated.

## Error Handling

Individual items can fail without failing the entire batch. Each item becomes its own run with independent retry and failure handling. A batch with 1,000 items where 3 fail gives you 997 successful runs and 3 failed runs to investigate, not a single failed batch.

## Configuration Reference

| Field | Type | Description |
| :--- | :--- | :--- |
| `id` | `string` | Unique batch identifier (auto-generated). |
| `job_id` | `string` | The job that was bulk-triggered. |
| `item_count` | `int` | Number of items submitted in the request. |
| `created_count` | `int` | Number of runs actually created. |
| `created_by` | `string` | Actor who initiated the batch (user ID or `apikey:<id>`). |
| `created_at` | `timestamp` | When the batch was created. |
| `finished_at` | `timestamp` | When all items were processed. Null while in progress. |
