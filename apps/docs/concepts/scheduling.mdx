Strait gives you several ways to control when your jobs and workflows run — recurring schedules, one-time delayed execution, and time-based windows that restrict when work can happen.

## Cron Scheduling

Jobs and workflows can be configured with a `cron` expression for automatic, recurring execution.

- **Syntax**: Five-field cron expressions. The five fields are, in order: minute (0-59), hour (0-23), day of month (1-31), month (1-12), and day of week (0-6, where 0 is Sunday). For example, `0 * * * *` means "every hour, on the hour" and `30 9 * * 1-5` means "weekdays at 9:30 AM".
- **Timezone Support**: You can pin schedules to a specific timezone (e.g., `America/Los_Angeles`) using the `timezone` field (for jobs) or `cron_timezone` field (for workflows). Without this, schedules run in UTC.

<Note>
If a job is disabled (`enabled: false`), its cron schedule is ignored by the scheduler.
</Note>

## Delayed Execution

Runs can be scheduled to execute at a specific point in the future.

- **Mechanism**: When a run is triggered with a `scheduled_at` timestamp in the future, it is created with a `delayed` status.
- **Delayed Poller**: A background component periodically checks for `delayed` runs whose `scheduled_at` time has passed and transitions them to `queued`.

## Stale Run Reaper

Strait monitors for runs that have become stuck or unresponsive.

- **Heartbeat Monitoring**: While a run is executing, it sends periodic heartbeats. If a run stops sending heartbeats past a configurable threshold, Strait marks it as stale.
- **Stale Claimed Runs**: If a worker claims a run but never starts executing it, Strait catches that too.
- **Recovery**: Stale runs transition to `crashed` or `system_failed`. If retry attempts remain, Strait automatically retries them.

## Retention Policies

Strait automatically cleans up old run data to keep your database lean.

- **Short Retention (30 days)**: Applied to terminal runs in `completed`, `failed`, `canceled`, or `expired` states.
- **Long Retention (90 days)**: Applied to terminal runs in `timed_out`, `crashed`, or `system_failed` states, providing a longer window for debugging critical failures.

## Execution Windows

You can restrict when a job is allowed to run by setting an `execution_window_cron`. Strait only dequeues the job when the current time matches the expression.

<Example>
`0 9-17 * * 1-5` restricts a job to only run during business hours (9 AM - 5 PM, Monday through Friday).
</Example>

## Cron Overlap Policies

### Jobs

Jobs support a `cron_overlap_policy` field that controls what happens when a cron tick fires while previous runs for the same job are still active.

| Policy | Behavior |
| :--- | :--- |
| `allow` (default) | Always create a new queued run, regardless of active runs. |
| `skip` | If any run is in an active state (queued, claimed, executing, waiting, or delayed), skip the cron tick entirely. |
| `cancel_running` | Cancel all active runs for the job, then create a new queued run. Strait signals cancellation through the dispatch path (HTTP context cancellation or gRPC stream cancel) and child runs are also canceled. |

<CodeGroup>
```json Skip overlapping runs
{
  "cron": "*/5 * * * *",
  "cron_overlap_policy": "skip"
}
```
```json Cancel and replace
{
  "cron": "*/5 * * * *",
  "cron_overlap_policy": "cancel_running"
}
```
</CodeGroup>

### Workflows

Workflows support a `skip_if_running` flag. If enabled, the scheduler will skip triggering a new run if an instance of that workflow is already in a `running` state. This is useful for preventing overlapping executions of long-running maintenance tasks.
