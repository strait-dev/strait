Scheduling lets Strait create runs without a manual API call. Use it for recurring jobs, delayed work, and workloads that should only run during specific hours.

## Cron Schedules

Add a `cron` expression to a job or workflow to run it automatically.

```json
{
  "name": "Daily Report",
  "slug": "daily-report",
  "cron": "0 9 * * 1-5",
  "timezone": "America/New_York"
}
```

Strait uses standard five-field cron expressions:

| Expression | Runs |
| :--- | :--- |
| `0 * * * *` | Every hour |
| `*/15 * * * *` | Every 15 minutes |
| `0 9 * * 1-5` | Weekdays at 9:00 AM |
| `30 2 1 * *` | 2:30 AM on the first day of each month |

If no timezone is set, schedules run in UTC.

<Note>
  A disabled job does not create scheduled runs.
</Note>

## Delayed Runs

Set `scheduled_at` when triggering a job to create a run for a future time.

```bash
curl -X POST https://api.strait.dev/v1/jobs/send-email/trigger \
  -H "Authorization: Bearer strait_..." \
  -H "Content-Type: application/json" \
  -d '{
    "payload": {
      "to": "customer@example.com",
      "template": "trial-ending"
    },
    "scheduled_at": "2026-05-08T14:00:00Z"
  }'
```

Use delayed runs for reminders, follow-ups, billing checks, and other one-time work that should happen later.

## Execution Windows

Use `execution_window_cron` when a job may be triggered at any time but should only run during approved windows.

```json
{
  "execution_window_cron": "0 9-17 * * 1-5"
}
```

This example keeps execution inside weekday business hours. Runs triggered outside the window stay queued until the next allowed time.

## Overlap Policies

Use `cron_overlap_policy` to choose what happens when a scheduled job fires while an earlier run is still active.

| Policy | Behavior |
| :--- | :--- |
| `allow` | Create a new run every time the schedule fires. |
| `skip` | Skip this tick if another run is still active. |
| `cancel_running` | Cancel the active run, then create the new one. |

```json
{
  "cron": "*/5 * * * *",
  "cron_overlap_policy": "skip"
}
```

Use `skip` for maintenance jobs, syncs, and reports where only the latest scheduled run matters.

Use `allow` when each tick represents separate work that must run.

### Overlap Policies and Singletons

Overlap policies are enforced through [singletons](/concepts/singletons): a non-`allow` policy is treated as a per-job singleton with a fixed key. `skip` maps to the `drop` on-conflict policy and `cancel_running` maps to `replace`, so an overlapping cron fire is resolved against the singleton lock table atomically with run creation rather than by a separate check.

This has two consequences:

- If a job sets both an explicit singleton configuration and a `cron_overlap_policy`, the singleton configuration wins and the overlap policy is ignored. Prefer configuring [singletons](/concepts/singletons) directly; `cron_overlap_policy` remains supported as a shorthand.
- `cancel_running` cancels the single active key holder rather than every active run, which matches the singleton `replace` semantics.

## Workflow Schedules

Workflows can also run on cron schedules. Use `skip_if_running` to avoid overlapping workflow runs.

```json
{
  "cron": "0 2 * * *",
  "cron_timezone": "UTC",
  "skip_if_running": true
}
```

This is useful for long-running workflows where a second copy would produce duplicate work. Like job overlap policies, `skip_if_running` is enforced as a cron-only singleton: it maps to the `drop` policy for scheduled fires and does not affect workflows triggered directly through the API. An explicit workflow singleton configuration takes precedence.
