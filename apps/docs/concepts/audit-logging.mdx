title: "Audit Logging"
description: "Track every change to your jobs, workflows, runs, and settings."
---

Audit logging records every change to your jobs, workflows, runs, and settings. Strait writes audit events for platform integrity. In Strait Cloud, customer-facing audit log access, export, and chain verification require Scale or above. Self-hosted deployments have no billing gate.

## What Gets Logged

Every API mutation is recorded with who performed it, what changed, and when:

- Job created, updated, or deleted
- Workflow created, updated, or deleted
- Run triggered, cancelled, or state changed
- API key created or rotated
- Team member or role changes
- Webhook subscription changes
- Secret access (reads and writes)

### Actor Identity

Each event captures the actor:

- **Users**: identity from JWT or OIDC claims (dashboard or CLI actions).
- **Service accounts**: identity from the API key used.
- **System**: actions started by Strait or trusted backend calls.

## Querying Audit Logs

List audit events for a project via the API:

```bash
curl "https://your-strait-instance/v1/audit-events?limit=25" \
  -H "Authorization: Bearer strait_..."
```

Each event includes the action, resource type, resource ID, actor, timestamp, and request metadata (IP address, user agent, request ID, trace ID).

## Integrity

Every audit event is signed and linked to the previous event in the same project. This creates a tamper-evident log. Any insertion, deletion, or modification breaks the chain.

Verify the chain at any time:

```bash
curl "https://your-strait-instance/v1/audit-events/verify" \
  -H "Authorization: Bearer strait_..."

# Incremental verification (from last verified point)
curl "https://your-strait-instance/v1/audit-events/verify?incremental=true" \
  -H "Authorization: Bearer strait_..."
```

Key rotation is supported via `POST /v1/projects/{id}/audit/rotate-key`. Existing audit history remains verifiable after rotation.

## Forwarding to SIEM

Operators can forward persisted audit events to an external security platform in real time by setting `AUDIT_SIEM_ENDPOINT` on the Strait server. Events are batched as newline-delimited JSON.

This is infrastructure configuration for self-hosted or managed operations. It is not a launch self-serve plan entitlement in Strait Cloud, and should not be sold as active SIEM export.

| Variable | Default | Description |
| :--- | :--- | :--- |
| `AUDIT_SIEM_ENDPOINT` | _(disabled)_ | HTTP(S) URL to POST batched events to. |
| `AUDIT_SIEM_BATCH_SIZE` | `100` | Events per batch. |
| `AUDIT_SIEM_FLUSH_INTERVAL` | `10s` | Max time between flushes. |

## Export

Export audit events via `GET /v1/audit-events/export` in JSON, NDJSON, or CSV format. In Strait Cloud, export requires Scale or above. Exports are rate-limited to 10 per project per hour. The response includes an `X-Audit-Signature` header for integrity verification.

## Retention

Audit events are automatically pruned after a configurable retention window. The server-wide default is 365 days.

Per-project overrides are available:

- `PUT /v1/projects/{id}/audit/retention` with body `{"days": 90}`: set a custom window.
- Set `days` to `0` to disable pruning (events are never deleted).
- `GET /v1/projects/{id}/audit/retention`: check the current setting.

## Configuration Reference

| Variable | Default | Description |
| :--- | :--- | :--- |
| `SECRET_ENCRYPTION_KEY` | _(required)_ | Base key used to derive the audit signing key. |
| `AUDIT_RETENTION_DEFAULT_DAYS` | `365` | Default retention window in days. |
| `AUDIT_SIEM_ENDPOINT` | _(disabled)_ | SIEM forwarding target URL. |
| `AUDIT_SIEM_BATCH_SIZE` | `100` | Events per SIEM batch. |
| `AUDIT_SIEM_FLUSH_INTERVAL` | `10s` | Max interval between SIEM flushes. |
