Audit logging records every change to your jobs, workflows, runs, and settings. Use it to investigate incidents, meet compliance requirements, or understand who changed what. Audit logging is enabled by default.

## What Gets Logged

Every API mutation is recorded with who performed it, what changed, and when:

- Job created, updated, or deleted
- Workflow created, updated, or deleted
- Run triggered, cancelled, or state changed
- API key created or rotated
- Team member or role changes
- Webhook subscription changes
- Secret access (reads and writes)

### Actor Identity

Each event captures the actor:

- **Users** — identity from JWT or OIDC claims (dashboard or CLI actions).
- **Service accounts** — identity from the API key used.
- **Internal** — system-initiated actions or service-to-service calls.

## Querying Audit Logs

List audit events for a project via the API:

```bash
curl "https://strait.example.com/v1/audit-events?project_id=proj_1" \
  -H "Authorization: Bearer strait_..."
```

Each event includes the action, resource type, resource ID, actor, timestamp, and request metadata (IP address, user agent, request ID, trace ID).

## Integrity

Every audit event is signed with HMAC-SHA256 and chained to the previous event in the same project. This creates a tamper-evident log — any insertion, deletion, or modification breaks the cryptographic chain.

Verify the chain at any time:

```bash
curl "https://strait.example.com/v1/audit-events/verify?project_id=proj_1" \
  -H "Authorization: Bearer strait_..."

# Incremental verification (from last verified point)
curl "https://strait.example.com/v1/audit-events/verify?project_id=proj_1&incremental=true" \
  -H "Authorization: Bearer strait_..."
```

Key rotation is supported via `POST /v1/audit-events/rotate-key`. A tombstone event marks the boundary so verification spans key generations.

## Forwarding to SIEM

Forward audit events to an external security platform in real time by setting `AUDIT_SIEM_ENDPOINT` to the target URL. Events are batched as newline-delimited JSON.

| Variable | Default | Description |
| :--- | :--- | :--- |
| `AUDIT_SIEM_ENDPOINT` | _(disabled)_ | HTTP(S) URL to POST batched events to. |
| `AUDIT_SIEM_BATCH_SIZE` | `100` | Events per batch. |
| `AUDIT_SIEM_FLUSH_INTERVAL` | `5s` | Max time between flushes. |

## Export

Export audit events via `GET /v1/audit-events/export` in JSON, NDJSON, or CSV format. Exports are rate-limited to 10 per project per hour. The response includes an `X-Audit-Signature` header for integrity verification.

## Retention

Audit events are automatically pruned after a configurable retention window. The server-wide default is 365 days.

Per-project overrides are available:

- `PUT /v1/projects/{id}/audit/retention` with body `{"days": 90}` — set a custom window.
- Set `days` to `0` to disable pruning (events are never deleted).
- `GET /v1/projects/{id}/audit/retention` — check the current setting.

## Configuration Reference

| Variable | Default | Description |
| :--- | :--- | :--- |
| `SECRET_ENCRYPTION_KEY` | _(required)_ | Base key used to derive the audit signing key. |
| `AUDIT_RETENTION_DEFAULT_DAYS` | `365` | Default retention window in days. |
| `AUDIT_ASYNC_BUFFER_SIZE` | `8192` | Capacity of the async emit channel. |
| `AUDIT_SIEM_ENDPOINT` | _(disabled)_ | SIEM forwarding target URL. |
| `AUDIT_SIEM_BATCH_SIZE` | `100` | Events per SIEM batch. |
| `AUDIT_SIEM_FLUSH_INTERVAL` | `5s` | Max interval between SIEM flushes. |
