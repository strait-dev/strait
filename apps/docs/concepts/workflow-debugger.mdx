The workflow debugger shows what happened during a workflow run: which steps ran, how long they took, what data moved between them, and where a failure started.

The debug view comes from `GET /v1/workflow-runs/{workflowRunID}/debug`, which returns a timeline with per-step status, timing, cost, input and output, and the data flow between steps.

## What You See

The run view shows each step:

- **Status**: completed, failed, pending, waiting, running, skipped, or canceled.
- **Timing**: when each step started, finished, and how long it took.
- **Input and output**: JSON payloads passed into and returned from each step.
- **Error details**: error message and class for failed steps.
- **Attempt count**: which retry attempt this execution represents.
- **Data flow**: what data passed between dependent steps, with byte sizes.

The workflow debugger works for both completed and in-progress runs. Running workflows show pending and waiting steps with their current status.

## Run Comparison

Compare any two runs of the same workflow with `GET /v1/workflow-runs/{workflowRunID}/compare/{otherRunID}` to find what changed:

- **Status differences**: steps that reached different final states.
- **Duration differences**: timing changes per step.
- **Missing steps**: steps present in only one run, often because of a condition.

Unchanged steps are hidden from the comparison so you can focus on what changed.

## Data Flow Tracking

Data flow shows which step output feeds another step:

```json
{
  "data_flow": [
    {"from_step_ref": "validate", "to_step_ref": "charge", "data_size_bytes": 256},
    {"from_step_ref": "charge", "to_step_ref": "notify", "data_size_bytes": 128}
  ]
}
```

Use this to trace payload changes and find steps producing unexpectedly large outputs.
