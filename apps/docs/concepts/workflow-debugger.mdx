The workflow debugger shows you exactly what happened during a workflow run — which steps ran, how long each took, what data flowed between them, and where things went wrong.

## What You See

The debug view assembles a timeline of every step in a run:

- **Status** — completed, failed, pending, waiting, running, skipped, or canceled
- **Timing** — when each step started, finished, and how long it took
- **Input/Output** — full JSON payloads passed into and returned from each step
- **Error details** — error message and classification for failed steps
- **Attempt count** — which retry attempt this execution represents
- **Data flow edges** — arrows showing what data passed between dependent steps, with byte sizes

The debug view works for both completed and in-progress runs. Running workflows show pending and waiting steps with their current status.

## Run Comparison

Compare any two runs of the same workflow to find what changed:

- **Status diff** — highlights steps that reached different terminal states
- **Duration diff** — timing differences per step, useful for spotting performance regressions
- **Missing steps** — steps present in only one run, common with conditional branches

Steps with identical status and timing are omitted from the diff to focus on what actually changed.

## Data Flow Tracking

Data flow edges show which step's output feeds into which step's input:

```json
{
  "data_flow": [
    {"from_step_ref": "validate", "to_step_ref": "charge", "data_size_bytes": 256},
    {"from_step_ref": "charge", "to_step_ref": "notify", "data_size_bytes": 128}
  ]
}
```

Use this to trace how data transforms as it moves through the workflow, and to identify steps producing unexpectedly large payloads.
