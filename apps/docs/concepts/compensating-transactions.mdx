description: "Undo completed workflow steps when a later step fails."

When a later workflow step fails, earlier steps may need to be undone.

Compensating transactions let you attach an undo job to a step. If the workflow fails, Strait runs the undo jobs for completed steps.

## When to Use Compensation

Use compensation when workflow steps have side effects that may need to be reversed:

- **Payments**: Charge a card, then something fails downstream. Refund the charge.
- **Reservations**: Reserve inventory or capacity, then a later step fails. Release the reservation.
- **External API calls**: Create a resource in a third-party system, then delete it if a later step fails.

If a step is read-only, it usually does not need a compensation job.

## Example: Order Fulfillment Saga

```json
{
  "steps": [
    {
      "step_ref": "create-order",
      "job_id": "order-create",
      "compensation_job_id": "order-cancel",
      "compensation_timeout_secs": 30
    },
    {
      "step_ref": "charge-card",
      "job_id": "payment-charge",
      "depends_on": ["create-order"],
      "compensation_job_id": "payment-refund",
      "compensation_timeout_secs": 30
    },
    {
      "step_ref": "reserve-inventory",
      "job_id": "inventory-reserve",
      "depends_on": ["charge-card"],
      "compensation_job_id": "inventory-release",
      "compensation_timeout_secs": 30
    },
    {
      "step_ref": "send-confirmation",
      "job_id": "notifications-send",
      "depends_on": ["reserve-inventory"]
    }
  ]
}
```

If `reserve-inventory` fails after all retries, Strait:

1. Runs `payment-refund` (with the output from `charge-card` as input)
2. Runs `order-cancel` (with the output from `create-order` as input)

Only completed steps with a `compensation_job_id` are undone. `send-confirmation` has no compensation job, so Strait skips it.

## Failure Behavior

1. A workflow step fails after retries are exhausted.
2. Strait finds completed steps with compensation jobs.
3. Each compensation job receives the original step output.
4. The workflow moves to a terminal compensation state.

## Workflow Run States

| State | Meaning |
| :--- | :--- |
| `compensating` | Compensation jobs are running. |
| `compensated` | All compensation jobs completed. Terminal. |
| `compensation_failed` | At least one compensation job failed. Terminal. Requires manual intervention. |

Common flow: `failed` or `timed_out` -> `compensating` -> `compensated` or `compensation_failed`.

<Warning>
Compensation jobs should be idempotent. If a compensation job fails, the workflow moves to `compensation_failed` and needs review.
</Warning>

## Configuration Reference

| Field | On | Description |
| :--- | :--- | :--- |
| `compensation_job_id` | Step | The job to run as the undo action for this step. |
| `compensation_timeout_secs` | Step | Timeout for the compensation job. |
| `on_failure` | Step | Set to `"compensate"` to trigger compensation on step failure. Default is `"fail_workflow"`. |
