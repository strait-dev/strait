When step 3 of your workflow fails, you need to undo steps 1 and 2. Compensating transactions define undo logic for each step so Strait can automatically roll back completed work on failure.

## When to Use Compensation

Any workflow where steps have real-world side effects that need reversal:

- **Payments**: Charge a card, then something fails downstream -- refund the charge.
- **Reservations**: Reserve inventory or capacity, then a later step fails -- release the reservation.
- **External API calls**: Create a resource in a third-party system, then roll back -- delete it.

If a step is read-only or idempotent, it does not need a compensation handler.

## Example: Order Fulfillment Saga

```json
{
  "steps": [
    {
      "step_ref": "create-order",
      "job_id": "order-create",
      "compensation_job_id": "order-cancel",
      "compensation_timeout_secs": 30
    },
    {
      "step_ref": "charge-card",
      "job_id": "payment-charge",
      "depends_on": ["create-order"],
      "compensation_job_id": "payment-refund",
      "compensation_timeout_secs": 30
    },
    {
      "step_ref": "reserve-inventory",
      "job_id": "inventory-reserve",
      "depends_on": ["charge-card"],
      "compensation_job_id": "inventory-release",
      "compensation_timeout_secs": 30
    },
    {
      "step_ref": "send-confirmation",
      "job_id": "notifications-send",
      "depends_on": ["reserve-inventory"]
    }
  ]
}
```

If `reserve-inventory` fails after all retries, Strait automatically:

1. Runs `payment-refund` (with the output from `charge-card` as input)
2. Runs `order-cancel` (with the output from `create-order` as input)

Steps are compensated in reverse dependency order -- latest-completed first. `send-confirmation` has no `compensation_job_id`, so it is skipped. Only steps that actually completed are compensated.

## How It Works

1. A workflow step fails after exhausting retries.
2. Strait collects all completed steps that have a `compensation_job_id`.
3. Compensation jobs run in reverse dependency order, each receiving the original step's output as input.
4. The workflow run transitions through compensation states.

## Workflow Run States

| State | Meaning |
| :--- | :--- |
| `compensating` | Compensation jobs are running. |
| `compensated` | All compensation jobs completed. Terminal. |
| `compensation_failed` | At least one compensation job failed. Terminal. Requires manual intervention. |

Valid transitions: `failed` or `timed_out` ã `compensating` ã `compensated` or `compensation_failed`.

<Warning>
Compensation jobs must be idempotent. If a compensation job fails, the run moves to `compensation_failed` and you will need to resolve it manually.
</Warning>

## Configuration Reference

| Field | On | Description |
| :--- | :--- | :--- |
| `compensation_job_id` | Step | The job to run as the undo action for this step. |
| `compensation_timeout_secs` | Step | Timeout for the compensation job. |
| `on_failure` | Step | Set to `"compensate"` to trigger compensation on step failure. Default is `"fail_workflow"`. |
