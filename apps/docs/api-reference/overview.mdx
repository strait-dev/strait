## Authentication

The Strait API supports four authentication methods depending on the client and use case.

### Server Secret
Used for trusted backend calls.
- **Header**: `X-Internal-Secret: <your-secret>`
- **Bearer Token**: `Authorization: Bearer <your-secret>`

### API Keys
Used for project-specific operations. API keys are prefixed with `strait_`.
- **Header**: `Authorization: Bearer strait_...`

### OIDC Bearer Tokens (Optional)
Used for user API access when OIDC bearer verification is enabled. This is not hosted dashboard SSO/SAML.
- **Header**: `Authorization: Bearer <oidc-token>`

### JWT Run Tokens
Used by the SDK within a job run context to authenticate heartbeats, logs, and checkpoints.
- **Header**: `Authorization: Bearer <jwt-token>`

## Error Response Format

All API errors use the envelope described in [Error Envelope](#error-envelope). SDKs branch on `error.code`; operators should quote `request_id` when reporting failures.

## Pagination

List endpoints use cursor-based pagination.
- **Query Parameters**:
  - `limit`: Number of items to return (default: 50, max: 100).
  - `cursor`: The cursor for the next page (usually a timestamp or ID).
- **Response**: Includes a `next_cursor` field if more results are available.

## Rate Limiting

Cloud API rate limits are plan-based when the request resolves to a project or API key. API-key and project quota overrides can lower the plan cap, but cannot raise it above the plan limit.

- **Free**: 60 requests per minute.
- **Starter**: 300 requests per minute.
- **Pro**: 1,000 requests per minute.
- **Scale**: 3,000 requests per minute.
- **Business and Enterprise**: unlimited by plan.

When no project or API key limit can be resolved, Strait falls back to the server-wide IP limit configured by `RATE_LIMIT_REQUESTS` and `RATE_LIMIT_WINDOW`.

Additional endpoint-specific limits:
- **Job Trigger**: 10 triggers per minute.
- **Bulk Trigger**: 5 bulk trigger requests per minute.

## Request Limits

- **Body Size**: 1MB default limit for JSON payloads.
- **Content-Type**: `application/json` is required for all POST/PATCH requests.

## Common Query Parameters

- `project_id`: Required for most list endpoints to scope results to a specific project.
- `status`: Used on run and workflow-run listings to filter by current state.

|Category|Base Path|Description|
|---|---|---|
|Jobs|`/v1/jobs`|Create, update, trigger, and manage job definitions|
|Runs|`/v1/runs`|Track, cancel, retry, and inspect job runs|
|Workflows|`/v1/workflows`|Define and manage workflows|
|Workflow Runs|`/v1/workflow-runs`|Monitor and control workflow executions|
|Events|`/v1/events`|Send events to triggers and dispatch to sources|
|Event Sources|`/v1/event-sources`|Manage inbound event channels and subscriptions|
|Projects|`/v1/projects`|Create and manage projects and quotas|
|Environments|`/v1/environments`|Configure per-environment variable overrides|
|API Keys|`/v1/api-keys`|Create, rotate, and revoke API keys|
|Webhook Subscriptions|`/v1/webhooks/subscriptions`|Manage project-wide webhook subscriptions|
|Audit|`/v1/audit-events`|Query and export audit events|
|RBAC|`/v1/roles`, `/v1/members`|Manage roles, members, and policies|

All endpoints accept and return JSON. Use the interactive API explorer below for full request/response schemas.

  "id": "job_abc123",
  "name": "Send Welcome Email",
  "slug": "send-welcome-email",
  "created_at": "2026-03-15T10:00:00Z"

## Error Envelope

Every error response follows the same shape:

```json
{
  "error": {
    "code": "validation_failed",
    "message": "validation failed",
    "details": ["Field 'name' is required"]
  },
  "request_id": "req_abc123"
}
```

- `error.code`: machine-readable canonical code (see table below). SDKs branch on this field.
- `error.message`: human-readable summary suitable for logs.
- `error.details`: optional array of contextual messages (per-field validation errors, downstream causes).
- `request_id`: present when the request reached the router; quote this when filing support tickets.

## Error Codes

|Code|HTTP Status|Description|
|---|---|---|
|`bad_request`|400|Malformed JSON, invalid value, or other request-level problem.|
|`authentication_required`|401|Missing or invalid authentication.|
|`forbidden`|403|Authenticated but insufficient permissions.|
|`not_found`|404|The requested resource does not exist.|
|`conflict`|409|Idempotency key conflict: a run with this key already exists.|
|`validation_failed`|422|Request body failed schema or field-level validation. See `details`.|
|`rate_limited`|429|Too many requests. Wait and retry.|
|`enqueue_throttled`|429|Per-job or per-project enqueue throttle exceeded. Retry with backoff.|
|`internal_error`|500|Unexpected server-side failure. Quote `request_id` when reporting.|
|`service_unavailable`|503|Dependency degraded or shedding load. Retry with backoff.|

## Workflow and Dependency Endpoints

### Planning and Diagnostics

- `POST /v1/workflows/{workflowID}/plan`: Topological plan preview for the current workflow version (supports step overrides).
- `GET /v1/runs/{runID}/dependency-status`: Returns dependency definitions for a run and whether it is currently unblocked.
- `GET /v1/workflow-runs/{workflowRunID}/graph`: Workflow graph for a run.
- `GET /v1/workflow-runs/{workflowRunID}/explain`: Scheduler/condition decision stream with optional `step_ref` and `decision_type` filters.

### Recovery and Governance

- `POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/retry`: Reset one terminal step to `pending` and resume progression.
- `POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/replay-subtree`: Reset the selected step and descendants, then resume progression.
- `GET /v1/workflow-policies/{projectID}` / `PUT /v1/workflow-policies/{projectID}`: Read and update project-level workflow policy.

### Bulk Operations

- `POST /v1/jobs/{jobID}/trigger/bulk`: Trigger up to the configured `MAX_BULK_TRIGGER_ITEMS` limit in a single request. Supports per-item payload, dispatch priority within the shared platform range, idempotency keys, and tags. This `priority` field is not the roadmap paid priority-queue entitlement. Returns structured results with per-item status. Rate limited to 5 requests per minute.
- `POST /v1/runs/bulk-cancel`: Cancel up to 100 runs in a single request. Returns per-run cancel status with error details for runs that could not be canceled.

### Version Analysis

- `POST /v1/workflows/{workflowID}/simulate`: Returns snapshot ordering metadata (`predicted_order`, `step_count`).
- `GET /v1/workflows/{workflowID}/versions/{fromVersionID}/diff/{toVersionID}`: Added/removed step refs between versions.
- `GET /v1/workflows/{workflowID}/versions/{versionID}/impact`: How many sampled runs currently match a version.

## Example Requests

```bash
# Runtime graph snapshot
curl -X GET "https://api.strait.dev/v1/workflow-runs/wr_abc123/graph" \
  -H "Authorization: Bearer strait_live_abc123"
```

```bash
# Explain why a step is waiting/skipped
curl -X GET "https://api.strait.dev/v1/workflow-runs/wr_abc123/explain?step_ref=deploy&decision_type=resource" \
  -H "Authorization: Bearer strait_live_abc123"
```

```bash
# Replay a failed branch from step 'deploy'
curl -X POST "https://api.strait.dev/v1/workflow-runs/wr_abc123/steps/deploy/replay-subtree" \
  -H "Authorization: Bearer strait_live_abc123"
```
