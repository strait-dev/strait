Strait runs background jobs and multi-step workflows. You define the work, where it runs, and when it should happen. Strait handles queueing, retries, schedules, dependencies, durable waits, and run history.

Most teams should start with Strait Cloud. Strait hosts the control plane at `https://api.strait.dev`; your job code stays in your infrastructure and receives HTTP requests from Strait, or connects back through the worker plane. Use the community self-hosted edition when you need to run the control plane yourself.

## What You Can Build

- Background jobs for imports, reports, email, cleanup, cache warming, and maintenance.
- Scheduled jobs with cron expressions, time zones, retries, and history.
- Event-driven jobs that process external events with replay for failed work.
- Workflows where one step waits for another step, event, approval, or long-running process.
- Operational recovery flows for failed runs, `dead_letter` review, and replay.

## How Strait Runs Work

1. Create a project in Strait Cloud or your self-hosted instance.
2. Create a job with an `endpoint_url` or `execution_mode: "worker"`.
3. Trigger the job from your app, an SDK, a schedule, or an event source.
4. Strait records a run, dispatches the work, retries failures, and stores the result.
5. You inspect, stream, replay, or recover runs from the API or dashboard.

A normal run moves from `queued` to `executing` to `completed`. If the endpoint fails and retries are exhausted, the run moves to `dead_letter` so you can inspect and replay it after the underlying problem is fixed.

## Cloud And Community

| Need | Use |
|---|---|
| Fastest production setup, hosted upgrades, hosted run history, and billing controls | Strait Cloud |
| Local evaluation before creating a Cloud account | Community self-host with Docker Compose |
| Compliance, residency, air-gap, or full operational control requirements | Community self-host or a dedicated Cloud plan |
| Job code that must stay inside your own network | Cloud with private workers, or community self-host |

The API shape is the same across editions. The base URL and setup steps change.
