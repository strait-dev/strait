This report summarizes the local benchmark pass for the PgQue-inspired queue work. The latest run was captured on May 24, 2026 on an Apple M1 Pro with testcontainer Postgres.

## Result

Do not switch `QUEUE_ENGINE` or `OUTBOX_ENGINE` to `batchlog` by default yet. The optimized batchlog queue is now close to the fully denormalized dequeue path in this local benchmark, and it should generate less `job_runs` churn, but it did not beat the current fastest dequeue path. The optimized batchlog outbox improved substantially after moving archival off the promotion path, but it is still slower than the legacy flusher benchmark.

Keep `QUEUE_USE_DENORMALIZED_DEQUEUE=true` as the production default. Continue optimizing batchlog before removing the remaining engine selection knobs.

## Optimized Follow-up Measurements

| Area | Variant | Result |
|---|---:|---:|
| Queue claim benchmark | batchlog after hot-path maintenance removal | 38.35 ms/op |
| Queue claim benchmark | legacy `DequeueN` | 10.01 ms/op |
| Queue claim benchmark | denormalized | 4.22 ms/op |
| Queue claim benchmark | fully denormalized | 36.61 ms/op |
| Outbox promotion after write-time claims and async archive | legacy | 192.58 ms/op |
| Outbox promotion after write-time claims and async archive | batchlog | 282.15 ms/op |

The queue batchlog path improved from 483.45 ms/op to 38.35 ms/op in the local benchmark, mainly by moving backfill, sealing, and lease reclamation out of `DequeueN` and maintaining queue entries at write time. The outbox batchlog path improved from 397.63 ms/op after write-time claims to 282.15 ms/op after removing immediate archive from the flusher transaction, but still trails legacy.

The phase 6 gate was run with packages serialized to avoid local testcontainer contention. Results were noisier than the shorter package-level runs, but the conclusion is unchanged: batchlog queue/outbox should not become the default until the remaining claim overhead is removed.

## Reliability Checks

Passed targeted checks covered:

- no duplicate claims under concurrent batchlog queue workers.
- batchlog lease expiry redelivery before worker start.
- direct `queued -> executing` transition for batchlog queue starts.
- batchlog lease counts blocking additional claims while a run is leased but not yet executing.
- outbox batchlog retryable release, terminal quarantine, concurrent flusher ownership, write-time claims, empty-batch avoidance, and async promoted-row archive.
- workflow progression event claim, dedupe, ack, and fuzzed grouping behavior.

## Rollout

Recommended flags:

```bash
QUEUE_USE_DENORMALIZED_DEQUEUE=true
QUEUE_ENGINE=legacy
OUTBOX_ENGINE=legacy
WORKFLOW_PROGRESSION_ENGINE=legacy
```

Current batchlog experiments:

```bash
QUEUE_ENGINE=batchlog
OUTBOX_ENGINE=batchlog
WORKFLOW_PROGRESSION_ENGINE=batchlog
```

Rollback is immediate: set each engine value back to `legacy` and restart workers/API processes. The batchlog tables are additive and can remain in place while the legacy paths run.

## Follow-ups

- Reduce batchlog queue claim latency further, especially around batch ordering and lease-count joins.
- Reduce outbox batchlog claim overhead or remove the remaining `enqueue_outbox` join from the claim path.
- Add dashboards for `queue_entries`, `outbox_claims`, and `workflow_progression_events` ready/leased/acked counts, plus lease age.
