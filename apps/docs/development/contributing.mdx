This page covers the developer workflow for contributing to Strait. It mirrors the key sections of [`CONTRIBUTING.md`](https://github.com/strait-dev/strait/blob/master/CONTRIBUTING.md) in the repo root — that file is the canonical source. When in doubt, defer to it.

## Prerequisites

Install these before you begin:

| Tool | Version | Purpose |
|---|---|---|
| [Go](https://go.dev/dl/) | 1.26+ | API server, worker, scheduler |
| [Bun](https://bun.sh/) | 1.3+ | Frontend tooling, monorepo scripts |
| [Docker](https://docs.docker.com/get-docker/) | with Compose v2 | Postgres, Redis, Sequin (CDC) |
| [golangci-lint](https://golangci-lint.run/welcome/install/) | latest | Go linting |
| [lefthook](https://github.com/evilmartians/lefthook) | latest | Git hooks (pre-commit, pre-push, commit-msg) |

<Info>
  If you just want to try Strait without a full dev setup, see [Self-hosting](/self-hosting) for the quickest path to a running instance.
</Info>

## Setup

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/strait-dev/strait.git
    cd strait
    ```
  </Step>

  <Step title="Install frontend dependencies">
    ```bash
    bun install
    ```
  </Step>

  <Step title="Install git hooks">
    ```bash
    lefthook install
    ```
    Lefthook enforces Conventional Commits, runs gitleaks, Biome formatting, type-checking, and lint on every push. Never bypass hooks with `--no-verify`.
  </Step>

  <Step title="Start infrastructure">
    ```bash
    cd apps/strait
    docker compose up -d
    ```
    This starts PostgreSQL, Redis, and Sequin (CDC). Dev-compose maps Postgres to port **15432** and Redis to **16379** to avoid collisions with services already running on your machine.
  </Step>

  <Step title="Configure environment">
    ```bash
    cd apps/strait
    cp .env.example .env
    ```
    The defaults point at the dev compose ports:
    ```
    DATABASE_URL=postgres://strait:strait@localhost:15432/strait?sslmode=disable
    REDIS_URL=redis://localhost:16379
    ```
    Fill in `INTERNAL_SECRET` and `JWT_SIGNING_KEY` (each 32+ characters). See [Environment variables](/configuration/environment-variables) for the full reference.
  </Step>
</Steps>

## Running locally

<Tabs>
  <Tab title="API server">
    ```bash
    cd apps/strait
    go run ./cmd/strait
    ```
    - API: `http://localhost:8080`
    - OpenAPI reference: `http://localhost:8080/reference`

    Migrations are embedded in the binary and auto-applied on startup.
  </Tab>

  <Tab title="Dashboard">
    ```bash
    cd apps/app
    bun dev
    ```
    Dashboard at `http://localhost:5173` (TanStack Start / React).
  </Tab>

  <Tab title="Docs site">
    ```bash
    cd apps/docs
    bun dev
    ```
  </Tab>
</Tabs>

## Testing

All test commands run from `apps/strait`:

```bash
cd apps/strait

# Unit tests
go test ./... -count=1 -timeout=2m

# Unit tests with race detector
go test ./... -race -timeout=5m

# Integration tests (requires running Postgres + Redis via docker compose)
go test -tags=integration ./internal/store/... ./internal/queue/... ./internal/e2e/...
```

<Warning>
  Integration tests talk to real Postgres and Redis via `testcontainers-go`. Make sure Docker is running and the dev compose stack is up before running them.
</Warning>

Frontend tests:

```bash
cd apps/app
bun test
```

## Linting

```bash
# Go
cd apps/strait
golangci-lint run --timeout=10m ./...

# Frontend
cd apps/app
bun run biome:lint

# All pre-commit checks at once
lefthook run pre-commit
```

<Tip>
  The `--timeout=10m` flag is intentional. The full lint suite with all enabled linters needs the headroom, especially on first run before the cache is warm.
</Tip>

## Commit conventions

Strait uses [Conventional Commits](https://www.conventionalcommits.org/). The `commit-msg` hook enforces this.

```
type(scope): summary
```

**Allowed types:** `feat`, `fix`, `docs`, `test`, `refactor`, `perf`, `build`, `ci`, `chore`, `revert`.

Examples:

```
feat(api): add bulk trigger endpoint
fix(worker): prevent race in managed dispatch
test(store): add integration tests for job dependencies
```

Append `!` after the type for breaking changes and explain in the commit body.

**Do not** add AI attribution lines, "Co-Authored-By" headers, or vague messages like `update` or `fix stuff`.

## Pull request guidelines

<Steps>
  <Step title="Branch from master">
    Create a feature branch off `master`.
  </Step>

  <Step title="Make changes">
    Keep changes small, focused, and reversible. Tests ship in the same PR as the behavior change.
  </Step>

  <Step title="Validate">
    Run the full check suite before pushing:
    ```bash
    cd apps/strait
    go build ./...
    go test ./...
    go test -race ./...
    golangci-lint run --timeout=10m ./...
    ```
    Add integration tests (`go test -tags integration ./...`) when your change touches DB, queue, workflow, scheduler, or pubsub code.
  </Step>

  <Step title="Open the PR">
    Write a substantive description covering:
    - **Summary** -- what the PR does in plain language
    - **Why** -- context and motivation
    - **What changed** -- grouped by area
    - **Validation** -- exact commands run and their outcomes
    - **Tests** -- what was added or updated, and why
    - **Docs / OpenAPI / env impact** -- or explicit "none"
    - **Risks and follow-ups**
  </Step>

  <Step title="CI">
    CI runs test, lint, and security checks automatically. Fix failures before requesting review.
  </Step>
</Steps>

<Info>
  If your change is large or touches multiple subsystems, consider opening an issue first to discuss the approach.
</Info>
