Strait is a Go monorepo. Every dependency earns its place. This page explains what we use, why we chose it, and what we considered instead.

## Language: Go

| | |
|---|---|
| **Version** | Go 1.26 |
| **Why** | Single static binary with no runtime dependencies. Predictable, low-tail-latency GC. First-class concurrency via goroutines and channels. Fast compilation. Strong stdlib for HTTP, crypto, and encoding. |
| **Considered** | Rust (higher complexity, slower iteration), Node.js/TypeScript (GC pauses, single-threaded without workers), Java (JVM startup cost, memory footprint). |

Go gives us a single `strait` binary that deploys identically everywhere -- from a developer laptop to a Kubernetes pod. Goroutines map naturally to our workload: one per job run, one per webhook delivery, one per SSE connection.

## HTTP: chi + huma

| | |
|---|---|
| **Packages** | `go-chi/chi/v5` v5.2.5, `danielgtaylor/huma/v2` v2.37.3 |
| **Why** | chi is a lightweight, idiomatic `net/http`-compatible router with composable middleware. huma layers on typed request/response handling and automatic OpenAPI 3.1 spec generation. |
| **Considered** | Fiber (fasthttp, breaks `net/http` compatibility), Echo (heavier, overlapping middleware story), hand-rolled (too much boilerplate for 250+ operations). |

chi gives us standard `http.Handler` compatibility -- every stdlib and third-party middleware works without adapters. huma eliminates the JSON marshal/unmarshal boilerplate and produces an OpenAPI spec from Go types, which drives our SDK generation and Scalar API reference.

```go
// routes.go -- chi router with huma adapter
r := chi.NewRouter()
r.Use(chimw.RequestID)
r.Use(otelchi.Middleware("strait", otelchi.WithChiRoutes(r)))
```

The middleware stack includes CORS (`go-chi/cors`), rate limiting (`go-chi/httprate`), OTel tracing (`otelchi`), Sentry error capture, and request logging -- all composed as standard chi middleware.

<Info>
The `huma_bridge.go` file provides a `TypedHandler` that extracts path/query params into typed Go structs, validates them, and returns typed responses -- bridging chi's `http.HandlerFunc` with huma's type system.
</Info>

## Database: pgx/v5 (No ORM)

| | |
|---|---|
| **Packages** | `jackc/pgx/v5` v5.9.2, `georgysavva/scany/v2` v2.1.4, `golang-migrate/migrate/v4` v4.19.1 |
| **Why** | Direct Postgres access via pgx gives us full control over SQL, connection pooling, LISTEN/NOTIFY, and advisory locks. No ORM abstraction hiding query behavior. |
| **Considered** | sqlx (simpler but less capable pool), GORM (hides SQL, unpredictable queries, hard to optimize), ent (schema-first codegen adds build complexity). |

Every query is hand-written SQL. The store layer defines a `DBTX` interface that accepts either a pool connection or a transaction:

```go
type DBTX interface {
    Exec(ctx context.Context, sql string, arguments ...any) (pgconn.CommandTag, error)
    Query(ctx context.Context, sql string, args ...any) (pgx.Rows, error)
    QueryRow(ctx context.Context, sql string, args ...any) pgx.Row
}
```

scany handles struct scanning from rows. Migrations are managed by `golang-migrate` with plain SQL files in `apps/strait/migrations/`. OTel instrumentation is wired in via `exaring/otelpgx` so every query is traced.

<Warning>
We intentionally avoid ORMs. When a query is slow, you read the SQL, not a chain of method calls. When Postgres adds a feature, you use it immediately -- no waiting for ORM support.
</Warning>

## Queue: Postgres SKIP LOCKED

| | |
|---|---|
| **Package** | None (raw SQL via pgx) |
| **Why** | `SELECT ... FOR UPDATE SKIP LOCKED` turns Postgres into a reliable, transactional job queue with zero additional infrastructure. Jobs are enqueued and dequeued in the same database that stores their metadata. |
| **Considered** | RabbitMQ (extra infra, separate failure domain), Redis queues (no transactional guarantees with job state), SQS (vendor lock-in, at-least-once only). |

The `queue.PostgresQueue` uses `FOR UPDATE SKIP LOCKED` to let multiple worker replicas dequeue concurrently without contention. Because the queue lives in the same Postgres transaction as the job state update, enqueue-and-update-status is atomic -- no two-phase commit, no distributed saga.

```sql
SELECT jr.id, jr.job_id, jr.status, ...
FROM job_runs jr
WHERE jr.status = 'queued'
ORDER BY jr.priority DESC, jr.created_at ASC
FOR UPDATE OF jr SKIP LOCKED
LIMIT 1
```

Backpressure is handled via a token bucket controller attached to the queue. An outbox pattern with `SKIP LOCKED` is used for webhook deliveries and event fan-out.

## Cache: Redis + otter + gocache

| | |
|---|---|
| **Packages** | `redis/go-redis/v9` v9.18.0, `maypok86/otter` v1.2.4, `eko/gocache/lib/v4` v4.2.3 |
| **Why** | Redis for distributed cache and pub/sub across replicas. otter for high-performance in-process caching (lock-free, W-TinyLFU eviction). gocache as a unified cache abstraction that layers them together. |
| **Considered** | Memcached (no pub/sub, no data structures), bigcache (no TTL variance, no eviction policy control), groupcache (read-only, no TTL). |

The `cache/otterstore` package implements gocache's store interface backed by otter, giving us variable-TTL, capacity-bounded local caching with an eviction callback. Redis handles cross-instance invalidation and pub/sub for real-time event streaming to SSE connections.

<Tip>
otter uses the W-TinyLFU admission policy, which provides near-optimal hit rates with minimal memory overhead -- significantly better than a plain LRU for mixed workloads.
</Tip>

## Concurrency: conc + pond

| | |
|---|---|
| **Packages** | `sourcegraph/conc` v0.3.1, `alitto/pond/v2` v2.7.1 |
| **Why** | conc provides structured concurrency with automatic panic recovery and error collection. pond provides a bounded worker pool with backpressure. Together they prevent goroutine leaks and unhandled panics. |
| **Considered** | Raw goroutines + errgroup (no panic safety, manual leak prevention), ants (less ergonomic API, no structured concurrency). |

The worker `Executor` uses conc's `WaitGroup` to fan out work with guaranteed panic recovery:

```go
import "github.com/sourcegraph/conc"

var wg conc.WaitGroup
wg.Go(func() { /* panic-safe work */ })
wg.Wait() // collects panics as errors
```

The `worker.Pool` wraps pond/v2 for bounded task execution with configurable queue size for backpressure:

```go
pool := pond.NewPool(concurrency, pond.WithQueueSize(queueSize))
```

When all slots and queue positions are full, `Submit` blocks until a slot opens or the context is canceled -- preventing runaway goroutine creation.

## Resilience: failsafe-go

| | |
|---|---|
| **Package** | `failsafe-go/failsafe-go` v0.9.6 |
| **Why** | Composable resilience policies (retries, circuit breakers, timeouts) as reusable building blocks. Type-safe generics. |
| **Considered** | sony/gobreaker (circuit breaker only), cenkalti/backoff (retries only, no composition), hand-rolled (duplicated logic across services). |

failsafe-go is used across multiple subsystems:

- **CDC client** -- retry policy + circuit breaker for change data capture HTTP calls
- **Log drains** -- retry policy + circuit breaker for external log forwarding
- **Webhook delivery** -- retry policy with exponential backoff and jitter
- **Notification webhooks** -- retry policy for alert delivery

```go
retryPolicy := retrypolicy.Builder[*http.Response]().
    WithMaxRetries(3).
    WithBackoff(time.Second, 30*time.Second).
    Build()

result, err := failsafe.With[WebhookResult](rp).Run(func() (WebhookResult, error) {
    // deliver webhook
})
```

The worker package also has a local `CircuitBreaker` for per-endpoint circuit breaking on webhook delivery, using a custom state machine (closed/open/half-open) stored in the database.

## Container Runtimes: client-go + Docker CLI

| | |
|---|---|
| **Packages** | `k8s.io/client-go` v0.36.0, `docker/cli` v29.4.1, `moby/buildkit` v0.29.0 |
| **Why** | client-go is the official Kubernetes Go client -- the only serious option for managing K8s Jobs, watching pod status, and streaming logs. Docker CLI/BuildKit for local development and image building. |
| **Considered** | CRI directly (too low-level), Nomad (less ecosystem support), custom container runtime (not worth the investment). |

The `compute` package provides two runtime backends behind a common interface:

- **`K8sRuntime`** -- creates Kubernetes Jobs with configurable resource limits, node pool affinity, TTL-based garbage collection, and label-based isolation. Uses in-cluster config or kubeconfig for auth.
- **`DockerRuntime`** -- shells out to Docker for local development, with image URI validation to prevent injection.

```go
// k8s.go -- creating a K8s Job
import (
    batchv1 "k8s.io/api/batch/v1"
    "k8s.io/client-go/kubernetes"
)
```

Both runtimes support multi-region routing via the `compute.Router`, which dispatches to the appropriate cluster based on the job's configured region.

## Observability: OTel + Prometheus + Pyroscope + Sentry

| | |
|---|---|
| **Packages** | `go.opentelemetry.io/otel` v1.43.0, `prometheus/client_golang` v1.23.2, `grafana/pyroscope-go` v1.2.8, `getsentry/sentry-go` v0.46.0 |
| **Why** | OTel for distributed tracing and log correlation (vendor-neutral). Prometheus for metrics (pull model, native Kubernetes integration). Pyroscope for continuous profiling. Sentry for error tracking with stack traces. |
| **Considered** | Datadog agent (vendor lock-in, cost), Jaeger directly (OTel is the standard now), pprof-only (no continuous profiling). |

The telemetry stack is layered:

<Steps>
  <Step title="Tracing">
    OTel with OTLP/HTTP export. Every HTTP request, database query (via otelpgx), and inter-service call carries trace context. The `otelchi` middleware injects spans per route.
  </Step>
  <Step title="Metrics">
    Prometheus client with OTel bridge. Custom metrics for queue depth, worker pool utilization, circuit breaker state, and endpoint health scores. Exported via `/metrics` endpoint.
  </Step>
  <Step title="Profiling">
    Pyroscope continuous profiling captures CPU, goroutine, allocation, mutex, and block profiles in production with negligible overhead.
  </Step>
  <Step title="Errors">
    Sentry captures panics (via `sentryhttp` middleware) and explicit error reports with full stack traces and request context.
  </Step>
  <Step title="Logging">
    `log/slog` with `lmittmann/tint` for human-readable local output and `samber/slog-multi` for multi-destination routing. OTel log bridge (`otelslog`) correlates logs with traces.
  </Step>
</Steps>

## Helpers: samber/lo + samber/oops

| | |
|---|---|
| **Packages** | `samber/lo` v1.53.0, `samber/oops` v1.21.0 |
| **Why** | lo provides generic slice/map utilities (Map, Filter, Contains, Uniq) that eliminate repetitive loops. oops adds structured error context with stack traces, component scoping, and error codes. |
| **Considered** | Hand-written helpers (duplicate code across packages), pkg/errors (archived, no generics), cockroachdb/errors (heavier than needed). |

lo is used sparingly in API handlers, store queries, and workflow logic for collection transforms:

```go
import "github.com/samber/lo"

ids := lo.Map(runs, func(r domain.WorkflowRun, _ int) string {
    return r.ID
})
```

oops is wrapped in a thin `internal/errors` package that provides `Wrap`, `Wrapf`, `New`, and `In` for component-scoped errors:

```go
import "strait/internal/errors"

// Scoped to a component for structured error context
return errors.In("worker").Wrapf(err, "dispatch run %s", runID)
```
