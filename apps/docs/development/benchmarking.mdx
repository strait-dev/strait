Strait ships a load test harness under `apps/strait/internal/loadtest`. It exercises the API, queue, worker pool, scheduler, and webhook pipeline end-to-end against a real Postgres and Redis. The harness is built with the `loadtest` build tag and is intended for local performance exploration, regression checks before performance-sensitive changes, and reference-machine baselines.

This page is informational. Numbers captured here are not gates in CI.

## Running a scenario

Every scenario is a named entry in `AllScenarios()` (see `internal/loadtest/scenarios.go`). Run one from the loadtest CLI under `internal/loadtest/cmd`:

```bash
cd apps/strait
go run -tags loadtest ./internal/loadtest/cmd --scenario=BackpressureCeiling
```

The process writes a Markdown summary to `apps/strait/internal/loadtest/reports/<scenario>-<ISO8601>.md`. That directory is gitignored.

## New scenarios (Phase 4)

| Scenario | What it exercises |
|---|---|
| `BackpressureCeiling` | Ramps enqueue across N projects past the per-project token bucket; measures rejection rate and steady-state dequeue throughput. |
| `CircuitBreakerChaos` | Uses `internal/testutil/pgxslow` to inject DB latency; asserts the circuit opens within threshold, dequeue short-circuits, and recovery completes after backoff. |
| `OutboxBurst` | Bulk enqueues via the outbox with the flusher paused, then releases the flusher; measures throughput and the `outbox_lag_seconds` histogram. |
| `DenormalizedDequeueDelta` | Runs `ThroughputCeiling` with `QUEUE_USE_DENORMALIZED_DEQUEUE` true and false; the flag selects the fully denormalized dequeue path. |
| `PartitionCycleMatrix` | Varies `partitionCycle` (1, 4, 12) at fixed enqueue rate; reports per-partition P99 dequeue latency via `partition_dequeue_lag_seconds`. |

## Metrics to watch

During a scenario, the Prometheus scrape endpoint on the server exposes the canonical signals. The ones worth watching first:

- `strait_dequeue_duration` — P50 / P95 / P99 of the dequeue path.
- `strait_queue_depth` — backlog per project and per partition.
- `strait_queue_partition_dequeue_lag_seconds` — per-partition P99 dequeue latency (Phase 1).
- `strait_queue_outbox_lag_seconds` — time between enqueue-via-outbox and flusher pickup (Phase 1).
- `strait_queue_backpressure_tokens_available` — per-project token bucket headroom; drops toward zero under rejection pressure.
- `strait_pool_running_workers` / `strait_pool_waiting_tasks` — pool saturation.
- `strait_queue_circuit_state_transitions_total` — circuit breaker transitions on the worker -> DB path.
- `strait_queue_dlq_oldest_unmasked_age_seconds` — DLQ age-out falling behind.

Pyroscope captures CPU / alloc profiles for the same run when the server is configured with `PYROSCOPE_SERVER_ADDRESS`.

## Reference-machine baseline

These numbers are informational — captured on a 16-core workstation against a co-located Postgres 18 and Redis 8 in Docker. They should not be treated as acceptance gates; your hardware will move them noticeably.

| Scenario | Metric | Reference value |
|---|---|---|
| `ThroughputCeiling` | Sustained jobs/sec at P99 < 5s | ~1.8k |
| `BackpressureCeiling` | Rejection rate at 2x bucket | ~48% |
| `OutboxBurst` | Flusher drain rate | ~6k rows/sec |
| `DenormalizedDequeueDelta` | Dequeue P99 delta | 35-45% lower with fully denormalized dequeue on |
| `PartitionCycleMatrix` | Per-partition P99 spread | < 30% across cycles |

Capture your own numbers before a performance-sensitive change and again after. The Markdown report is designed to be diffed across runs.

## Extending the harness

- New scenarios live in `scenarios.go` next to the existing `ThroughputCeiling` / `ConcurrencyCeiling` entries. Follow the `RampConfig` or `TenantSimulatorConfig` pattern already in use.
- For DB chaos, use `internal/testutil/pgxslow` as a `pgx.QueryTracer`. It supports unconditional and regex-matched latency injection and honours context cancellation.
- Emit a Markdown summary on completion via `loadtest.WriteMarkdownReport` so runs are diffable.
