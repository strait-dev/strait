This runbook covers day-2 operations for the audit event subsystem. It assumes
familiarity with [Audit Logging](/concepts/audit-logging) and the Strait
[Monitoring & Alerts](/operations/monitoring-and-alerts) baselines.

## Audit DML guard probe

Migration `000187_audit_events_dml_restrictions` revokes `UPDATE` + `DELETE`
on `audit_events` from the `strait_app` role, leaving only `INSERT`, `SELECT`,
and `UPDATE(signature)` for chain signing. The migration is **silently a
no-op when the `strait_app` role is absent** — the common case on self-hosted
installs whose default connection role is `strait` (superuser).

The `audit_dml_guard` component of `/health/ready` reports:

- **ok** — `has_table_privilege(current_user, 'audit_events', 'UPDATE')`
  returns `false`. The REVOKE landed; a compromised application process
  cannot rewrite audit rows without an explicit role escalation.
- **degraded** — the current role has full `UPDATE`. The HMAC chain still
  detects tampering forensically, but the defense-in-depth role restriction
  is not enforced. Self-hosted operators should provision `strait_app` per
  `SELFHOST.md`. SOC 2 evidence gates that require role-level enforcement
  must treat this as a blocker.

A startup log line (`audit_events DML restriction enforced...` or
`audit_events UPDATE is not restricted...`) captures the same posture at
process boot, and the `strait_audit_dml_restriction_status` counter emits
one sample per boot labeled `status=enforced|degraded` for dashboard
panels and long-horizon alerting.

## Audit emit health probe

The `/health/ready` endpoint registers an audit probe (`health.NewAuditProbe`)
that reports component state based on two signals:

- **ok** — the async emit drainer is running and the DLQ is empty.
- **degraded** — the drainer is running but `audit_events_deadletter` has one
  or more rows. Emit is not stalled; rows landed in the DLQ because a chain
  insert failed (unique-violation on `chain_seq`, HMAC re-sign failure, or
  transient store error). The scheduler reclaimer retries them on the next
  tick.
- **down** — the drainer has stopped accepting events (buffer wedged or
  context cancelled). Emitting callers fall back to the synchronous path and
  tail latency rises.

Treat a sustained degraded state (> 5 minutes) as a paging signal. Inspect the
DLQ before assuming a chain problem.

## Inspecting the DLQ

List dead-lettered events with the admin endpoint. Requires a role granting
`audit:admin`.

```bash
curl -H "Authorization: Bearer $STRAIT_ADMIN_TOKEN" \
  "https://api.strait.dev/v1/audit/deadletter?project_id=$PROJECT&limit=50"
```

Response entries include `id`, `enqueued_at`, `reason`, `attempts`, and the
original payload. Use the `cursor` field for pagination.

## Replaying a DLQ event

After diagnosing the failure (typical causes: transient Postgres timeout,
duplicate `chain_seq` from a prior partial write), replay the event. The
handler re-signs under the current epoch, inserts into `audit_events`, removes
the DLQ row, and emits `audit.deadletter_replayed`.

```bash
curl -X POST \
  -H "Authorization: Bearer $STRAIT_ADMIN_TOKEN" \
  "https://api.strait.dev/v1/audit/deadletter/$ID/replay"
```

If replay fails with `4xx`, the row stays in the DLQ and no audit is emitted;
fix the root cause and retry. A `DELETE /v1/audit/deadletter/{id}` accepts
data loss explicitly and emits `audit.deadletter_dropped` with the operator
actor recorded.

## Chain verification (CLI and API)

Verification re-derives HMACs per `rotation_epoch` group, threads the anchor
rows, and reports breaks.

CLI (recommended for out-of-band SOC 2 evidence):

```bash
strait audit verify --project $PROJECT_ID --output json
```

Exit code is `0` when the chain is intact, non-zero on any break. The CLI
emits `audit.chain_verified` with `actor_type=cli` so the run is itself
audited.

API (for dashboards or automated probes):

```bash
curl -H "Authorization: Bearer $STRAIT_ADMIN_TOKEN" \
  "https://api.strait.dev/v1/audit/verify?project_id=$PROJECT"
```

## Rotating the audit signing key

The HMAC signing secret is bound to a `rotation_epoch` column on every event.
Rotation inserts a forensic **anchor row** (`action=audit.key_rotated`,
`is_anchor=true`) carrying the previous chain tip hash, the previous epoch,
the new epoch, and the rotating actor. `VerifyAuditChain` splits the chain on
these anchors, verifies each epoch independently under the matching secret,
and stitches epochs via the anchor's `previous_hash`.

Rotate when:

- An operator with access to the old secret leaves the team.
- The secret may have been exposed (incident response).
- Scheduled rotation per your compliance policy (we suggest annually).

Procedure:

1. Generate a new 32-byte secret and stage it in your secret manager.
2. Call `RotateAuditSigningKey` via the internal admin surface. The call is
   serialized under a Postgres advisory lock; concurrent rotations are safe.
3. Verify the chain — the anchor row for the new epoch must be present and
   `strait audit verify` must exit 0.
4. Decommission the old secret from your secret manager only after at least
   one full retention window has elapsed; `VerifyAuditChain` needs all prior
   epoch secrets to verify historical rows.

## Retention overrides

Global default is `AUDIT_RETENTION_DEFAULT_DAYS`. Per-project overrides live
on `project_quotas.audit_retention_days`. A value of `0` disables trim for
that project (useful for regulated tenants with external retention).

```bash
# Inspect effective retention
curl -H "Authorization: Bearer $STRAIT_ADMIN_TOKEN" \
  "https://api.strait.dev/v1/projects/$PROJECT/audit/retention"

# Set a 2555-day retention (7 years)
curl -X PUT \
  -H "Authorization: Bearer $STRAIT_ADMIN_TOKEN" \
  -H "content-type: application/json" \
  -d '{"days": 2555}' \
  "https://api.strait.dev/v1/projects/$PROJECT/audit/retention"
```

Every retention trim that actually deletes rows writes a tombstone anchor
(`action=audit.retention_trimmed`) inside the same transaction so the chain
stays verifiable after trim.

## Deadletter retention

The audit deadletter table (`audit_events_deadletter`) holds events that
could not reach the primary chain after exhausting the in-process retry
budget. Three controls keep it bounded:

| Variable | Default | Purpose |
|---|---|---|
| `AUDIT_DLQ_RECLAIM_BATCH` | 200 | Max DLQ rows the scheduler reclaimer processes per tick. |
| `AUDIT_DLQ_MAX_AGE_DAYS` | 30 | Drop DLQ rows older than N days. `0` disables the sweep. |
| `AUDIT_DLQ_MAX_RECLAIM_ATTEMPTS` | 10 | Skip a row after N failed chain inserts. `0` disables the cap. |

Drops emit `audit.deadletter_aged` events into the affected project's chain
(`{dropped_count, reason: "max_age_exceeded"}`) and bump
`strait_audit_deadletter_aged_total{project_id}`. Skipped rows count toward
`strait_audit_reclaimer_abandoned_total`; they stay in the DLQ for operator
triage via `GET /v1/audit/deadletter`.

Replay is at-least-once with idempotency: a successful chain insert sets
`reclaimed_event_id` on the DLQ row. A subsequent retry observes the marker
and only deletes the DLQ entry — it never inserts a duplicate chain event.

## SIEM drain troubleshooting

SIEM forwarding is opt-in via `AUDIT_SIEM_ENDPOINT`. Relevant env vars:

| Variable | Default | Purpose |
|---|---|---|
| `AUDIT_SIEM_ENDPOINT` | (empty) | HTTPS endpoint accepting NDJSON batches. Empty disables forwarding. |
| `AUDIT_SIEM_AUTH_TOKEN` | (empty) | Bearer token. Required when endpoint is set. |
| `AUDIT_SIEM_BATCH_SIZE` | 100 | Max events per batch before flush. |
| `AUDIT_SIEM_FLUSH_INTERVAL` | 10s | Max time between flushes. |

Resilience behavior:

- `5xx` responses retry with full-jitter backoff (100ms, 400ms, 1.6s).
- Five consecutive failures open the circuit breaker for 30 seconds. While
  open, batches short-circuit and increment `strait_audit_siem_circuit_open_total`.
  The breaker never blocks primary chain emit — forwarding is best-effort.
- `4xx` responses (excluding `429`) drop the batch to a SIEM sub-DLQ with
  reason `siem_4xx`. The sub-DLQ is distinct from the primary
  `audit_events_deadletter` table; operators drain it via the logdrain admin
  surface.

Common failure modes:

- **Breaker open > 5m** (`AuditSIEMForwardFailing` alert) — inspect endpoint
  health; rotate `AUDIT_SIEM_AUTH_TOKEN` if the provider rotated it.
- **Sub-DLQ growth with `siem_4xx`** — payload schema drift; validate against
  `audit_schema_generated.json` and fix the SIEM receiver mapping.
- **No events arriving at SIEM but chain healthy** — confirm the endpoint is
  set at both the API and worker processes (the drain runs in every mode that
  emits).
