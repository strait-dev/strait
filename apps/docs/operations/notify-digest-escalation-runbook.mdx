This runbook covers day-one Strait Notify operations for:

- digest batching (`instant`, `hourly`, `daily`)
- scheduled notification dispatch
- approval reminder escalation tiers

Use this during incidents where notifications are delayed, missing, duplicated, or escalation reminders appear stuck.

## Scope and Prerequisites

Applies to:

- API mode (`--mode api`)
- worker mode (`--mode worker`)
- combined mode (`--mode all`)

Before triage, verify:

1. API and worker health checks (`/health`, `/health/ready`)
2. database and redis dependencies are healthy
3. scheduler loops are running in at least one process

## Fast Triage Checklist

1. Confirm trigger intake:
   - check `POST /v1/notify` caller logs and response IDs
2. Confirm message lifecycle:
   - inspect `GET /v1/notify/deliveries`
   - look for `scheduled`, `processing`, `failed`, `delivered`
3. Confirm digest path:
   - if recipient has `digest_policy=hourly|daily`, non-critical unscheduled sends are batched
4. Confirm escalation path:
   - check `GET /v1/notify/escalations/step-runs/{stepRunID}`
   - verify state progression (`active` → `processing` → terminal)

## Triage Script (CLI)

Use the script-only triage helper for a quick database snapshot:

```bash
cd apps/strait
DATABASE_URL='postgres://...' go run ./scripts/notify-triage --project-id <project_id>
```

Useful flags:

- `--window=6h` to widen/narrow recent failure inspection (also drives SES feedback event-type and suppression-reason aggregation)
- `--stale-threshold=30m` to tune stuck escalation detection
- `--limit=20` to control failed message row output size

The script also prints SES-focused sections when data exists:

- `ses feedback event types` from `notify_provider_callback_receipts`
- `ses suppression reasons` from `notify_suppression_events`

For explicit queue + callback verification, use the SES check helper:

```bash
cd apps/strait
NOTIFY_SES_FEEDBACK_SQS_URL='https://sqs.us-east-1.amazonaws.com/123456789012/notify-feedback' \
DATABASE_URL='postgres://...' \
go run ./scripts/notify-ses-feedback-check --project-id <project_id> --window 6h
```

## Common Failure Modes

### 1) Messages remain in `scheduled`

Likely causes:

- worker process not running notify dispatcher
- system clock skew between app nodes
- backlog caused by repeated delivery errors

Checks:

- verify worker/scheduler process is healthy
- validate host clocks (NTP)
- inspect recent failures in `GET /v1/notify/deliveries?status=failed`

### 2) Digest appears delayed or never sent

Likely causes:

- event is non-critical and correctly deferred to window end
- digest batch repeatedly requeued due to downstream send errors
- provider config issue (email)

Checks:

- confirm expected policy (`instant` vs `hourly` vs `daily`)
- verify recipient/channel payload is valid
- verify provider credentials/config for email path

### 3) Escalation state stays active after approval resolution

Likely causes:

- approval action failed before callback completion
- timeout/cleanup path failed under transient store error

Checks:

- verify approval status and step status on workflow run
- call escalation lifecycle endpoints:
  - `POST /v1/notify/escalations/step-runs/{stepRunID}/acknowledge`
  - `POST /v1/notify/escalations/step-runs/{stepRunID}/complete`

### 4) SES feedback events are not reflected in delivery status

Likely causes:

- `NOTIFY_SES_FEEDBACK_SQS_URL` is unset or incorrect
- SES event destination does not forward `Delivery` / `Bounce` / `Complaint` to SNS/SQS
- feedback event payload is missing `strait_message_id` tag or references a deleted message

Checks:

- confirm notify email provider is `ses` (`NOTIFY_EMAIL_PROVIDER=ses`)
- confirm `NOTIFY_SES_FEEDBACK_SQS_URL` points to the expected queue
- confirm SES event destination emits events to SNS and SNS forwards to the SQS queue
- confirm outbound messages include provider tags (`strait_message_id`, `strait_project_id`)
- inspect callback ingestion metric distribution:
  - `sum(rate(strait_notify_provider_callbacks_total{provider="ses"}[5m])) by (event_type, outcome)`
  - `sum(rate(strait_notify_provider_callbacks_total{provider="ses",suppression_reason!="none"}[5m])) by (suppression_reason)`
- inspect notify delivery rows for transitions to `delivered`, `bounced`, or `failed`
- inspect subscriber preference changes when bounces/complaints occur (`channel_prefs.email=false`)

## SES Callback Drilldown Query Pack

Use these during incidents to isolate whether the issue is parser-path, state-transition, or recipient-quality related.

### PromQL

```promql
# Outcome split (primary signal)
sum(rate(strait_notify_provider_callbacks_total{provider="ses"}[5m])) by (outcome)
```

```promql
# Event type split (delivery vs bounce vs complaint)
sum(rate(strait_notify_provider_callbacks_total{provider="ses"}[5m])) by (event_type)
```

```promql
# Processing failures only
sum(rate(strait_notify_provider_callbacks_total{provider="ses",outcome=~".*_error|malformed"}[5m])) by (outcome)
```

```promql
# Duplicate callback ratio
100 * sum(rate(strait_notify_provider_callbacks_total{provider="ses",outcome="duplicate"}[5m]))
  / clamp_min(sum(rate(strait_notify_provider_callbacks_total{provider="ses"}[5m])), 0.0001)
```

```promql
# Bounce/complaint ratio among status-changing callbacks
100 * sum(rate(strait_notify_provider_callbacks_total{provider="ses",suppression_reason=~"provider_callback:ses.bounce|provider_callback:ses.complaint",outcome=~"status_updated|status_skipped"}[5m]))
  / clamp_min(sum(rate(strait_notify_provider_callbacks_total{provider="ses",outcome=~"status_updated|status_skipped"}[5m])), 0.0001)
```

### SQL (PostgreSQL)

```sql
-- callback receipts by event_type in last 6h
SELECT COALESCE(NULLIF(event_type, ''), 'unknown') AS event_type, COUNT(*)
FROM notify_provider_callback_receipts
WHERE provider = 'ses'
  AND created_at >= NOW() - interval '6 hours'
GROUP BY event_type
ORDER BY COUNT(*) DESC;
```

```sql
-- SES suppression reasons in last 6h
SELECT COALESCE(NULLIF(reason, ''), 'unknown') AS reason, COUNT(*)
FROM notify_suppression_events
WHERE source = 'provider_callback'
  AND channel = 'email'
  AND action = 'suppressed'
  AND reason LIKE 'provider_callback:ses.%'
  AND created_at >= NOW() - interval '6 hours'
GROUP BY reason
ORDER BY COUNT(*) DESC;
```

```sql
-- email message status distribution in last 6h
SELECT status, COUNT(*)
FROM notification_messages
WHERE channel = 'email'
  AND created_at >= NOW() - interval '6 hours'
GROUP BY status
ORDER BY COUNT(*) DESC;
```

## SES Feedback DLQ Triage

Use this when callback outcomes show sustained `*_error`/`malformed`, SQS backlog keeps growing, or callbacks stop updating delivery states.

1. Verify primary queue depth and callback ratios with the script gates:

```bash
cd apps/strait
go run ./scripts/notify-ses-feedback-check \
  --queue-url "$NOTIFY_SES_FEEDBACK_SQS_URL" \
  --prometheus-url "$PROMETHEUS_URL" \
  --max-callback-error-rate 5 \
  --max-callback-duplicate-ratio 20
```

2. Inspect DLQ attributes and latest payload samples:

```bash
aws sqs get-queue-attributes \
  --queue-url "$NOTIFY_SES_FEEDBACK_DLQ_URL" \
  --attribute-names ApproximateNumberOfMessages ApproximateNumberOfMessagesNotVisible QueueArn

aws sqs receive-message \
  --queue-url "$NOTIFY_SES_FEEDBACK_DLQ_URL" \
  --max-number-of-messages 5 \
  --visibility-timeout 30
```

3. Classify DLQ samples with the query pack above:
- malformed envelope / SNS wrapping mismatch
- missing `strait_message_id` / `strait_project_id` tags
- project mismatch or message-not-found due to stale ids

4. Redrive only after root cause is fixed; monitor for regression:
- callback `outcome` error rates return to baseline
- no new sustained DLQ growth over two poll windows

## Safe Remediation Steps

Prefer API remediation first:

1. **Acknowledge escalation** when human acted but state did not finalize.
2. **Complete escalation** when the approval path is already terminal.
3. **Re-trigger notifications** only after resolving root cause (provider, payload, permissions).

Avoid bulk/manual SQL updates during active incident response unless API-based recovery is not possible.

## Verification After Recovery

- New notifications transition to `delivered` for inbox/email.
- No persistent growth in `scheduled` or `processing` message counts.
- Escalation states for resolved approvals return 404 on active lookup.
- No repeated error spikes in worker logs for notify dispatcher.

## Escalation and Ownership

Escalate to on-call if any of the following persist beyond 15 minutes:

- scheduled queue age keeps increasing
- digest batches repeatedly requeue without progress
- escalation states cannot be acknowledged/completed via API
- approval reminders are not emitted for pending approvals

Include in handoff:

- affected project IDs
- representative message IDs and step run IDs
- latest `notify` endpoint responses
- relevant API/worker log excerpts
