This page covers the alerts Strait fires and what to do when you see them. Each alert includes the condition, severity, and resolution steps.

## Queue alerts

### Queue depth high

**Trigger.** Queue depth exceeds 10,000 for 5 minutes.

**Severity.** Warning.

**What to check.**

- Which queue is backing up? Look at queue depth broken down by status.
- Is the problem an enqueue spike or a dequeue stall? Compare inbound rate against worker throughput.

**How to fix.**

1. Scale worker replicas up.
2. If a single job type is responsible, raise its `concurrency` limit or check for backpressure issues.
3. Check database health: long-running queries, connection saturation, autovacuum pressure.

**Escalation.** If depth keeps climbing after scaling, page queue on-call. Sustained saturation risks downstream SLA breach.

---

### Dequeue latency high

**Trigger.** p99 dequeue latency exceeds 60 seconds for 10 minutes.

**Severity.** Warning.

**What to check.**

- Is the database under pressure? Check connection count, CPU, and disk I/O.
- Are workers healthy and actively claiming jobs?
- Is a single partition or job type responsible?

**How to fix.**

1. Investigate and resolve database bottlenecks (long queries, lock contention).
2. Scale workers if claim throughput is the constraint.
3. If a specific partition is hot, rebalance or increase its concurrency.

**Escalation.** Escalate if latency breaches your SLO after initial remediation.

## Run alerts

### Review queue growing

**Trigger.** Review queue depth has a sustained positive slope for 15 minutes.

**Severity.** Warning.

**What to check.**

- Which job types are sending runs to the review queue? Filter by `job_id` via `GET /v1/admin/review-queue`.
- Is the failure caused by a code bug, a dependency outage, or a misconfiguration?
- Cross-reference with webhook delivery failures or external service errors.

**How to fix.**

1. Identify the failure class from the run's `error` and `error_class` fields.
2. Fix the underlying cause (deploy a patch, restore a dependency).
3. Replay recoverable runs via `POST /v1/admin/review-queue/{run_id}/replay`.
4. Purge non-recoverable runs after documenting the reason.

See [Review queue recovery](/operations/dlq-recovery) for the full workflow.

**Escalation.** Page the domain owner if the cause is application logic. Otherwise treat as a platform incident.

---

### Stale runs detected

**Trigger.** A review queue entry has been waiting for triage for more than 24 hours.

**Severity.** Low (non-paging).

**What to check.**

- Pull the oldest entries: `GET /v1/admin/review-queue?order=oldest&limit=20`.
- Is triage blocked on a team or individual?

**How to fix.**

1. Investigate each entry and replay, purge, or explicitly acknowledge it.
2. Follow the [Review queue recovery](/operations/dlq-recovery) guide.

**Escalation.** If entries age past 72 hours, escalate to platform lead.

---

### High failure rate

**Trigger.** Run failure rate exceeds threshold for a sustained period.

**Severity.** Warning.

**What to check.**

- Which job types are failing? Check error patterns across recent runs.
- Is a dependency down or returning errors?
- Were there recent deploys that could have introduced a regression?

**How to fix.**

1. Identify the root cause from run error details.
2. Roll back if a recent deploy is responsible.
3. If an external dependency is down, wait for recovery and let retries handle it.
4. Adjust `max_attempts` or retry strategy if the failure is transient but exhausting attempts too quickly.

**Escalation.** Page application on-call if failure rate impacts end users.

## Worker alerts

### Pool exhausted

**Trigger.** No available worker capacity (backpressure tokens at zero) for 5 minutes.

**Severity.** Page.

**What to check.**

- Are workers running and healthy? Check for crash loops or restarts.
- Is a single long-running job type consuming all capacity?
- Has traffic spiked beyond current worker scale?

**How to fix.**

1. Scale worker replicas.
2. If a specific job type is monopolizing the pool, lower its concurrency or investigate why runs are taking longer than expected.
3. Check for stuck runs that are not completing or heartbeating.

**Escalation.** Immediate page to platform on-call if workers cannot recover capacity.

---

### Dispatch timeout

**Trigger.** Worker event channel is dropping messages for more than 5 minutes.

**Severity.** Warning.

**What to check.**

- Is the publisher outpacing consumers, or is a consumer stalled?
- Check Redis health and SSE client count.
- Is the event channel buffer sized correctly for current load?

**How to fix.**

1. Increase the event channel buffer size if fan-out pressure is confirmed.
2. Investigate and fix slow consumers (Redis connectivity, stalled SSE connections).
3. Scale workers horizontally to reduce contention.

**Escalation.** Escalate to queue on-call if drops persist past 30 minutes.

---

## Infrastructure alerts

### Database connection errors

**Trigger.** Database circuit breaker transitions to open state.

**Severity.** Page.

**What to check.**

- Is the database reachable? Check connection count, replication lag, CPU, and disk.
- Are other services experiencing database issues?
- Was there a recent infrastructure change (failover, maintenance window)?

**How to fix.**

1. Stabilize the database. The circuit breaker will half-open and close automatically as the error rate drops.
2. If the breaker is flapping, raise its error threshold temporarily via configuration. Do not disable it.
3. For a confirmed database outage, initiate your database failover procedure.

**Escalation.** Immediate page to database on-call. Treat as SEV-2 by default, SEV-1 if sustained longer than 15 minutes.

---

### Redis connection lost

**Trigger.** Redis pub/sub connection drops or reconnect rate spikes.

**Severity.** Warning.

**What to check.**

- Is Redis reachable from the application network?
- Check Redis memory usage, connection count, and CPU.
- Are SSE consumers receiving events? Redis pub/sub powers real-time delivery.

**How to fix.**

1. Restore Redis connectivity (restart, failover, or fix network issue).
2. Verify that SSE and pub/sub resume after reconnection.
3. Workers fall back to periodic polling when notifications are unavailable, so jobs still execute — but with higher latency.

**Escalation.** Escalate if Redis stays unreachable for more than 10 minutes or if real-time delivery is an SLA requirement.
