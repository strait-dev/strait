---
title: "Review queue recovery"
description: "How to inspect failed jobs in the review queue and replay them."
---

When a job exhausts all retry attempts, it moves to the review queue. This guide shows how to inspect failed jobs and replay them.

## Inspecting the review queue

List entries in the review queue for a specific job:

```bash
curl -H "Authorization: Bearer $STRAIT_API_KEY" \
  "https://api.strait.dev/v1/admin/review-queue?job_id=send-welcome-email&limit=50"
```

You can filter by `job_id`, `masked` (`true` or `false`), and paginate with `limit` and `cursor`. Omit `masked` to see all entries.

For each entry, inspect the run's `error` and `error_class` fields to understand the failure before taking action.

## Replaying a job

Replay re-queues a failed run and resets its attempt counter:

```bash
curl -X POST \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  https://api.strait.dev/v1/admin/review-queue/01JC4R9D5VJ7KN5WM7N5K2JCE9/replay
```

Response:

```json
{
  "id": "01JC4R9D5VJ7KN5WM7N5K2JCE9",
  "status": "queued",
  "attempt": 1
}
```

The run is moved back to the queue and will be picked up by workers. Always fix the underlying failure before replaying — replaying a poison pill creates a tight failure loop.

## Bulk replay

To replay multiple runs, call the replay endpoint for each affected run. Filter the review queue by `job_id` to find all entries for a specific job type, then replay each one:

```bash
# List affected runs
curl -H "Authorization: Bearer $STRAIT_API_KEY" \
  "https://api.strait.dev/v1/admin/review-queue?job_id=send-welcome-email&limit=100"

# Replay each run by ID
curl -X POST -H "Authorization: Bearer $STRAIT_API_KEY" \
  https://api.strait.dev/v1/admin/review-queue/{run_id}/replay
```

## Purging entries

To permanently remove a run from the review queue:

```bash
curl -X POST \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  https://api.strait.dev/v1/admin/review-queue/01JC4R9D5VJ7KN5WM7N5K2JCE9/purge
```

Purge is irreversible. Use it only for runs that are not recoverable and do not need to be retained. Document the reason before purging — every mutation is recorded in the audit trail via `/v1/audit-events`.

## Prevention

To reduce how often jobs land in the review queue:

- **Adjust `max_attempts`.** If transient failures exhaust retries too quickly, increase the limit.
- **Review your retry strategy.** Use exponential backoff with jitter for external dependencies. See [Retry strategies](/concepts/retry-strategies).
- **Monitor endpoint health.** If a downstream service is consistently failing, fix or circuit-break it before runs pile up.
