---
title: "Review queue recovery runbook"
description: "Audited, RBAC-gated HTTP endpoints for recovering failed runs from the review queue without manual SQL."
---

The Strait queue moves a run to the review queue once it has exhausted all
retry attempts, failed dependency checks that cannot be satisfied, or
tripped a protective circuit. The admin review-queue endpoints under
`/v1/admin/review-queue` give on-call operators a controlled, audited surface for
recovery -- replacing the manual SQL that previous runbooks required.

## Endpoints

All endpoints live under `/v1/admin/review-queue` and require a bearer token or
internal secret. The caller's effective scopes must include the scope
listed below; internal-secret callers are fully trusted and bypass the
scope check.

| Method | Path | Scope | Purpose |
| --- | --- | --- | --- |
| `GET` | `/v1/admin/review-queue` | `review-queue:read` | Paginated listing of runs in the review queue for the caller's project. Supports `job_id`, `masked` (`true`/`false`), `limit`, and `cursor` query parameters. The `masked` filter selects rows whose `visible_until` has been set (`true`) or is still null (`false`); omit for no filter. Filtering is pushed into SQL so results remain correct across pages. |
| `POST` | `/v1/admin/review-queue/{run_id}/replay` | `review-queue:replay` | Re-enqueues a run from the review queue and records a `replayed_run_id` back-pointer on the original row. |
| `POST` | `/v1/admin/review-queue/{run_id}/unmask` | `review-queue:replay` | Clears `visible_until` on a review-queue row so it is no longer hidden by the age-out masker. |
| `POST` | `/v1/admin/review-queue/{run_id}/purge` | `review-queue:purge` | Hard-deletes a run from the review queue. Irreversible -- log context via the audit trail before calling. |

## RBAC

Three new scopes are added to the platform:

- `review-queue:read` -- granted by default to the `viewer` and `operator` system roles.
- `review-queue:replay` -- granted by default to the `operator` system role.
- `review-queue:purge` -- NOT granted by default. Assign it explicitly on a role
  that is tightly audited; the scope escalation check in
  `handleCreateAPIKey` enforces that only callers holding `review-queue:purge`
  can issue keys that carry it.

The `admin` system role has the wildcard scope and therefore inherits
all three permissions implicitly.

## Auditing

Every mutation writes a row into `audit_events`:

- `action` = `review-queue.replay`, `review-queue.unmask`, or `review-queue.purge`
- `resource_type` = `job_run`
- `resource_id` = the target run id
- `details.before` / `details.after` capture the pre- and post-mutation
  state (status, masked flag, replayed pointer, etc.)

Use the standard `/v1/audit-events` endpoint to retrieve the trail. The
HMAC chain signing applies to review-queue actions just like any other audited
control-plane operation, so post-hoc tampering is detectable.

## Example: replay a run from the review queue

```bash
curl -X POST \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  https://api.strait.dev/v1/admin/review-queue/01JC4R9D5VJ7KN5WM7N5K2JCE9/replay
```

Response (abbreviated):

```json
{
  "id": "01JC4R9D5VJ7KN5WM7N5K2JCE9",
  "status": "queued",
  "attempt": 1
}
```

After the call, the row referenced by `id` is back in the queue. The
replay helper reuses the existing row via a compare-and-swap back to
`queued`, so `replayed_run_id` is populated with that same id -- it is a
marker that the row was recovered through the admin replay path, not a
pointer to a distinct new run. The row is no longer counted towards the
review-queue depth gauge on the next scrape.

## Example: list runs in the review queue for a specific job

```bash
curl -H "Authorization: Bearer $STRAIT_API_KEY" \
  "https://api.strait.dev/v1/admin/review-queue?job_id=send-welcome-email&limit=50"
```

## Diagnostic workflow

1. Spot the spike on the `StraitReviewQueueGrowing` alert or the review-queue panel in
   the Grafana "Queue Internals" row.
2. Call `GET /v1/admin/review-queue` scoped to the affected project and (if
   known) job to enumerate candidates.
3. For each candidate, inspect the run's `error`, `error_class`, and
   last event via the standard run endpoints -- understand the cause
   before replaying. Replaying a poison pill without addressing the
   root cause produces a tight failure loop.
4. Fix the upstream bug, deploy the fix, then `POST /replay` for the
   affected runs. Use `POST /unmask` first if the row has been age-out
   masked and you want it to appear in dashboards again.
5. Reserve `POST /purge` for rows that must not be retained for
   compliance reasons; the action is irreversible.
