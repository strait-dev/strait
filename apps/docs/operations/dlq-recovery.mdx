---
title: "Dead-letter recovery"
description: "How to inspect dead-lettered runs and replay them."
---

When a run exhausts all retry attempts, or poison pill detection stops retries early, it moves to `dead_letter`. This guide shows how to inspect dead-lettered runs and replay them.

## Inspecting the dead-letter queue

List dead-lettered runs for the authenticated project:

```bash
curl -H "Authorization: Bearer $STRAIT_API_KEY" \
  "https://api.strait.dev/v1/admin/dlq?job_id=send-welcome-email&limit=50"
```

You can filter by `job_id`, `masked` (`true` or `false`), and paginate with `limit` and `cursor`. Omit `masked` to see all entries.

For each entry, inspect the run's `error` and `error_class` fields to understand the failure before taking action.

## Replaying a run

Replay re-queues a dead-lettered run:

```bash
curl -X POST \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  https://api.strait.dev/v1/admin/dlq/01JC4R9D5VJ7KN5WM7N5K2JCE9/replay
```

Response:

```json
{
  "id": "01JC4R9D5VJ7KN5WM7N5K2JCE9",
  "status": "queued",
  "attempt": 1
}
```

The run is moved back to the queue or superseded by a replay run, depending on the replay path. Always fix the underlying failure before replaying. Replaying a poison pill creates a tight failure loop.

## Bulk replay

To replay multiple runs, use the bulk replay endpoint with explicit run IDs, or filter the DLQ by `job_id` and replay each affected run:

```bash
# List affected runs
curl -H "Authorization: Bearer $STRAIT_API_KEY" \
  "https://api.strait.dev/v1/admin/dlq?job_id=send-welcome-email&limit=100"

# Bulk replay by run ID
curl -X POST \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"run_ids":["run_1","run_2"]}' \
  https://api.strait.dev/v1/runs/bulk-dlq-replay
```

## Purging entries

To permanently remove a run from the dead-letter queue:

```bash
curl -X POST \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  https://api.strait.dev/v1/admin/dlq/01JC4R9D5VJ7KN5WM7N5K2JCE9/purge
```

Purge is irreversible. Use it only for runs that are not recoverable and do not need to be retained. Document the reason before purging — every mutation is recorded in the audit trail via `/v1/audit-events`.

## Prevention

To reduce how often runs land in `dead_letter`:

- **Adjust `max_attempts`.** If transient failures exhaust retries too quickly, increase the limit.
- **Review your retry strategy.** Use exponential backoff with jitter for external dependencies. See [Retry strategies](/concepts/retry-strategies).
- **Monitor endpoint health.** If a downstream service is consistently failing, fix or circuit-break it before runs pile up.
