---

This playbook covers the full lifecycle of Stripe product management for the
orchestration-only pricing model:

1. Create sandbox products and prices (safe to repeat; idempotent).
2. Verify sandbox integration end-to-end.
3. Cut over to live mode (requires human approval before any live API call).
4. Rollback if something goes wrong.

## Sandbox product creation

Run these steps in the Stripe **test** (sandbox) dashboard or via the
`stripe-sandbox` MCP server. Never use live-mode keys for sandbox setup.

### Prerequisites

- Stripe sandbox API key (`sk_test_...`) set in your local environment or CI secret.
- `stripe` CLI authenticated: `stripe login` (uses test mode by default).

### Step 1 — Archive obsolete container-execution products

If any legacy container-execution or compute-credit products exist in sandbox,
archive them to keep the catalog clean:

```bash
# List products to identify legacy IDs
stripe products list --test-mode

# Archive each obsolete product (replace <PROD_ID> with the actual ID)
stripe products update <PROD_ID> --active=false --test-mode
```

### Step 2 — Create base plan products and prices

```bash
# Strait Starter — $19.99/mo
STARTER_PROD=$(stripe products create \
  --name="Strait Starter" \
  --description="Starter plan: 10K runs/mo, 10 concurrent, 3 workers" \
  --test-mode \
  --json | jq -r '.id')

stripe prices create \
  --product="$STARTER_PROD" \
  --unit-amount=1999 \
  --currency=usd \
  --recurring[interval]=month \
  --test-mode

# Strait Pro — $49.99/mo
PRO_PROD=$(stripe products create \
  --name="Strait Pro" \
  --description="Pro plan: 100K runs/mo, 50 concurrent, 20 workers" \
  --test-mode \
  --json | jq -r '.id')

stripe prices create \
  --product="$PRO_PROD" \
  --unit-amount=4999 \
  --currency=usd \
  --recurring[interval]=month \
  --test-mode

# Strait Scale — $149.99/mo
SCALE_PROD=$(stripe products create \
  --name="Strait Scale" \
  --description="Scale plan: 500K runs/mo, 200 concurrent, 100 workers" \
  --test-mode \
```

### Step 3 — Create overage prices (graduated tiers)

Overage is metered per 1,000 runs. Create one price object per plan so that
the tier breakpoints can differ in future. For now all three use the same
graduated structure.

```bash
# Graduated overage price for Starter (example; repeat for Pro and Scale)
stripe prices create \
  --product="$STARTER_PROD" \
  --currency=usd \
  --billing-scheme=tiered \
  --tiers-mode=graduated \
  --recurring[interval]=month \
  --recurring[usage-type]=metered \
  --tiers[0][up-to]=10000 \
  --tiers[0][unit-amount]=30 \
  --tiers[1][up-to]=100000 \
  --tiers[1][unit-amount]=25 \
  --tiers[2][up-to]=inf \
  --tiers[2][unit-amount]=20 \
  --test-mode
```

### Step 5 — Capture price IDs and populate environment

After each `stripe prices create` call above, copy the returned `id` field
(format: `price_...`) and set it in your `.env` or CI secret store:

```bash
# .env (sandbox / staging)
STRIPE_SECRET_KEY=sk_test_...
STRIPE_WEBHOOK_SECRET=whsec_...

STRIPE_STARTER_PRICE_ID=price_...
STRIPE_PRO_PRICE_ID=price_...
STRIPE_SCALE_PRICE_ID=price_...
STRIPE_ENTERPRISE_PRICE_ID=price_...

STRIPE_STARTER_OVERAGE_PRICE_ID=price_...
STRIPE_PRO_OVERAGE_PRICE_ID=price_...
STRIPE_SCALE_OVERAGE_PRICE_ID=price_...

STRIPE_RETENTION_PACK_PRICE_ID=price_...
STRIPE_PRIORITY_SLOT_PACK_PRICE_ID=price_...
STRIPE_LOG_DRAIN_VOLUME_PRICE_ID=price_...
STRIPE_WORKER_CONNECTION_PRICE_ID=price_...
```

### Step 6 — Configure sandbox webhook endpoint

```bash
stripe listen --forward-to localhost:7676/v1/webhooks/stripe --test-mode
```

For a deployed sandbox environment, register the endpoint in the Stripe
dashboard under **Developers > Webhooks**, pointing to
`https://<your-sandbox-domain>/v1/webhooks/stripe`. Enable at minimum:

- `customer.subscription.created`
- `customer.subscription.updated`
- `customer.subscription.deleted`
- `invoice.payment_succeeded`
- `invoice.payment_failed`
- `checkout.session.completed`

### Step 7 — Smoke test subscription flow

```bash
# Create a test customer and subscribe to Starter
stripe customers create --email="test+sandbox@example.com" --test-mode

stripe subscriptions create \
  --customer=<CUSTOMER_ID> \
  --items[0][price]=<STRIPE_STARTER_PRICE_ID> \
  --test-mode

# Simulate a successful payment
stripe trigger invoice.payment_succeeded --test-mode
```

Verify that the Strait billing handler receives the webhook and updates the
org's subscription state (`SELECT * FROM subscriptions WHERE ...`).

## Live cutover checklist

**Do not proceed without explicit human sign-off. No live Stripe API calls
should be made by automation or agents.**

- [ ] Sandbox smoke test passes end-to-end (subscription create, update, cancel, overage).
- [ ] Staging environment has been running on sandbox prices for at least 48 hours with no billing errors.
- [ ] QA has verified checkout, upgrade, downgrade, and cancellation flows.
- [ ] Backup of current `subscriptions` and `invoices` tables taken.
- [ ] Stripe dashboard live-mode access verified (separate API key, separate webhook secret).
- [ ] Deployment plan communicated to on-call.

### Repeat Steps 1–6 in live mode

Replace all `--test-mode` flags with no flag (live mode is the default) and
switch to live-mode API keys:

```bash
export STRIPE_SECRET_KEY=sk_live_...

# Archive any legacy live-mode products (same pattern as Step 1)
# Create products and prices (same commands as Steps 2–4, minus --test-mode)
# Update .env.production with the new live price IDs
# Register the live webhook endpoint in the dashboard
```

### Webhook endpoint update

Change the webhook endpoint URL in the Stripe live dashboard to:
`https://api.strait.dev/v1/webhooks/stripe` (or your custom domain).

Update `STRIPE_WEBHOOK_SECRET` in the live environment secrets to the new
endpoint's signing secret.

### Customer migration plan

There are **no existing production container-execution customers** at the time
of this migration. The live cutover is a clean-slate deployment. If that
changes before go-live, a migration plan must be drafted separately.

## Rollback plan

If the live cutover causes billing errors:

1. Revert `STRIPE_*_PRICE_ID` env vars to the previous values and redeploy.
   Existing Stripe subscriptions reference price IDs by ID; reverting the
   config does not cancel or alter subscriptions.
2. If incorrect subscriptions were created, cancel them in the Stripe
   dashboard under **Customers** (do not delete — cancellation preserves audit
   history). Refund any erroneous charges manually.
3. Re-enable the previous product set in the Stripe dashboard if it was
   archived.
4. File a post-mortem within 24 hours following the standard incident process.
