This walkthrough sets up Strait on your machine, creates a project, registers a job, triggers a run, and chains two jobs into a workflow. If you're using Strait Cloud, skip to [step 4](#step-4-create-a-project) — your base URL is `https://api.strait.dev` and your API key replaces `INTERNAL_SECRET` below.

## Prerequisites

You need Go 1.26+ (`go version`), Docker with Compose v2 (`docker compose version`), and `curl` if you want to run the API examples as-is. The CLI is installable later and works for everything `curl` does.

## Step 1: start the infrastructure

```bash
git clone https://github.com/strait-dev/strait.git
cd strait

docker compose -f apps/strait/docker-compose.yml up -d
docker compose -f apps/strait/docker-compose.yml ps
```

This runs PostgreSQL on host port 15432, Redis on 16379, and Sequin on 7376. All three should report `(healthy)`. If a port is already in use, edit the port mapping in `apps/strait/docker-compose.yml` or stop the conflicting service.

## Step 2: configure environment variables

```bash
export DATABASE_URL=postgres://strait:strait@localhost:15432/strait?sslmode=disable
export REDIS_URL=redis://localhost:16379

# Both must be at least 32 characters. Generate with `openssl rand -base64 32`.
export INTERNAL_SECRET=my-dev-secret-change-in-production-min-32-chars
export JWT_SIGNING_KEY=my-jwt-key-change-in-production-min-32-chars
```

`INTERNAL_SECRET` is the bearer token for the management API. `JWT_SIGNING_KEY` signs the run tokens that SDK code uses inside a running job. For local work you only interact with `INTERNAL_SECRET`.

## Step 3: build and run Strait

```bash
cd apps/strait && go build -o strait ./cmd/strait
./strait --mode all
```

Mode `all` runs the API, the worker, and the scheduler in one process. On startup Strait connects to PostgreSQL, applies any pending migrations, opens the HTTP server on `:8080`, and starts the worker pool. In production you'd split this into `--mode api` instances behind a load balancer and `--mode worker` instances on their own pool.

## Step 4: create a project

Projects scope jobs, workflows, API keys, and quotas.

```bash
curl -X POST http://localhost:8080/v1/projects \
  -H "Authorization: Bearer $INTERNAL_SECRET" \
  -H "Content-Type: application/json" \
  -d '{ "name": "My Project", "slug": "my-project" }'
```

Save the returned `id` (it looks like `proj_01JA...`). You'll pass it as `project_id` for the next two calls.

## Step 5: create a job

A job is a template: an endpoint URL, a timeout, and a retry budget.

<CodeGroup>
```bash curl
curl -X POST http://localhost:8080/v1/jobs \
  -H "Authorization: Bearer $INTERNAL_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_abc123",
    "name": "Hello World",
    "slug": "hello-world",
    "endpoint_url": "https://httpbin.org/post",
    "timeout_secs": 60,
    "max_attempts": 3
  }'
```

```bash CLI
./strait jobs create \
  --name "Hello World" \
  --slug "hello-world" \
  --endpoint "https://httpbin.org/post" \
  --timeout-secs 60 \
  --max-attempts 3
```
</CodeGroup>

## Step 6: trigger a run

```bash
curl -X POST http://localhost:8080/v1/jobs/hello-world/trigger \
  -H "Authorization: Bearer $INTERNAL_SECRET" \
  -H "Content-Type: application/json" \
  -d '{ "payload": { "greeting": "Hello from Strait!" } }'
```

The response gives you a run ID and `status: queued`. The worker claims it, POSTs the payload to `https://httpbin.org/post`, and writes the result back. A 2xx response transitions the run to `completed`; any other response schedules a retry using the job's retry strategy.

## Step 7: watch the run

```bash
curl http://localhost:8080/v1/runs/run_xyz789 \
  -H "Authorization: Bearer $INTERNAL_SECRET"

# Or stream live updates
./strait runs watch run_xyz789
```

Live updates also stream over SSE at `/v1/runs/{runID}/stream`. See [Real-Time Streaming](/concepts/real-time) for the full event format.

## Step 8: chain two jobs into a workflow

Workflows are DAGs of steps. Each step has a `step_ref`, a `job_id`, and an optional `depends_on` list.

```bash
curl -X POST http://localhost:8080/v1/workflows \
  -H "Authorization: Bearer $INTERNAL_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_abc123",
    "name": "Data Pipeline",
    "slug": "data-pipeline",
    "steps": [
      { "step_ref": "extract", "job_id": "job_abc123", "payload": {"action": "extract"} },
      { "step_ref": "transform", "job_id": "job_abc123", "depends_on": ["extract"], "payload": {"action": "transform"} }
    ]
  }'

curl -X POST http://localhost:8080/v1/workflows/data-pipeline/trigger \
  -H "Authorization: Bearer $INTERNAL_SECRET" \
  -H "Content-Type: application/json" \
  -d '{ "payload": { "source": "quickstart" } }'
```

`extract` runs first; `transform` waits for it. Steps with no dependency between them run in parallel.

## Step 9: trigger from your code

```bash
npm install @strait/ts
```

```typescript
import { createClient } from "@strait/ts";

const client = createClient({
  baseUrl: "http://localhost:8080",
  auth: { type: "bearer", token: process.env.INTERNAL_SECRET! },
});

const run = await client.triggerJob({
  pathParams: { jobID: "hello-world" },
  body: { payload: { greeting: "Hello from TypeScript!" } },
});

console.log("Run started:", run.id, run.status);
```

The other SDKs — Python, Go, Ruby, Rust — are documented under [SDKs](/sdks/overview).

## Common problems

**Run stuck in `queued`.** The worker isn't claiming it. Check that you started Strait with `--mode all` or `--mode worker`, and that `DATABASE_URL` and `REDIS_URL` are set in the same shell that runs the binary.

**Endpoint timing out.** If runs land in `timed_out`, raise `timeout_secs` on the job, or look at the latency of the endpoint itself. The default retry strategy will keep retrying until `max_attempts` is reached.

**SSRF rejection on a private endpoint.** Strait blocks private IP ranges, loopback, and link-local addresses by default. Use a public endpoint, or read [Security](/guides/security#ssrf-protection) for how to configure trusted internal targets.

**Repeated failures.** When a job exhausts `max_attempts` it moves to the review queue. Inspect the request and response in the dashboard, fix the underlying issue, and replay the run.
