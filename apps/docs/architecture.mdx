Strait is an orchestration service for background jobs and workflows. Your application keeps running the business logic; Strait handles triggering, scheduling, retries, workflow state, and visibility.

## The Model

```text
Your app or service
        |
        | trigger a job or workflow
        v
Strait API
        |
        | stores state and queues work
        v
Strait worker
        |
        | calls your HTTP endpoint or connected worker
        v
Your code
```

Strait does not execute customer code inside the Strait process. It dispatches work to code you own, then records the result so you can monitor, retry, replay, and audit it.

## What Runs

Strait ships as one Go binary with three modes:

| Mode | Use it for |
|---|---|
| `all` | Local development and smaller deployments. Runs the API, worker, and scheduler together. |
| `api` | Serving API traffic behind a load balancer. |
| `worker` | Processing queued jobs and workflow steps. |

You can start with `all` and split API and worker processes later as traffic grows.

## Required Services

| Service | Purpose |
|---|---|
| PostgreSQL | Stores projects, jobs, runs, workflows, and queue state. |
| Redis | Powers live updates and pub/sub notifications. |

Self-hosted Docker Compose includes the supporting services Strait needs. For setup instructions, see [Quick Start](/quickstart) and [Deployment](/guides/deployment).

## Execution Modes

Strait supports two ways to run customer code:

- **HTTP jobs** -- Strait sends a POST request to your configured endpoint.
- **Worker jobs** -- your worker keeps a connection open to Strait and receives work over that connection.

Use HTTP jobs when your code is already exposed as an endpoint. Use worker jobs when the code should stay inside your private network.

## Runs and Workflows

A job trigger creates a run. Most runs move from `queued` to `executing` to `completed`. Failed runs retry according to the job's retry policy. If retries are exhausted, the run moves to `dead_letter` so you can inspect and replay it.

Workflows connect multiple jobs into a directed graph. Steps can run in parallel, wait for approvals or external events, pass data between steps, and recover failed branches.

## Observability

Strait records run status, attempts, timing, logs, and workflow progress. You can watch runs in real time, subscribe to webhooks, export metrics, and inspect failures through the dashboard or API.
