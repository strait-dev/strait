---
title: "Event Triggers"
description: "Pause workflows until external events arrive, then continue automatically."
---

This guide walks through setting up event triggers to pause workflows until external events arrive. Use them for human approvals, third-party callbacks, cross-service coordination, or any step that depends on an external signal.

## Step 1: Define a Wait Step

Add a step with `type: "wait_for_event"` to your workflow. The `event_key` identifies which event this step waits for. Use template variables to make keys unique per run.

```json
{
  "name": "KYC Onboarding",
  "slug": "kyc-onboarding",
  "steps": [
    { "step_ref": "collect-data", "job_id": "job_collect_user_data" },
    {
      "step_ref": "aml-check",
      "type": "wait_for_event",
      "event_key": "aml:{{payload.user_id}}",
      "timeout_secs": 86400,
      "depends_on": ["collect-data"]
    },
    {
      "step_ref": "create-account",
      "job_id": "job_create_account",
      "depends_on": ["aml-check"]
    }
  ]
}
```

## Step 2: Trigger the Workflow

Start the workflow with a payload that fills in the template variables:

```bash
curl -X POST https://strait.dev/v1/workflows/kyc-onboarding/trigger \
  -H "Authorization: Bearer strait_..." \
  -H "Content-Type: application/json" \
  -d '{"payload": {"user_id": "user-456"}}'
```

The `collect-data` step runs. When it completes, the `aml-check` step starts waiting for event key `aml:user-456`.

## Step 3: Send the Event

When the external system completes its work, send the event to Strait:

```bash
curl -X POST https://strait.dev/v1/events/aml:user-456/send \
  -H "Authorization: Bearer strait_..." \
  -H "Content-Type: application/json" \
  -d '{"payload": {"result": "clear", "risk_score": 0.05}}'
```

```typescript
await client.sendEvent({
  pathParams: { eventKey: "aml:user-456" },
  body: { payload: { result: "clear", risk_score: 0.05 } },
});
```

## Step 4: Workflow Continues

Once the event arrives, the `aml-check` step completes and `create-account` starts automatically. The event payload is available to downstream steps via `{{parent_outputs.aml-check.result}}`.

## Timeout Handling

If the event never arrives within `timeout_secs`, the step fails. The workflow then follows its `on_failure` policy:

| Policy | Behavior |
| --- | --- |
| `fail_workflow` (default) | The entire workflow fails |
| `skip_dependents` | Downstream steps are skipped, other branches continue |
| `continue` | The timeout is ignored, dependents proceed |

You can also cancel a trigger explicitly to immediately fail the step:

```bash
curl -X DELETE https://strait.dev/v1/events/aml:user-456 \
  -H "Authorization: Bearer strait_..."
```

## Event Chaining

Use `event_emit_key` on a step to automatically emit an event when that step completes. This chains workflows without external HTTP calls.

**Workflow A** (data processing):
```json
{
  "steps": [
    {
      "step_ref": "process-batch",
      "job_id": "job_process",
      "event_emit_key": "batch-done:{{payload.batch_id}}"
    }
  ]
}
```

**Workflow B** (aggregation, triggered separately):
```json
{
  "steps": [
    {
      "step_ref": "wait-for-batch",
      "type": "wait_for_event",
      "event_key": "batch-done:{{payload.batch_id}}"
    },
    {
      "step_ref": "aggregate",
      "job_id": "job_aggregate",
      "depends_on": ["wait-for-batch"]
    }
  ]
}
```

When Workflow A's `process-batch` completes, it emits `batch-done:{id}`, which resumes Workflow B's `wait-for-batch` step.

## Batch Resolution

Resolve multiple triggers at once using prefix matching:

```bash
curl -X POST https://strait.dev/v1/events/prefix/batch-42:/send \
  -H "Authorization: Bearer strait_..." \
  -H "Content-Type: application/json" \
  -d '{"payload": {"approved": true}}'
```

This resolves all waiting triggers with keys starting with `batch-42:`.

## Durable Sleep

For time-based pauses, use a `sleep` step instead of an event trigger:

```json
{
  "step_ref": "wait-24h",
  "type": "sleep",
  "sleep_duration": "24h",
  "depends_on": ["send-email"]
}
```

No resources are held during the sleep -- Strait completes the step automatically when the time arrives.

## Real-Time Monitoring

Monitor a trigger's status with Server-Sent Events:

```bash
curl -N https://strait.dev/v1/events/aml:user-456/stream \
  -H "Authorization: Bearer strait_..."
```

The stream closes automatically when the trigger reaches a terminal state (`received`, `timed_out`, or `canceled`).
