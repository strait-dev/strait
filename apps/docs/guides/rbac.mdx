title: "RBAC & Policy Authorization"
description: "Roles, inheritance, resource policies, and tag policies in Strait."
---

Strait gives you four layers of authorization for user actors:

1. **Role permissions** (project roles)
2. **Role inheritance** (parent role chains -- a role inherits all permissions from its parent)
3. **Resource policies** (grant a user direct access to a specific resource)
4. **Tag policies** (grant access to any resource matching a tag)

API keys continue to use scope-based authorization.

## Core Endpoints

### Roles
- `POST /v1/roles`
- `GET /v1/roles`
- `GET /v1/roles/{roleID}`
- `PATCH /v1/roles/{roleID}`
- `DELETE /v1/roles/{roleID}`
- `POST /v1/seed-roles`

### Members
- `POST /v1/members`
- `POST /v1/members/bulk`
- `GET /v1/members`
- `DELETE /v1/members/{userID}`

### Role Lineage Introspection
- `GET /v1/roles/{roleID}?include_lineage=true`

### Resource Policies
- `POST /v1/resource-policies`
- `GET /v1/resource-policies` (cursor pagination)
- `DELETE /v1/resource-policies/{policyID}`

### Tag Policies
- `POST /v1/tag-policies`
- `GET /v1/tag-policies` (cursor pagination)
- `DELETE /v1/tag-policies/{policyID}`

## Permission Resolution Order

For user actors, Strait resolves access in this order:

1. Role/inherited role permissions
2. Resource policy permissions for `(resource_type, resource_id, user_id)`
3. Tag policy permissions for matching resource tags

## Notes

- Strait rate-limits RBAC mutation endpoints with stricter control-plane limits.
- The permission cache is automatically invalidated when you change memberships or policies.
- Audit events are emitted for RBAC mutations.
