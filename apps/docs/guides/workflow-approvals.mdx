Approval steps pause a workflow until a person approves or rejects. Use them for deployment gates, compliance reviews, or any step that needs human sign-off before continuing.

## Adding an Approval Step

Add a step with `type: "approval"` to your workflow definition. The step pauses when reached and waits for a human to act.

```json
{
  "name": "Deploy Pipeline",
  "slug": "deploy-pipeline",
  "steps": [
    { "step_ref": "build", "job_id": "job_build" },
    { "step_ref": "test", "job_id": "job_test", "depends_on": ["build"] },
    {
      "step_ref": "prod-review",
      "type": "approval",
      "approval_approvers": ["alice", "bob"],
      "approval_timeout_secs": 3600,
      "depends_on": ["test"]
    },
    { "step_ref": "deploy", "job_id": "job_deploy", "depends_on": ["prod-review"] }
  ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `step_ref` | string | Yes | Unique name for the step |
| `type` | string | Yes | Must be `"approval"` |
| `approval_approvers` | string[] | Yes | Who can approve (at least one required) |
| `approval_timeout_secs` | int | No | Seconds before auto-expiry (omit for no timeout) |
| `depends_on` | string[] | No | Steps that must complete first |

Approval steps do not require a `job_id`. They are pure control-flow gates.

## Approving or Rejecting

When the workflow reaches the approval step, call the approve endpoint:

```bash
curl -X POST https://api.strait.dev/v1/workflow-runs/{workflow_run_id}/steps/{step_ref}/approve \
  -H "Authorization: Bearer strait_..." \
  -H "Content-Type: application/json" \
  -d '{"approver": "alice"}'
```

```typescript
await client.approveWorkflowStep({
  pathParams: {
    workflowRunID: "wfr_def456",
    stepRef: "prod-review",
  },
  body: { approver: "alice" },
});
```

On success, the step completes and downstream steps start. The response includes the updated workflow run, step run, and approval record with `status: "approved"` and `approved_by`.

To reject, call the same endpoint pattern with the reject action. A rejected step transitions to `failed` and the workflow follows its normal failure handling.

## Timeout Behavior

If `approval_timeout_secs` is set and no one acts in time:

1. The approval status is set to `rejected` with a timeout error.
2. The step transitions to `failed`.
3. The workflow follows its configured failure handling (retry, skip dependents, or fail).

<Tip>
Set timeouts for production workflows. An approval that waits forever is a workflow that never finishes.
</Tip>

## Notifications

When a step enters `waiting_for_approval`, Strait publishes a workflow run webhook event. Wire this to your notification system:

- **Slack**: Forward the webhook to a Slack incoming webhook URL.
- **Email**: Route through your notification service.
- **PagerDuty**: Use for time-sensitive approvals.

Approval steps also create a parallel [event trigger](/concepts/event-triggers), so you can resolve them through the event API or monitor status via SSE streaming.

## Multiple Approval Gates

A single workflow can include several approval steps at different stages:

```json
[
  { "step_ref": "build", "job_id": "job_build" },
  {
    "step_ref": "qa-review",
    "type": "approval",
    "approval_approvers": ["qa-lead"],
    "approval_timeout_secs": 7200,
    "depends_on": ["build"]
  },
  { "step_ref": "staging", "job_id": "job_staging_deploy", "depends_on": ["qa-review"] },
  {
    "step_ref": "prod-review",
    "type": "approval",
    "approval_approvers": ["eng-manager", "cto"],
    "approval_timeout_secs": 86400,
    "depends_on": ["staging"]
  },
  { "step_ref": "production", "job_id": "job_prod_deploy", "depends_on": ["prod-review"] }
]
```
