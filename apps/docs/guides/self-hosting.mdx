Strait ships as a single Go binary, so self-hosting is mostly a matter of running that binary next to PostgreSQL, Redis, and Sequin. This guide covers the community edition. The hosted service at [strait.dev](https://strait.dev) is never required to run Strait yourself.

The community edition has no cloud billing limits. It does not include the hosted cloud dashboard as a service, vendor support, BYO-cloud contracts, or launch-disabled roadmap features such as SSO/SAML.

There are two common setups. Pick the one that matches where you want the dashboard to live.

| | Vercel dashboard | Full Docker stack |
|---|---|---|
| Setup | One click, then environment variables | `make selfhost` |
| Dashboard runs on | Your Vercel account | Your hardware |
| API runs on | Any host the dashboard can reach | Docker Compose on the same host |
| PostgreSQL | Neon, Supabase, or any hosted/self-hosted instance | Bundled `postgres:18-alpine` |
| Best for | Teams that want a managed dashboard | On-prem or Docker-first teams |

For production scaling, region strategy, and splitting API and worker capacity, read [Deployment](/guides/deployment) after you have a stack running.

## Option 1: Full Docker stack

This is the fastest way to get everything running on one host.

### Prerequisites

- [Docker](https://docs.docker.com/get-docker/) and Docker Compose v2 or newer
- 2 GB RAM minimum, 4 GB recommended
- Ports 3000, 8080, 5432, and 6379 free

### Quick start

```bash
git clone https://github.com/strait-dev/strait.git
cd strait
make selfhost
```

`make selfhost` generates secrets with `./packages/scripts/selfhost-init.sh`, then starts the stack with `docker compose --env-file .env.selfhost -f docker-compose.base.yml -f docker-compose.selfhost.yml --profile core --profile dashboard up -d`.

Check the API and open the dashboard:

```bash
curl http://localhost:8080/health
# {"edition":"community","status":"ok"}

open http://localhost:3000
```

Stop or reset the stack:

```bash
make selfhost-down    # stop containers, keep data
make selfhost-reset   # stop and wipe data and secrets
```

### What runs

| Service | Image | Port | Purpose |
|---|---|---|---|
| `strait-api` | `ghcr.io/strait-dev/strait` | 8080 | API server and worker |
| `strait-app` | `ghcr.io/strait-dev/strait-app` | 3000 | Dashboard |
| `strait-postgres` | `postgres:18-alpine` | 5432 | Source of truth and queue |
| `strait-redis` | `redis:8-alpine` | 6379 | Pub/sub, SSE, worker-plane signalling |
| `strait-sequin` | `sequin/sequin` | internal | Change data capture feeding real-time updates |

`docker-compose.base.yml` defines the required runtime and `docker-compose.selfhost.yml` pins the self-host images to known-good release tags, so `docker compose up` reproduces the same combination every time. Override `STRAIT_IMAGE` or `STRAIT_APP_IMAGE` to track a different tag.

## Option 2: Vercel dashboard

[![Deploy with Vercel](https://vercel.com/button)](https://vercel.com/new/clone?repository-url=https%3A%2F%2Fgithub.com%2Fstrait-dev%2Fstrait&project-name=strait-app&repository-name=strait-app&root-directory=apps%2Fapp&install-command=cd+..%2F..+%26%26+bun+install+--frozen-lockfile&build-command=cd+..%2F..+%26%26+cd+apps%2Fapp+%26%26+bun+run+build%3Avercel&env=AUTH_DATABASE_URL%2CBETTER_AUTH_URL%2CBETTER_AUTH_SECRET%2CSTRAIT_API_URL%2COIDC_ISSUER%2COIDC_AUDIENCE%2COIDC_PRIVATE_KEY_PEM)

The button forks the repo and starts a Vercel import. Keep the root directory set to `apps/app` and use the build command documented in [apps/app/README.md](https://github.com/strait-dev/strait/blob/master/apps/app/README.md).

The dashboard still needs a Strait API server. Run the API with Option 1, on a VPS, or on any public host Vercel can reach, then point `STRAIT_API_URL` at it in Vercel environment variables. `AUTH_DATABASE_URL` must point at a PostgreSQL database reachable from Vercel. Set `BETTER_AUTH_URL` to the final dashboard origin so cookies and OAuth callback URLs are scoped correctly.

## Dependencies

| Dependency | Required for |
|---|---|
| PostgreSQL 18+ | Source of truth and PgQue-backed ready-event log |
| Redis 8+ | Pub/sub, SSE streaming, gRPC worker-plane signalling |
| Sequin v0.14.6 | CDC stream from PostgreSQL to Strait |

All three are required. Strait fails fast when PostgreSQL, Redis, or Sequin is missing or unreachable. PostgreSQL must run with logical replication enabled (`wal_level=logical`) so Sequin can stream changes.

## Resource sizing

| Install size | Host size | Recommended settings |
|---|---|---|
| Laptop / development | 2 CPU, 2-4 GB RAM | `WORKER_CONCURRENCY=10`, `DB_MAX_CONNS=30` |
| Small self-host | 2 CPU, 4 GB RAM | `WORKER_CONCURRENCY=25`, `DB_MAX_CONNS=50` |
| Production single node | 4 CPU, 8 GB RAM | `WORKER_CONCURRENCY=50`, `DB_MAX_CONNS=100` |

Keep `DB_MAX_CONNS` greater than or equal to `WORKER_CONCURRENCY`, then add headroom for API requests, migrations, Sequin, and the dashboard. If you scale Strait horizontally, multiply `DB_MAX_CONNS` by the number of Strait containers and make sure PostgreSQL `max_connections` can handle the total.

## Configuration

Edit `.env.selfhost`, or add entries to the `environment:` block in `docker-compose.selfhost.yml`. The variables teams change most often:

| Variable | Default | Description |
|---|---|---|
| `WORKER_CONCURRENCY` | `25` | Maximum concurrent job executions |
| `LOG_LEVEL` | `info` | `debug`, `info`, `warn`, or `error` |
| `SEQUIN_API_TOKEN` | generated | Bearer token shared by Strait and Sequin for CDC management API calls |
| `RATE_LIMIT_REQUESTS` | `100` | Requests per minute per IP |
| `DEFAULT_JOB_TIMEOUT_SECS` | `300` | Default job timeout |
| `DEFAULT_JOB_MAX_ATTEMPTS` | `3` | Default retry attempts |

The full list lives in [Environment Variables](/configuration/environment-variables).

## Compose profiles

| Profile | Included services | Command |
|---|---|---|
| `core` | PostgreSQL, Redis, Sequin, migrations, Strait API/worker | `make selfhost-core` |
| `dashboard` | Dashboard and dashboard migrations | `make selfhost` |
| `observability` | Prometheus scrape endpoint | `make selfhost-observability` |

The default `make selfhost` command starts `core` and `dashboard`. Use `make selfhost-core` for an API-only host.

## Your first job

Open `http://localhost:3000`, sign up, and use the **Create project** wizard, which walks through installing an SDK and triggering a run.

To skip the dashboard, drive the API directly with the internal secret from `.env.selfhost`:

```bash
SECRET="<INTERNAL_SECRET from .env.selfhost>"

curl -X POST http://localhost:8080/v1/projects \
  -H "Content-Type: application/json" \
  -H "X-Internal-Secret: $SECRET" \
  -d '{"id": "my-project", "org_id": "my-org", "name": "My Project"}'

API_KEY=$(curl -s -X POST http://localhost:8080/v1/api-keys \
  -H "Content-Type: application/json" \
  -H "X-Internal-Secret: $SECRET" \
  -d '{"project_id": "my-project", "name": "dev-key", "scopes": ["*"]}' | jq -r '.key')

curl -X POST http://localhost:8080/v1/jobs \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"project_id":"my-project","name":"My Job","slug":"my-job","endpoint_url":"https://httpbin.org/post","max_attempts":3,"timeout_secs":30}'
```

## Upgrading

```bash
docker compose --env-file .env.selfhost -f docker-compose.base.yml -f docker-compose.selfhost.yml --profile core --profile dashboard pull
docker compose --env-file .env.selfhost -f docker-compose.base.yml -f docker-compose.selfhost.yml --profile core --profile dashboard up -d
```

Migrations run automatically on startup.

## Backups

Use the bundled script, which gzips a dump into `./backups` and prunes files older than 30 days:

```bash
./packages/scripts/selfhost-backup.sh
# Backup saved to ./backups/strait_20260325_120000.sql.gz
```

Restore a dump into the running database:

```bash
gunzip -c backups/strait_20260325_120000.sql.gz | docker exec -i strait-postgres psql -U strait strait
```

Schedule daily backups with cron:

```bash
0 3 * * * cd /path/to/strait && ./packages/scripts/selfhost-backup.sh
```
