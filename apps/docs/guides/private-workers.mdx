Use worker mode when job code should stay inside your private network. Instead of exposing an HTTP endpoint, you run a worker process that connects to Strait and receives work over that connection.

## When to Use Worker Mode

Worker mode is a good fit when:

- Your code runs in a private VPC or on-prem network.
- You do not want to expose a public job endpoint.
- The job needs local files, private databases, or long-lived connections.
- You want one process to handle many jobs with shared setup.

Use HTTP mode when you already have an endpoint Strait can call.

## Create a Worker Job

Set `execution_mode` to `worker`.

The request body fixture lives at `apps/docs/examples/requests/create-worker-job.json`.

```bash
curl -sS -X POST "$STRAIT_BASE_URL/v1/jobs" \
  -H "Authorization: Bearer $STRAIT_API_KEY" \
  -H "Content-Type: application/json" \
  --data @apps/docs/examples/requests/create-worker-job.json | jq
```

The `queue_name` lets you route work to the right set of workers.

## Write the Handler

A worker-mode handler is the same as an HTTP handler: it receives a run context and the payload, and returns a result. The difference is that the worker process connects to Strait instead of exposing an endpoint. With the TypeScript SDK, define the job once with `defineJob`:

```typescript
import { defineJob } from '@strait/ts';

export const trainModel = defineJob({
  name: 'train-model',
  timeout: '30m',
  handler: async (ctx, payload: { datasetUrl: string }) => {
    await ctx.log(`Training on ${payload.datasetUrl}`);

    const model = await train(payload.datasetUrl, {
      onProgress: (pct) => ctx.progress(pct),
    });

    return { modelId: model.id, accuracy: model.accuracy };
  },
});
```

The handler runs inside your network, so it can reach private databases, local files, and internal services directly. See [SDK Integration](/guides/sdk-integration) for the full run-context API.

## Start Workers

Workers are run with the Strait CLI, which scans your project's `dirs` for job definitions and drains the configured queue. Set the worker options in `strait.json`:

```json
{
  "worker": {
    "concurrency": 10,
    "queue_name": "ml"
  }
}
```

| Field | Description |
| :--- | :--- |
| `worker.concurrency` | How many runs one worker process executes at once. |
| `worker.queue_name` | The queue this worker drains. Must match the job's `queue_name`. |
| `worker.grpc_url` | gRPC endpoint for the worker connection. Defaults to a value derived from `sdk.base_url`. |

See [SDK Configuration](/sdks/configuration) for the full `worker` block and environment-variable overrides.

## Scale Workers

To increase throughput, run more worker processes for the same queue. Strait distributes matching runs across connected workers.

Keep workers close to the systems they call. If a job reads from a private database, run the worker near that database.

## Failure Behavior

If a worker disconnects while processing a run, Strait returns unfinished work to the queue. The run then follows the job's retry policy.

Use idempotent handlers when possible. A run may be attempted more than once after timeouts, disconnects, or process restarts.
