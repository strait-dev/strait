This playbook covers common workflow incidents and how to resolve them. For background on how the DAG engine works, see [DAG Runtime](/concepts/dag-runtime). For workflow modeling, see [Workflows](/concepts/workflows).

## Stuck Workflows

**Symptoms**: A `workflow_run` stays in `running` status with no step progression. The dashboard shows no active steps.

**Diagnosis**:

1. List steps for the workflow run:
   ```
   GET /v1/workflow-runs/{workflowRunID}/steps
   ```
2. Look for steps stuck in `pending` or `waiting` with all dependencies satisfied.
3. Check if the workflow is paused (`status: paused`).
4. Check for steps waiting on an [event trigger](/concepts/event-triggers) that has not arrived.

**Recovery**:

- If the workflow is paused, resume it:
  ```
  PATCH /v1/workflow-runs/{workflowRunID}
  { "status": "running" }
  ```
- If a step is stuck on an event trigger, either send the expected event or skip the step:
  ```
  POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/skip
  ```
- If a step's job run completed externally but the callback was lost, force-complete the step:
  ```
  POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/force-complete
  ```
- If the workflow is irrecoverable, cancel it:
  ```
  POST /v1/workflow-runs/{workflowRunID}/cancel
  ```

## Failed Steps

**Symptoms**: A step transitions to `failed`. Depending on the workflow's `on_failure` policy, the workflow may continue, skip dependents, or fail entirely.

**Diagnosis**:

1. Get the step's underlying job run to see logs and error details:
   ```
   GET /v1/workflow-runs/{workflowRunID}/steps/{stepRef}
   ```
2. Check the job run's `error` field and run events for the root cause.
3. Determine if the failure is transient (network timeout, rate limit) or permanent (bad input, missing resource).

**Recovery**:

- **Retry the step** (transient failures):
  ```
  POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/retry
  ```
  The step's attempt counter increments and it re-enters the execution pipeline. The workflow run stays in `running`.

- **Replay the subtree** (when downstream steps also need re-execution):
  ```
  POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/replay-subtree
  ```
  Resets the selected step and all its descendants to `pending`, then resumes execution. Steps outside the branch are untouched.

- **Skip the step** (non-critical step):
  ```
  POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/skip
  ```
  Marks the step as `skipped` and unblocks dependents.

- **Retry the entire workflow run** (widespread failure):
  Creates a new run that copies completed steps and re-executes from the first failure.

## Timeout Recovery

**What happens**: When a step exceeds its timeout, the engine marks it as `timed_out`. The workflow then follows its `on_failure` policy.

**Adjusting timeouts**: Update the workflow definition with a longer `timeout` on the affected step, then re-trigger the workflow. Timeouts are per-step, not per-workflow.

**Workflow-level timeout**: If `max_duration` is set on the workflow, the entire run is canceled when the duration exceeds the limit. All running steps are canceled.

**Running workflows**: You cannot extend the timeout of a running workflow. To work around this:

1. Cancel the current run.
2. Update the workflow definition with the new timeout.
3. Re-trigger the workflow (or retry the failed run, which picks up the new definition).

## Circular Dependency Errors

**How they manifest**: The workflow engine validates the DAG structure on creation and update. If you add a step dependency that creates a cycle, the API returns a `400` error with a message identifying the cycle.

**How to fix**:

1. Review the step `depends_on` fields in your workflow definition.
2. Draw or visualize the dependency graph. Use the simulation endpoint to get the DAG structure:
   ```
   POST /v1/workflows/{workflowID}/simulate
   { "mode": "dry_run" }
   ```
3. Remove or redirect the dependency that closes the cycle.

## Approval Gate Stuck

**Symptoms**: A step with `type: approval` stays in `waiting` indefinitely.

**Recovery**:

- Approve the step:
  ```
  POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/approve
  ```

- Reject the step:
  ```
  POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/reject
  ```

- Skip the step (bypasses the gate):
  ```
  POST /v1/workflow-runs/{workflowRunID}/steps/{stepRef}/skip
  ```

**Timeout behavior**: Approval steps respect the step's `timeout` setting. When the timeout expires, the step transitions to `timed_out` and follows the workflow's `on_failure` policy. Set generous timeouts for human approvals (hours or days).

## Bulk Recovery

For incidents affecting many workflow runs:

- **Bulk cancel stuck runs**:
  ```
  POST /v1/runs/bulk-cancel
  { "run_ids": ["run_1", "run_2", ...] }
  ```
  Cancels up to 100 runs per request. Returns per-run status.

- **Bulk replay failed runs**:
  ```
  POST /v1/runs/bulk-replay
  { "run_ids": ["run_1", "run_2", ...] }
  ```

For workflow-specific operations, iterate over affected workflow runs using the list endpoint with status filters:
```
GET /v1/workflow-runs?workflow_id={workflowID}&status=failed
```

## Prevention

**Set timeouts on every step.** Steps without timeouts can run (or wait) indefinitely. Default to a timeout that matches the expected duration plus a margin.

**Use stall detection.** The engine checks for workflow runs stuck in `running` with no step progression. Configure the threshold with `WF_STALL_THRESHOLD` (default 15m). The action is controlled by `WF_STALL_ACTION`:
- `log_only` (default): Logs a warning. No automatic intervention.
- You can build alerting on these logs to catch stuck workflows early.

**Monitor workflow duration metrics.** Track `workflow_run_analytics.duration_ms` in [ClickHouse](/concepts/clickhouse-analytics) or via the Prometheus metric `strait_workflow_run_duration_seconds` to spot regressions.

**Use the simulator before deploying workflow changes.** Dry-run mode catches structural issues (missing dependencies, unreachable steps) before they affect production:
```
POST /v1/workflows/{workflowID}/simulate
{ "mode": "dry_run" }
```

**Configure `on_failure` policies deliberately.** The default (`fail_workflow`) is safest but least resilient. For workflows where partial completion is acceptable, use `skip_dependents` or `continue`.
