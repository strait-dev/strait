Strait runs as a single Go binary. You can run everything in one process or split the API and workers as traffic grows.

## Required Services

| Service | Purpose |
|---|---|
| PostgreSQL | Stores projects, jobs, runs, workflows, and queued work. |
| Redis | Powers the worker transport plane, rate limiting, and pub/sub notifications. |
| Sequin | Streams PostgreSQL changes through CDC for real-time state propagation without database polling. |

Use managed PostgreSQL and Redis if you want the lowest operational overhead. Sequin must run alongside the database, and PostgreSQL must enable logical replication (`wal_level=logical`). Strait fails fast when PostgreSQL, Redis, or Sequin is missing or unreachable.

## Required Secrets

Set these through your platform's secret manager.

| Variable | Description |
|---|---|
| `DATABASE_URL` | PostgreSQL connection string. |
| `REDIS_URL` | Redis connection string. |
| `SEQUIN_BASE_URL` | Sequin API base URL. |
| `SEQUIN_CONSUMER_NAME` | Sequin HTTP pull consumer name. |
| `INTERNAL_SECRET` | Trusted service-to-service secret. |
| `JWT_SIGNING_KEY` | Signs SDK run tokens. |
| `ENCRYPTION_KEY` | Encrypts data at rest. |
| `SECRET_ENCRYPTION_KEY` | Encrypts stored secrets. |

Generate unique values per environment. Do not reuse production secrets in staging or development.

## Choose a Mode

| Mode | Use it when |
|---|---|
| `all` | You are running a small deployment or local environment. |
| `api` | You want separate capacity for API traffic. |
| `worker` | You want separate capacity for job execution. |

Most teams start with `all`, then split into `api` and `worker` when queue volume grows.

## Docker

Build the image:

```bash
docker build -t strait apps/strait/
```

Build with release metadata for `/health` and Sentry release tagging:

```bash
docker build \
  --build-arg VERSION=v1.2.3 \
  --build-arg COMMIT=$(git rev-parse --short HEAD) \
  --build-arg DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ) \
  -t strait:v1.2.3 apps/strait/
```

Run a single-process deployment:

```bash
docker run \
  -e DATABASE_URL=postgres://... \
  -e REDIS_URL=redis://... \
  -e SEQUIN_BASE_URL=http://sequin:7376 \
  -e SEQUIN_CONSUMER_NAME=strait-cdc \
  -e INTERNAL_SECRET=... \
  -e JWT_SIGNING_KEY=... \
  -e ENCRYPTION_KEY=... \
  -e SECRET_ENCRYPTION_KEY=... \
  strait --mode all
```

The deploy workflow runs preflight checks (`go vet`, `golangci-lint`, `go build`, `go test`) before pushing. If any check fails, the deploy is aborted.

```bash
# From apps/strait/. Runs vet, lint, build, test before deploying.
make deploy

# Or from repo root
bun run deploy
```

## Sentry Releases

When `SENTRY_DSN` is configured, Strait tags events with `SENTRY_RELEASE`.
If that variable is empty, the server derives the release from the binary
version and commit injected at build time (`VERSION+COMMIT`). Use the same
value when creating Sentry releases:

```bash
export STRAIT_VERSION=v1.2.3
export STRAIT_COMMIT=$(git rev-parse --short HEAD)
export SENTRY_RELEASE="${STRAIT_VERSION}+${STRAIT_COMMIT}"

sentry-cli releases new "$SENTRY_RELEASE"
sentry-cli releases set-commits "$SENTRY_RELEASE" --auto
sentry-cli releases finalize "$SENTRY_RELEASE"
```

Set `SENTRY_RELEASE` on every deployed API and worker process to keep errors,
transactions, and breadcrumbs tied to the exact build.

For dashboard deploys, upload sourcemaps after the frontend build and before
publishing the bundle:

```bash
cd apps/app
export STRAIT_VERSION="$STRAIT_VERSION"
export STRAIT_COMMIT="$STRAIT_COMMIT"
export SENTRY_ORG=your-org
export SENTRY_PROJECT=your-dashboard-project
export SENTRY_AUTH_TOKEN="<sentry-auth-token>"

bun run build
bun run sentry:upload-sourcemaps
```

The upload script uses the same release name (`SENTRY_RELEASE`, or
`STRAIT_VERSION+STRAIT_COMMIT`) as the Go server. It uploads generated `.map`
files from `dist` or `.output`, then removes `.map` files from public output
directories so the deployed dashboard bundle does not expose sourcemaps.

Manual smoke check: after deploy, trigger a handled dashboard error in the
release environment and confirm Sentry shows an un-minified stack trace under
the same release as the API and worker events.

## Process Deployment

Run Strait as one or more long-lived processes. For small deployments, run a
single `--mode all` process. For higher throughput, split API and worker
capacity into separate processes by passing `--mode api` or `--mode worker`.

## Sequin Deployment

Sequin watches your database for changes (new runs, status updates, workflow progress) and streams them through Redis so Strait can deliver real-time updates, SSE events, and webhooks. It runs as a separate container included in Docker Compose, pinned to `sequin/sequin:v0.14.6`. No build step needed.

```bash
# From repo root
bun run deploy:sequin
```

Sequin requires the following secrets configured in your deployment platform:

| Secret | Description |
| :--- | :--- |
| `PG_HOSTNAME` | PostgreSQL host |
| `PG_PORT` | PostgreSQL port |
| `PG_USERNAME` | PostgreSQL user |
| `PG_PASSWORD` | PostgreSQL password |
| `PG_DATABASE` | Database name |
| `PG_SSL` | Enable SSL (`true`) |
| `REDIS_URL` | Redis connection string |
| `SECRET_KEY_BASE` | Sequin internal encryption key |
| `VAULT_KEY` | Sequin vault encryption key |

### Region Strategy

- **Primary region**: Co-locate Strait with your database for lowest latency.
- **Edge regions**: Deploy additional instances in other regions for faster API responses globally.
- **Sequin**: Single instance, co-located with the database. Sequin maintains a persistent connection to the database to detect changes, so it must run in the same region. Multiple instances would cause duplicate event processing.

For community self-hosting, see the [Self-Hosting](/guides/self-hosting) guide.

## Scaling

Scale API instances for request volume. Scale worker instances for job throughput.

A common production layout:

```text
Load balancer
    |
    +-- strait --mode api
    +-- strait --mode api

Workers
    |
    +-- strait --mode worker
    +-- strait --mode worker
```

All instances share the same PostgreSQL, Redis, and Sequin runtime contract.

## Baseline sizing

| Deployment | Starting point | Notes |
|---|---|---|
| Development | 2 CPU, 2-4 GB RAM | Use `make dev`; the Compose stack exposes Postgres on `15432`, Redis on `16379`, Sequin on `7376`, and Strait on `8080`. |
| Small self-host | 2 CPU, 4 GB RAM | Keep `WORKER_CONCURRENCY=25` and `DB_MAX_CONNS=50` until load data says otherwise. |
| Production single node | 4 CPU, 8 GB RAM | Start with `WORKER_CONCURRENCY=50` and `DB_MAX_CONNS=100`, then tune from queue depth, DB pool, and Redis latency metrics. |

The Docker Compose defaults are intentionally conservative. Raise worker concurrency and Redis/Postgres connection pools together, and keep Sequin in the same region as PostgreSQL.

## Placement

Keep Strait close to PostgreSQL. Database latency affects API calls, scheduling, and worker throughput.

Place workers near the services they call. If a worker talks to private databases or private APIs, run it in the same network.

## Shutdown

Strait handles `SIGTERM` gracefully:

- API instances stop accepting new requests and finish in-flight requests.
- Worker instances stop claiming new work and let active runs finish.

Use a process manager or container platform that sends `SIGTERM` and gives the process time to exit.

## Production Checklist

- Use managed PostgreSQL backups or your own backup policy.
- Store secrets in a secret manager.
- Set `LOG_LEVEL=info` or `warn`.
- Configure tracing, metrics, or Sentry if your team uses them.
- Set job-level timeouts and retry limits.
- Use idempotency keys for important trigger requests.
- Run at least two API instances when availability matters.
- Run enough workers for your expected queue volume.
