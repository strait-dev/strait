Strait ships as a single Go binary. You can deploy it anywhere from Docker Compose locally to Docker or Kubernetes in production.

## Architecture Overview

Production deployment consists of the following services:

| Service | Platform | Purpose |
| :--- | :--- | :--- |
| **Strait** (Go backend) | Docker / Kubernetes | API server + worker |
| **Sequin** (CDC) | Docker / Kubernetes | Postgres WAL to Redis pub/sub |
| **PostgreSQL** | Managed (e.g., PlanetScale, RDS) | Primary data store |
| **Redis** | Managed (e.g., Upstash, ElastiCache) | Pub/sub, SSE streaming, CDC events |
| **Frontend** | -- | `apps/app` (Vite/React) |

### Region Strategy

- **Primary region**: Co-locate Strait with your database for lowest latency.
- **Edge regions**: Deploy additional instances in other regions for faster API responses globally.
- **Sequin**: Single instance, co-located with the database. CDC requires a persistent WAL connection and must run next to the database. Multiple instances would cause duplicate event processing.

## Secrets Management

All environment variables are managed via [Doppler](https://doppler.com) under the `strait` project with three configs:

| Config | Environment | Notes |
| :--- | :--- | :--- |
| `dev` | Development | Local defaults, localhost URLs |
| `stg` | Staging | Shared infra with production |
| `prd` | Production | Production secrets and URLs |

### Local Development

```bash
# Configure Doppler for this directory (one-time)
doppler setup  # select project: strait, config: dev

# Run any command with secrets injected
doppler run -- go run ./cmd/strait
doppler run -- bun run dev

# Or generate a local .env file
doppler secrets download --no-file --format env > .env
```

### Platform Integration

Secrets are synced from Doppler to your deployment platform via Doppler integrations. When you update a secret in Doppler, it automatically propagates to your containers.

## Operation Modes

Strait can run in three different modes, controlled by the `--mode` flag or the `MODE` environment variable:

- **`api`**: Runs only the HTTP API server. Stateless and can be scaled horizontally.
- **`worker`**: Runs only the job execution engine. Scales based on queue depth.
- **`all`** (Default): Runs both the API and the worker in a single process. Ideal for small deployments or development.

## Infrastructure Requirements

Strait requires the following backing services:

- **PostgreSQL 18**: Primary data store and job queue.
- **Redis 8**: Pub/sub, SSE streaming, and CDC event publishing.
- **Sequin**: Change Data Capture (CDC) -- streams database changes from the Postgres write-ahead log to Redis.

### Docker Compose (Development)
For local development, use `apps/strait/docker-compose.yml`:
```bash
docker compose -f apps/strait/docker-compose.yml up -d
```

## Container Deployment

### Project Structure

```
apps/strait/
├── Dockerfile      # Multi-stage Go build
├── Makefile        # Build, test, and deploy commands
└── .dockerignore
```

### Building and Deploying

The Dockerfile uses a multi-stage build: Go compile on Alpine, then copy the binary to a minimal runtime image (~16 MB).

```bash
# From apps/strait/ — runs vet, lint, build, test before deploying
make deploy

# Or from repo root
bun run deploy
```

The deploy workflow runs preflight checks (`go vet`, `golangci-lint`, `go build`, `go test`) before pushing. If any check fails, the deploy is aborted.

### Deploying Sequin

Sequin uses the official `sequin/sequin:latest` Docker image -- no build step needed.

```bash
# From repo root
bun run deploy:sequin
```

Sequin requires the following secrets configured in your deployment platform:

| Secret | Description |
| :--- | :--- |
| `PG_HOSTNAME` | PostgreSQL host |
| `PG_PORT` | PostgreSQL port |
| `PG_USERNAME` | PostgreSQL user |
| `PG_PASSWORD` | PostgreSQL password |
| `PG_DATABASE` | Database name |
| `PG_SSL` | Enable SSL (`true`) |
| `REDIS_URL` | Redis connection string |
| `SECRET_KEY_BASE` | Sequin internal encryption key |
| `VAULT_KEY` | Sequin vault encryption key |

### CI/CD

The GitHub Actions workflow (`.github/workflows/deploy.yml`) automatically deploys when:

1. Code is pushed to `master`
2. Changes are in `apps/strait/**`
3. All required workflows pass (`Test`, `Lint`, `Security`)

## Production Configuration

### Required Environment Variables
| Variable | Description |
| :--- | :--- |
| `DATABASE_URL` | Postgres connection string |
| `REDIS_URL` | Redis connection string |
| `INTERNAL_SECRET` | Secret for internal API auth (32+ chars, unique per environment) |
| `JWT_SIGNING_KEY` | Key for signing SDK run tokens (32+ chars, unique per environment) |
| `ENCRYPTION_KEY` | Data encryption at rest (unique per environment) |
| `SECRET_ENCRYPTION_KEY` | Secret values encryption (unique per environment) |

See [Environment Variables](/configuration/environment-variables) for the full reference.

### Scaling Strategy
To handle higher traffic, separate the API and worker processes:

1. **Scale API**: Add more instances in `api` mode to handle more concurrent HTTP requests.
2. **Scale Workers**: Add more instances in `worker` mode to process more jobs. Workers safely compete for jobs without double-processing.

## Graceful Shutdown

Strait shuts down gracefully in both modes:
- **API**: Stops accepting new requests and waits for active requests to finish (up to the configured timeout).
- **Worker**: Stops claiming new jobs and waits for currently executing jobs to complete or reach a checkpoint before exiting.

<Tip>
  Always use a process manager or orchestrator (like Kubernetes or Docker Compose) that respects SIGTERM signals to ensure no jobs are abruptly interrupted.
</Tip>

## Quick Reference

```bash
# Deploy main app (with preflight checks)
make deploy                        # from apps/strait/

# Deploy Sequin
bun run deploy:sequin              # from repo root

# Local dev with Doppler
doppler run -- go run ./cmd/strait
```
