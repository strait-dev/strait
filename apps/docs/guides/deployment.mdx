Strait ships as a single Go binary. Deploy it with Docker, Kubernetes, or any platform that runs containers.

## What You Need

| Service | Required | Purpose |
| :--- | :--- | :--- |
| **PostgreSQL** | Yes | Primary data store and job queue |
| **Redis** | Yes | Pub/sub, SSE streaming, real-time event distribution |
| **Sequin** | Yes | Streams database changes for real-time updates and webhooks |

See [Architecture](/concepts/architecture) for how these services interact.

### Managed Service Options

- **PostgreSQL**: RDS, Cloud SQL, or Supabase
- **Redis**: Upstash, ElastiCache, or Memorystore
- **Sequin**: Self-hosted Docker image (no managed offering)

## Secrets Management

Store secrets in your platform's secret manager: AWS Secrets Manager, GCP Secret Manager, Kubernetes Secrets, or environment variables.

### Required Environment Variables

| Variable | Description |
| :--- | :--- |
| `DATABASE_URL` | PostgreSQL connection string |
| `REDIS_URL` | Redis connection string |
| `INTERNAL_SECRET` | Secret for internal API auth (32+ chars, unique per environment) |
| `JWT_SIGNING_KEY` | Key for signing SDK run tokens (32+ chars, unique per environment) |
| `ENCRYPTION_KEY` | Data encryption at rest (unique per environment) |
| `SECRET_ENCRYPTION_KEY` | Secret values encryption (unique per environment) |

<Warning>
Never commit secrets to version control. Use your platform's secret injection to provide these at runtime.
</Warning>

```bash
# Start local infra
docker compose -f apps/strait/docker-compose.yml up -d

# Run the Go API against local Docker services
cd apps/strait
DATABASE_URL=postgres://strait:strait@localhost:5432/strait?sslmode=disable \
REDIS_URL=redis://localhost:6379 \
INTERNAL_SECRET=strait-local-internal-secret-32chars \
JWT_SIGNING_KEY=strait-local-jwt-signing-key-32chars \
go run ./cmd/strait --mode all

# In another terminal, run the dashboard locally
cd apps/app
bun run dev
```

`bun run dev` is the standard local path for contributors and self-host users.
It defaults the app to the local Docker PostgreSQL auth database and disables
email verification and Polar billing in development.

See [Environment Variables](/configuration/environment-variables) for the full reference.

## Operation Modes

Strait can run in three different modes, controlled by the `--mode` flag or the `MODE` environment variable:

- **`api`**: Runs only the HTTP API server. Stateless and can be scaled horizontally.
- **`worker`**: Runs only the job execution engine. Scales based on queue depth.
- **`all`** (Default): Runs both the API and the worker in a single process. Ideal for small deployments or development.

## Docker Deployment

### Docker Compose (Development)

For local development, use `apps/strait/docker-compose.yml`:

```bash
docker compose -f apps/strait/docker-compose.yml up -d
```

### Building the Image

The Dockerfile uses a multi-stage build: Go compile on Alpine, then copy the binary to a minimal runtime image (~16 MB).

```
apps/strait/
├── Dockerfile      # Multi-stage Go build
├── Makefile        # Build, test, and deploy commands
└── .dockerignore
```

```bash
# Build the image
docker build -t strait apps/strait/

# Run with required environment variables
docker run \
  -e DATABASE_URL=postgres://... \
  -e REDIS_URL=redis://... \
  -e INTERNAL_SECRET=... \
  -e JWT_SIGNING_KEY=... \
  -e ENCRYPTION_KEY=... \
  -e SECRET_ENCRYPTION_KEY=... \
  strait --mode all
```

The deploy workflow runs preflight checks (`go vet`, `golangci-lint`, `go build`, `go test`) before pushing. If any check fails, the deploy is aborted.

```bash
# From apps/strait/ — runs vet, lint, build, test before deploying
make deploy

# Or from repo root
bun run deploy
```

## Kubernetes Deployment

Deploy the Strait image as a Deployment with separate `api` and `worker` Deployments for independent scaling. Pass `--mode api` or `--mode worker` as the container command argument.

Store secrets in a Kubernetes Secret and mount them as environment variables:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: strait-secrets
type: Opaque
stringData:
  DATABASE_URL: "postgres://..."
  REDIS_URL: "redis://..."
  INTERNAL_SECRET: "..."
  JWT_SIGNING_KEY: "..."
  ENCRYPTION_KEY: "..."
  SECRET_ENCRYPTION_KEY: "..."
```

## Sequin Deployment

Sequin watches your database for changes (new runs, status updates, workflow progress) and streams them through Redis so Strait can deliver real-time updates, SSE events, and webhooks. It runs as a separate container included in Docker Compose, using the official `sequin/sequin:latest` image -- no build step needed.

```bash
# From repo root
bun run deploy:sequin
```

Sequin requires the following secrets configured in your deployment platform:

| Secret | Description |
| :--- | :--- |
| `PG_HOSTNAME` | PostgreSQL host |
| `PG_PORT` | PostgreSQL port |
| `PG_USERNAME` | PostgreSQL user |
| `PG_PASSWORD` | PostgreSQL password |
| `PG_DATABASE` | Database name |
| `PG_SSL` | Enable SSL (`true`) |
| `REDIS_URL` | Redis connection string |
| `SECRET_KEY_BASE` | Sequin internal encryption key |
| `VAULT_KEY` | Sequin vault encryption key |

### Region Strategy

- **Primary region**: Co-locate Strait with your database for lowest latency.
- **Edge regions**: Deploy additional instances in other regions for faster API responses globally.
- **Sequin**: Single instance, co-located with the database. Sequin maintains a persistent connection to the database to detect changes, so it must run in the same region. Multiple instances would cause duplicate event processing.

## Scaling

To handle higher traffic, separate the API and worker processes:

1. **Scale API**: Add more instances in `api` mode to handle more concurrent HTTP requests.
2. **Scale Workers**: Add more instances in `worker` mode to process more jobs. Workers safely compete for jobs without double-processing.

## Graceful Shutdown

Strait handles shutdown cleanly in both modes:
- **API**: Stops accepting new requests and waits for active requests to finish (up to the configured timeout).
- **Worker**: Stops claiming new jobs and waits for currently executing jobs to complete or reach a checkpoint before exiting.

<Tip>
  Always use a process manager or orchestrator (like Kubernetes or Docker Compose) that respects SIGTERM signals so in-flight jobs finish before the process exits.
</Tip>

## CI/CD

The GitHub Actions workflow (`.github/workflows/deploy.yml`) automatically deploys when:

1. Code is pushed to `master`
2. Changes are in `apps/strait/**`
3. All required workflows pass (`Test`, `Lint`, `Security`)

## Quick Reference

```bash
# Deploy with Docker
docker build -t strait apps/strait/
docker run -e DATABASE_URL=... -e REDIS_URL=... strait --mode all

# Deploy Sequin
docker run -e PG_HOSTNAME=... sequin/sequin:latest
```
