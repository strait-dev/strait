---
title: "Idempotency"
description: "Prevent duplicate job executions with idempotency keys and payload-based deduplication."
---

A user double-clicks "Place Order." Without idempotency, they get charged twice. Strait's idempotency prevents this -- every trigger with the same key returns the original run instead of creating a duplicate.

## Idempotency Keys

Pass an `X-Idempotency-Key` header when triggering a job. Strait checks whether a run with the same `job_id` and key already exists. If it does, Strait returns the existing run. If not, it creates a new one.

```bash
curl -X POST https://strait.dev/v1/jobs/{jobID}/trigger \
  -H "Authorization: Bearer $API_KEY" \
  -H "X-Idempotency-Key: order_123_charge" \
  -H "Content-Type: application/json" \
  -d '{
    "payload": { "order_id": "order_123" }
  }'
```

```typescript
const run = await client.triggerJob({
  pathParams: { jobID: "charge-card" },
  body: { payload: { orderId: "order_123" } },
  headers: { "X-Idempotency-Key": "order_123_charge" },
});
```

The key is enforced at the database level with a unique constraint on `(job_id, idempotency_key)`, so duplicates are impossible even under concurrent requests.

<Note>
If you trigger with the same idempotency key but a different payload, Strait returns the original run. The new payload is ignored.
</Note>

## Payload-Based Deduplication

For cases where you cannot control the caller (retry loops, webhook retries), configure automatic deduplication on the job itself with `dedup_window_secs`. Strait normalizes the JSON payload (sorting keys, stripping whitespace), hashes it, and checks for a matching run within the configured window.

```json
{
  "name": "charge-card",
  "dedup_window_secs": 60
}
```

Any trigger with an identical payload within the window returns the existing run.

## When to Use Each

| Method | Set by | Scope | Window |
| --- | --- | --- | --- |
| **Idempotency Key** | `X-Idempotency-Key` header | Per job | Permanent (until run deleted) |
| **Payload Dedup** | `dedup_window_secs` on the job | Per job | Configurable (e.g. 60s) |

<Tip>
Use `X-Idempotency-Key` for critical operations where you control the caller. Use `dedup_window_secs` as a safety net against accidental double-submits or client retry loops.
</Tip>

## Webhook Delivery Headers

Outbound webhook deliveries include two identity headers:

- `X-Strait-Idempotency-Key`: formatted as `<delivery_id>:<attempt>`, changes on every retry. Useful for tracing an individual attempt.
- `X-Strait-Replay-Key`: stable across every retry of the same delivery. Dedup on this header -- if your handler sees the same replay key twice, treat the second call as a no-op.
  - **Subscription deliveries** (event triggers with a `webhook_subscription`) use `"rk_" + hex(hmac_sha256(subscription_secret, delivery_id))[:32]`. A subscriber holding the secret can re-derive the key from `X-Strait-Delivery-ID` and verify it matches.
  - **Run-terminal deliveries** (the per-job `webhook_url` fired on terminal states) use `"rk_" + run_id`. Treat this as an opaque discriminator -- authenticate these deliveries via the `X-Strait-Signature` payload HMAC instead.
