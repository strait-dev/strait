Strait supports three authentication methods: a server secret for trusted backend access, API keys for applications, and OIDC bearer tokens for user-facing API requests. Strait Cloud users normally use project API keys. Self-hosted operators also use the server secret for management actions such as creating the first project.

OIDC bearer-token verification is operator-configured API authentication. It is separate from hosted dashboard SSO/SAML, which is a launch roadmap item and is not included in self-serve cloud plans.

## Quick Decision

| Method | Use when | Example |
|---|---|---|
| Server Secret | Trusted backend calls | `Authorization: Bearer YOUR_SECRET` |
| API Key | Applications and integrations | `Authorization: Bearer strait_sk_...` |
| OIDC bearer token | User API requests from your identity provider | `Authorization: Bearer eyJ...` |

## Server Secret

Use the server secret for trusted backend calls. Set it with the `INTERNAL_SECRET` environment variable.

```bash
curl -H "Authorization: Bearer $INTERNAL_SECRET" \
  "$STRAIT_BASE_URL/v1/jobs"
```

The server secret grants full access to all endpoints. Do not use it from browsers or mobile apps.

Your application uses this path to pass user identity to Strait via headers:

```bash
curl -H "Authorization: Bearer $INTERNAL_SECRET" \
     -H "X-Project-Id: proj_abc123" \
     -H "X-Actor-Id: user_abc123" \
     -H "X-Actor-Email: alice@example.com" \
     "$STRAIT_BASE_URL/v1/jobs"
```

When these headers are present, Strait records the actor for audit logging and applies RBAC checks for that user.

<Warning>
  The server secret grants unrestricted access. In production, keep it in backend services and secret managers only.
</Warning>

## API Keys

Per-project keys with scoped permissions. Use API keys for external integrations, CI/CD pipelines, and third-party access.

### Creating a Key

```bash
curl -X POST "$STRAIT_BASE_URL/v1/api-keys" \
  -H "Authorization: Bearer $INTERNAL_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_abc123",
    "name": "ci-deploy-key",
    "scopes": ["jobs:read", "jobs:write", "jobs:trigger"],
    "expires_in_days": 90
  }'
```

Set `expires_in_days` for short-lived automation keys. If the project has a `max_key_lifetime_days` policy, omitted expirations are auto-capped to that policy. The response includes the raw key once. Store it in your secret manager immediately.

```json
{
  "id": "key_abc123",
  "key": "strait_abc123...",
  "scopes": ["jobs:read", "jobs:write", "jobs:trigger"]
}
```

### Using a Key

```bash
curl -H "Authorization: Bearer strait_abc123..." \
  "$STRAIT_BASE_URL/v1/jobs"
```

### Scope Categories

| Category | Scopes | Controls |
|---|---|---|
| Jobs | `jobs:read`, `jobs:write`, `jobs:trigger` | View, manage, and trigger jobs |
| Runs | `runs:read`, `runs:write` | View and manage runs |
| Workflows | `workflows:read`, `workflows:write`, `workflows:trigger` | View, manage, and trigger workflows |
| Secrets | `secrets:read`, `secrets:write` | View and manage job secrets |
| Webhooks | `webhooks:read`, `webhooks:write` | View and manage webhook subscriptions |
| Projects | `projects:read`, `projects:write`, `projects:manage` | View, update, and administer projects |
| Workers | `workers:connect` | Connect a worker process over gRPC |
| Dead-letter runs | `dlq:read`, `dlq:replay`, `dlq:purge` | Inspect, replay, and purge dead-lettered runs |
| Outbox | `outbox:read`, `outbox:retry`, `outbox:purge` | Inspect, retry, and purge outbox events |
| Admin | `api-keys:manage`, `rbac:manage`, `stats:read` | Manage keys, roles, and view stats |
| Wildcard | `*` | Full access |

Keys with empty scopes (`[]`) retain full access for backwards compatibility with keys created before scope enforcement.

### Revoking a Key

```bash
curl -X DELETE "$STRAIT_BASE_URL/v1/api-keys/key_abc123" \
  -H "Authorization: Bearer $INTERNAL_SECRET"
```

Revoked keys immediately return 401 on subsequent requests.

## API Key Rotation

Rotate keys with a grace period for zero-downtime credential rollovers.

### Endpoint

`POST /v1/api-keys/{keyID}/rotate`

```bash
curl -X POST "$STRAIT_BASE_URL/v1/api-keys/key_abc123/rotate" \
  -H "Authorization: Bearer $INTERNAL_SECRET" \
  -H "Content-Type: application/json" \
  -d '{ "grace_period_minutes": 60 }'
```

### Grace Period Behavior

- A new key is created and returned (shown only once).
- The old key is marked with `replaced_by_key_id` and `grace_expires_at`.
- The old key remains valid until `grace_expires_at`.
- After grace expiry, the old key returns 401.

### Recommended Flow

1. Rotate the key with a grace window.
2. Deploy the new key to all clients.
3. Verify traffic uses the new key (check `last_used_at`).
4. Let the grace period expire, or revoke the old key early.

Keep grace windows short. Never log raw keys.

See [CLI API Keys](/cli/api-keys) for CLI rotation commands.

## OIDC Authentication

When your users authenticate through an identity provider (Clerk, Auth0, Okta), Strait can validate their tokens directly instead of routing through your backend.

This verifies already-issued bearer tokens on API requests. It does not provision hosted dashboard SSO/SAML or SCIM.

### When to Use

Use OIDC when user requests should carry their own IdP tokens. Backend services can continue using the server secret.

### Token Routing

Strait resolves bearer tokens in this order:

1. Token starts with `strait_`: API key auth path.
2. Other bearer token: OIDC verifier (if configured).
3. Server secret header: trusted backend auth path.

### Configuration

Turn on OIDC by configuring your server with:

- **Issuer**: your IdP's issuer URL (e.g., `https://clerk.example.com`)
- **Audience**: expected audience claim
- **Public key**: verification key settings

### Context Mapping

On successful verification, Strait maps token claims into:

- **Actor identity**: the authenticated user (used for audit and RBAC)
- **Project context**: when provided by claim or header mapping

Authorization still flows through RBAC policies and permission checks.

<Tip>
  Run OIDC and server-secret user identity in parallel during migration. Once user traffic carries OIDC tokens, stop sending actor headers for those requests.
</Tip>

## SDK Run Tokens

When Strait triggers a job, it issues a short-lived JWT scoped to that specific run.

- **Algorithm:** HS256, signed with `JWT_SIGNING_KEY`
- **Subject:** The run ID
- **Expiry:** 60 seconds (refreshed automatically by SDKs)

You do not need to manage these tokens. The SDK receives and uses them automatically when your job handler is called.

## RBAC

Role-based access control gives fine-grained permissions for users (as opposed to API keys, which use scopes).

1. **Roles** define a set of permissions (same scope strings as API keys).
2. **Members** link a user to a role within a project.
3. When a user makes a request, Strait loads their role and checks permissions.

Four built-in roles are available on Basic RBAC plans: **admin** (full access), **operator** (manage and trigger jobs/workflows), **viewer** (read-only), and **triggerer** (read and trigger only). Custom roles require Full RBAC.

For full details on managing roles, members, and custom permissions, see [RBAC & Policy Authorization](/guides/rbac).
