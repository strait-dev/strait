## Queue Mechanics

The orchestrator uses PostgreSQL as a message queue to minimize operational complexity and ensure transactional consistency.

- **SELECT FOR UPDATE SKIP LOCKED**: Allows multiple workers to poll the same table simultaneously without blocking.
- **Atomic Dequeue**: Uses a subquery to select a row and an outer `UPDATE` to transition status atomically.
- **Batch Dequeue**: Uses a CTE pattern to claim up to N rows in a single database round-trip.
- **Priority and Ordering**: Respects job priority (`priority DESC`) and FIFO (`created_at ASC`).

## Execution Lifecycle

<Steps>
  <Step title="Trigger">
    API receives a trigger request and checks cost budgets.
  </Step>
  <Step title="Enqueue">
    A `job_run` is created with `status=queued` and a JWT is generated.
  </Step>
  <Step title="Dequeue">
    A worker claims the run via `SKIP LOCKED` and updates status to `dequeued`.
  </Step>
  <Step title="Execute">
    The execution is submitted to the worker pool using `context.WithoutCancel`.
  </Step>
  <Step title="Dispatch">
    The executor sends an HTTP POST to the resolved endpoint URL.
  </Step>
  <Step title="Result Handling">
    Based on the response, the run becomes `completed`, `failed`, or is scheduled for `retry`.
  </Step>
</Steps>

## Workflow Engine

The workflow engine manages the execution of Directed Acyclic Graphs (DAGs).

- **Atomic Fan-In**: Uses a dependency counter on the step run to handle concurrent parent completions.
- **Payload Merging**: Merges trigger payload, static step payload, and parent outputs.
- **Template Rendering**: Supports `{{var}}` placeholders with dot-notation.
- **Output Transform**: Applies JSONPath extraction to step results before persisting.

## Design Decisions

1. **Postgres as Queue**: Chosen to avoid the operational overhead of an external broker.
2. **Optimistic Locking**: Prevents race conditions during state transitions.
3. **Raw SQL over ORM**: `pgx/v5` provides full control over performance-critical queries.
4. **Single Binary, Three Modes**: Simplifies deployment while allowing independent scaling.
5. **Micro-USD Cost Tracking**: Uses integer micro-USD to avoid floating-point precision issues.
6. **Dead Letter as FSM State**: DLQ is modeled as a first-class FSM state.
