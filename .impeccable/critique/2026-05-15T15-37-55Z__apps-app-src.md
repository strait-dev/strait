## Anti-Patterns Verdict

**LLM verdict: Leaning yes** on the template verdict, with meaningful mitigation.

Surface construction is nearly indistinguishable from a shadcn-base-nova starter. Every list page: `Shell` → search Input → DropdownMenu filter → DataTable. Repeats verbatim across jobs, runs, workflows, schedules, DLQ, webhooks. Detail pages: `Shell` → Card (Execution Overview) → Card (secondary) → Tabs. The dashboard is 8 evenly-spaced Card + recharts blocks, commented as "Row 1 ... Row 5." Metric cards are `CardHeader / title / icon / CardContent / number` without exception.

**Two confirming tells of slop lineage:**
1. `packages/ui/src/components/data-table.tsx` still has the comment `// Note: DataTableColumnHeader is removed as it needs Column instance, not ColumnDef` and the default empty state points to `/app/customers/add` — a non-existent route. Unmodified scaffold code.
2. `execution-trace-bar.tsx` uses raw Tailwind palette (`bg-blue-500`, `bg-indigo-500`, `bg-violet-500`, `bg-amber-500`, `bg-emerald-500`, `bg-cyan-500`) instead of the OKLCH tokens. The color system stops at the design-system boundary.

**Category-reflex first-order: FAIL.** Dark sidebar + neutral content + four stat cards + stacked charts + search-plus-filter list = exact mental model for a dev dashboard. Warm orange appears almost only as sidebar active accent.

**Category-reflex second-order: FAIL.** Response to "not blue-gradient SaaS" is "neutral dark with rounded cards." Chart palette is all blue (`oklch hue 251-265`); the one place color is used beyond neutral is the blue family. Brand orange sits unused.

**Deterministic detector:** 2 findings in app (`overused-font` + `single-font` from `__root.tsx:99`, Geist on Google Fonts), 4 findings in `packages/ui` (all `pure-black-white` from `bg-black` overlay scrims in alert-dialog/dialog/drawer/sheet — false positives, those are semi-transparent backdrops). Detector confirms: no rule violations of substance; problems are taste/composition.

## Cognitive Load

**5 of 8 fail** on dashboard + run detail:
1. Single primary action per screen ✗ (dashboard has none; run detail has Retry + Cancel equal weight)
2. ≤5–7 chunks in first viewport ✓ (4 stat cards + chart + cost; stat sparklines are empty arrays so they read as placeholder)
3. Status legible without decoding ✓
4. Nav says where you are ✗ ("Overview" vs "Dashboard" — sidebar shows both as top-level items)
5. Errors actionable ✗ (raw error text; no guidance beyond Retry)
6. Empty states instructive ✓
7. Actions weighted by consequence ✗ (Retry vs Cancel identical `variant="outline"`)
8. No competing routes ✗ (`/app` and `/app/dashboard`)

## Design System Audit

**Strait-ified vs vanilla in `packages/ui/src/components/` (~101 files):**
- Genuinely customized: ~25 (`button.tsx`, `badge.tsx` 15 variants, `tabs.tsx`, `sidebar.tsx`, toast wrappers, `input-with-addons.tsx`, `combobox.tsx`, `select-with-search.tsx`, `stepper.tsx`, `calendar-with-presets.tsx`, `multiselect.tsx`)
- Token-swapped vanilla: ~55–60 (`card.tsx`, `alert.tsx`, `dialog.tsx`, `sheet.tsx`, `table.tsx`, `accordion.tsx`, `checkbox.tsx`, `radio-group.tsx`, `separator.tsx`, `skeleton.tsx`, `progress.tsx`, `scroll-area.tsx`, `switch.tsx`, `tooltip.tsx`, `hover-card.tsx`, `popover.tsx`...)
- Utility/RAC wrappers: ~20

**~25% Strait-specific.** The other 75% is shadcn with neutrals.

**Cracks:**
- **Default radius:** `card.tsx:13` uses Tailwind's `rounded-xl` (0.75rem), ignoring the `--radius` token (0.45rem). Tokens defined but not consumed.
- **Default elevation:** No box shadows on cards. `ring-foreground/10` is the only depth signal. All cards on same Z-plane.
- **Default focus rings:** `--ring` in dark mode is `oklch(0.556 0 0)` — gray. Focus invisible against dark surfaces.
- **Default density:** Every CardHeader/CardContent is `px-4`. Every table cell `px-4 py-3`. No density variation across forms / tables / detail / billing — same rhythm everywhere.

**Color strategy: Restrained — by accident, not by choice.** Warm orange/red appears: sidebar active icon tint, primary buttons, chart-1..5 (which are blue family, unrelated). Semantic colors (success/warning/destructive/info) appear only in badges and banners. Never used as surface, accent block, or data encoding.

**Generic dashboard surfaces:**
- `dashboard/dashboard-content.tsx` — 5 rows of uniform grid-cols, indistinguishable from any Tremor/shadcn admin.
- `runs/$id.tsx:134-168` — "Execution Overview" Card with grid-cols-2 DetailCell rows. Reads as a settings form, not a run inspector.
- `billing/metrics-card.tsx` — Card > CardContent > p label + p value. Zero distinction from dashboard MetricsCard.
- `jobs/$id.tsx:524-533` — `StatCard` = Card > CardContent > text-center > text-2xl value + text-xs label. Textbook shadcn.
- Sidebar environment dots use `bg-green-500 / bg-blue-500 / bg-yellow-500` — raw Tailwind, ignoring tokens.

## What's Working

1. **StatusBadge** (`apps/app/src/components/dashboard/status-badge.tsx`) — dot + label driven by `RUN_STATUS_CONFIG`, multi-size, consistent throughout. Real engineering.
2. **DataTable floating bar** (`ui/data-table/data-table-floating-bar.tsx`) — contextual multi-select actions as a centered floating bar; appropriate for a control tool, consistently applied.
3. **Auth flow** (`(auth)/auth-layout.tsx`) — centered card, no decoration, no gradient hero, logo at top, title in tracking-tight normal weight. Spare and on-brand.

## Priority Issues

**[P0] Two overlapping "home" routes with duplicated metric cards.**
`/app` renders `OverviewMetrics` (8 stat cards, no charts) + "View dashboard" button. `/app/dashboard` renders `DashboardContent` (4 stat cards + 9 chart components). Sidebar lists both "Overview" and "Dashboard" as top-level items. First-session evaluators hit `/app`, see numbers, click "View dashboard," see more numbers + charts, conclude the product is unfinished.
Fix: `/app` redirects to `/app/dashboard` OR renders `DashboardContent` directly. Keep getting-started as the empty-state path. Eliminate `OverviewMetrics` or absorb its cards into the dashboard header row.
Command: `$impeccable distill`

**[P0] "Execution Overview" card on run detail is a generic settings form.**
`apps/app/src/routes/app/runs/$id.tsx:134-168` — Card with grid of DetailCell label/value pairs (Status, Duration, Started, Completed, Triggered By, Attempt, Job Version, Priority). This is the surface a developer uses at 2am to debug a failed run. Reads as configuration table, not a timeline/inspector.
Fix: Replace grid with a horizontal timeline strip (queue-wait / dispatch / execute phases). Use existing `ExecutionTraceBar` above the fold. Move 8 metadata fields to secondary "Details" or a collapsible. Make the error (if any) the dominant visual element, not a standard Alert below the header.
Command: `$impeccable layout`

**[P1] Color tokens defined but not consumed in product surfaces.**
`execution-trace-bar.tsx` uses raw `bg-blue-500/indigo-500/violet-500/amber-500/emerald-500/cyan-500`. Sidebar env dots use `bg-green-500/blue-500/yellow-500`. Banners in `upgrade.tsx` and billing use `bg-yellow-50/border-yellow-200/text-yellow-800`. Yellow banners break in dark mode without explicit `dark:` variants. Trace bar colors have no semantic meaning.
Fix: Map trace segments to chart tokens. Map env dots to `var(--success)`/`var(--info)`/`var(--warning)`. Convert all `yellow-50/200/800` banners to `bg-warning/10 border-warning/30 text-warning-foreground`.
Command: `$impeccable colorize`

**[P1] Dashboard = 5 uniform grid rows, no visual hierarchy.**
Primary charts (Run Activity, Status Distribution) carry the same visual weight as secondary charts (Top Jobs, Queue Health). All `CardTitle font-medium text-sm`. Eye has nowhere to land.
Fix: Rows 1–2 (live stats + run activity) get prominence: `text-base` titles, ruled-edge layout instead of card borders, main chart 3/4 width at all breakpoints. Rows 3–5 secondary register. `var(--primary)/5` tint on top stat row to anchor.
Command: `$impeccable layout`

**[P2] Identical list-page structure across six routes.**
Jobs, Runs, Workflows, Schedules, DLQ, Webhooks all use identical DOM structure, filter toolbar, DataTable, FloatingBar. Only column definitions vary. Efficient engineering, product reads as one template applied six times — the opposite of Linear's per-view opinionatedness.
Fix: Targeted differentiation, not rebuild: (a) Runs row → sparkline of recent failure pattern; (b) Schedules row → next-run as countdown not timestamp; (c) DLQ toolbar → prominent "Retry all" not behind multi-select. Each page needs ONE opinionated element.
Command: `$impeccable shape`

**[P2] Dark mode is structurally correct but perceptually flat.**
`--background: oklch(0.145 0 0)`, `--card: oklch(0.205 0 0)`. Delta of 0.06 L barely perceptible at typical brightness. Cards disappear into background. Ring border `ring-foreground/10` at `oklch(0.985 0 0 / 10%)` ≈ `oklch(0.2 0 0)` is invisible against the 0.205 card.
Fix: Card lightness to `oklch(0.22 0 0)` OR brand-tinted `oklch(0.21 0.005 37)`. `--border` dark to `oklch(1 0 0 / 14%)`. Add `box-shadow: 0 1px 3px oklch(0 0 0 / 0.3)` on Card.
Command: `$impeccable harden`

**[P3] Log viewer is a raw `<pre>` dump.**
`runs/$id.tsx:191-199` — `<pre>` with `max-h-[500px]` `whitespace-pre-wrap`, lines like `[ISO_TIMESTAMP] [LEVEL] message`. No filter, no search, no copy-all, no line count, no timestamp localization, no highlighting.
Fix: Parse `[LEVEL]` and apply `text-destructive` to ERROR/WARN. Copy-all button in card header. Level filter chip group. Cap at 1000 lines with "show all."
Command: `$impeccable polish`

**[P3] `data-table.tsx` contains dead scaffold code.**
Line 33: `href="/app/customers/add"` — non-existent route. Line 215: dead comment about removed `DataTableColumnHeader`. `DataTableRowActions` uses `<DropdownMenuLabel>Actions</DropdownMenuLabel>` as non-interactive header inside a menu — a11y anti-pattern. Library wasn't audited after fork.
Fix: Remove dead default empty state. Remove dead comment. Restructure Actions label or use proper MenuSeparator.
Command: `$impeccable audit`

## Persona Red Flags

**The New SRE who just signed up:** First route `/app`. 8 zero number cards + "View dashboard." Clicks through to `/app/dashboard`. 4 more zero cards + 9 chart containers with "No run activity yet." Returns to sidebar — "Overview" + "Dashboard" both there, doesn't know which is authoritative. Clicks "Jobs" — empty table "No jobs yet." Entire first session is zeros and empty states with no guidance. Getting-started only renders when `hasProject` is false; once a project exists it disappears and is unreachable. No persistent quick-start rail.

**The Power User dropped into run detail at 2am:** Arrives at `/app/runs/{id}` via Slack alert. Page shows: monospace run ID, status badge, 8 label-value pairs in a grid card, ExecutionTraceBar below (if present), Logs tab with raw pre block. To find error: (1) read Alert banner at top if status failed and error set, (2) switch to Logs tab, (3) scan pre for ERROR lines. If error is in response body: click Response tab, find pre, read raw JSON. No fast-path "here is why it failed, here is the line." Surface deposits raw data, expects user to parse.

**The Free-tier evaluator on `/upgrade`:** PlanSelection is 4-col card grid. Selected state = `border-2`, unselected = `border-foreground/20`. Visual distinction is border weight. Feature list truncates at 8 with "+N more features" — no expand affordance. Comparison table below is full-bleed HTML with no sticky header. Mobile = horizontal scroll, no overflow indicator. Checkout error banners use `bg-yellow-50/border-yellow-200` and `bg-red-50/border-red-200` raw Tailwind — invisible in dark mode (red-50 against dark card).

## Minor Observations

- `rounded-custom` appears in several components (`auth-layout.tsx:11`, `table-empty-state.tsx:26`, `plan-selection.tsx:239`) but is never defined in `globals.css` or tailwind config. Likely resolves to 0 or default.
- `layout.tsx` header at `h-16 bg-background` — same color as page, header/page boundary invisible in dark mode.
- `ThemeToggle` is `hidden sm:inline-flex` — forces mobile users into light mode with no toggle.
- Sidebar environment switcher is local `useState` only — switching "Production" → "Staging" changes the label, nothing in the data layer. Misleading.
- `jobs/$id.tsx:505-518` "Job Settings" tab is a Card with "Job configuration management coming soon." — should not ship as a visible tab.
- `StatCard` in `jobs/$id.tsx:524` duplicates `MetricsCard` from `dashboard/metrics-card.tsx`. Two implementations of the same pattern in one codebase.
- `GettingStarted` CLI install line is `go install ./cmd/strait` — internal dev command, not a user-facing install path.

## Questions to Consider

1. If `/app` (Overview) and `/app/dashboard` serve different purposes, what is that purpose? If they don't, which one survives?
2. The warm orange primary is correctly tokenized but appears only as sidebar active accent. Intent for it to remain a chrome accent only? If so, what is the product's color story?
3. ExecutionTraceBar is the one genuinely domain-specific visualization. Why is it below the fold, inside a conditional Card, while 8 generic label-value pairs sit above the fold?
4. Environment switcher (prod/staging/dev) changes label only. Planned feature in progress, or UI element to remove until functional?
5. The plan comparison table has no sticky headers, no mobile adaptation, plan names duplicate the card titles above. Has this been tested with an actual procurement decision-maker?
