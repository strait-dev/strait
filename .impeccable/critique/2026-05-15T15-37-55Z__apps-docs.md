## Anti-Patterns Verdict

**Verdict: Not default Mintlify, but not yet Linear/Vercel/Resend either. Disciplined vanilla.**

Docs avoid the most obvious Mintlify tells: no gratuitous icon-with-heading grids, no gradient hero blocks, no "Get Started" card spam, no exclamation marks. Content is engineer-written, not marketing-stakeholder-written.

The problem is structural blandness, not slop. Every page follows the same rhythm: one-sentence intro paragraph, code block, table, "Related" or "Next" CardGroup. No page has a moment that makes a first-time reader stop and think "this team knows their product cold." Component palette is narrow: Cards, Tables, Code blocks, occasional Warning/Tip. No Accordions for progressive disclosure, no Tabs for SDK-switching inside concepts (only in SDK overview), no Frames for illustrating data flow.

The "too shadcn" problem maps less here than to the app, but docs inherit it by being aesthetically neutral. Nothing repels, nothing attracts.

## docs.json Audit

- **Primary color**: `#FF4F00` across light/dark/primary — brand-aligned. Mintlify default is `#0D9373`; this is a deliberate override.
- **Theme**: `"luma"` — non-default, cleaner typography than legacy.
- **Dark mode**: `"appearance": { "default": "dark" }` ✓
- **Fonts**: NOT CONFIGURED. No `"font"` block. Mintlify falls back to its built-in Inter. PRODUCT.md specifies Geist + Geist Mono. **This is one concrete reason the docs feel like a hosted template rather than a branded property.**
- **Logo/favicon**: Custom SVGs ✓
- **Anchors/CTA buttons in navbar**: Not configured. Linear and Vercel docs prominently have these.
- **`twitter:site`**: `@leonardomso` (personal). Minor brand leak.
- **Sidebar duplication**: SDK quickstart pages (`sdks/typescript-quickstart`, `sdks/python-quickstart`, `sdks/go-quickstart`) appear in both the Documentation tab under `Guides` AND the SDKs tab under `Start Here`.

## Information Architecture

**Works:** Getting Started first, Use Cases second (problem-first framing), Compare present (valuable for evaluators), Advanced collects deep material correctly.

**Doesn't work:**
1. **Guides before Core Concepts.** Reader who skips quickstart and hits `guides/authentication` sees `dead_letter`, `run`, `workflow step` with no definition page visible above.
2. **SDK quickstarts duplicated.** Land between deployment and tutorials in the Guides group with no clear "why SDK quickstart is a guide."
3. **"Choose Your Path" placement.** 51-line page sits between Introduction and Quickstart. Adds a navigation click for prerequisite content that could be a callout inside quickstart.
4. **Architecture buried in Advanced** — ninth item under Durable Workflows, Compensating Transactions, Adaptive Concurrency, Canary Deployments, Cost Budgets, SLOs, Debug Bundles. An evaluator doing a 5-minute skim won't find it. Architecture is early-funnel, not advanced.
5. **Integrations is a one-page stub** — Postgres, Redis, OpenTelemetry, Grafana, ClickHouse, Sentry, Datadog, Splunk listed; only ClickHouse and log drains have hrefs. The rest are inert cards. A broken promise at the end of the sidebar.

Cross-link health is generally good within pages. Onboarding path Introduction → choose-your-path → quickstart is lean but breaks for Cloud users: quickstart still lists Docker/build steps with a prose note to "skip if Cloud."

## Voice Audit

**Strengths:** No exclamation marks. No marketing fluff. No emojis. Direct, imperative, calm.

**Issues:**
- `architecture.mdx:54-58` — `--` used as em dash 6 times: "HTTP jobs -- Strait sends a POST...", "Worker jobs -- your worker keeps a connection open...", etc. PRODUCT.md explicitly bans em dashes; `--` is semantically the same. Same pattern in `tutorials/webhook-processor.mdx:109`.
- `tutorials/webhook-processor.mdx:9` uses contraction "What you'll build"; `quickstart.mdx:14` uses "What You Will Do." Pick one.
- `api-reference/overview.mdx:9` Info block reads as a placeholder note, not authored content.
- `architecture.mdx` slight self-repetition with intro on execution modes.

## What's Working

1. **Voice discipline across 16 sampled files.** Consistent technical register; no exclamation marks; no marketing superlatives. `compare/message-queues.mdx` is the standout — calibrated decision framework without disparaging competitors.
2. **Quickstart completeness.** `quickstart.mdx` delivers a working first run in 7 numbered steps using only `curl`. Every step has visible output. Common Problems addresses the three real friction points (queued, timeout, private endpoint). Vercel/Resend quality.
3. **Tutorial depth.** `tutorials/webhook-processor.mdx` is the strongest content page — realistic end-to-end Stripe webhook scenario with full code across 5 steps, including the non-obvious concern (verifying Strait's completion webhook).

## Priority Issues

**[P0] Geist font is not configured.**
`docs.json` has no `font` block. PRODUCT.md spec calls for Geist + Geist Mono (Linear/Vercel/Resend defaults). Without it, every page renders in Inter fallback — the single most visible signal that docs are "default Mintlify."
Fix: Add to `docs.json`:
```json
"font": {
  "headings": { "family": "Geist" },
  "body": { "family": "Geist" },
  "code": { "family": "Geist Mono" }
}
```
Command: `$impeccable typeset`

**[P1] `--` used as em dash throughout prose.**
PRODUCT.md bans em dashes. The `--` pattern is semantically identical. Appears in at least `architecture.mdx` (6 instances) and `tutorials/webhook-processor.mdx`. Looks like a keyboard shortcut, not a deliberate choice.
Fix: Replace `--` em-dash uses with comma, period, or colon. `"HTTP jobs -- Strait sends a POST..."` → `"HTTP jobs: Strait sends a POST..."`. `"Return 200 immediately -- Strait owns the retries from here."` → `"Return 200 immediately. Strait owns the retries from here."`
Command: `$impeccable audit`

**[P1] Architecture buried in Advanced.**
Ninth item under Advanced. The most important trust-building doc for a technical evaluator should appear before or inside Getting Started.
Fix: Move `architecture` to the end of Getting Started, or create a "Platform" group with `[introduction, architecture, choose-your-path]` followed by `quickstart`. Remove from Advanced.
Command: `$impeccable layout`

**[P1] Quickstart assumes self-host; Cloud users are told to "skip" steps they still see.**
`quickstart.mdx:10` — "If you use Strait Cloud, use `https://api.strait.dev`... skip local setup steps." Steps 1–3 (clone, Docker Compose, build) still listed in full. First-time Cloud users read 3 unnecessary steps before being told they can start. Breaks the "one completed run" promise.
Fix: `<Tabs>` block at top splitting Cloud and Self-host paths. Cloud path starts at "Create a Project" with the Cloud API URL pre-filled.
Command: `$impeccable onboard`

**[P2] SDK quickstarts duplicated across two sidebar locations.**
Same three pages under Documentation > Guides and SDKs > Start Here.
Fix: Remove from Guides. Add single `sdks/overview` entry at end of Guides labeled "SDK Quickstarts."
Command: `$impeccable layout`

**[P2] Integrations cards have no hrefs.**
`integrations/index.mdx` — PostgreSQL, Redis, OpenTelemetry, Grafana Cloud cards have no `href`. Inert tiles. Reader clicks Integrations expecting setup, finds dead ends.
Fix: Either link each card to a real page or replace inert cards with a prose list. Do not ship Cards without destinations.
Command: `$impeccable harden`

**[P2] "Choose Your Path" adds a step without information density.**
51-line page; code snippet for Cloud previews quickstart without completing it; self-host section gives different instructions than the quickstart. Either duplicate or stub.
Fix: Fold Cloud/self-host decision into the quickstart's tab structure. Remove standalone page or expand into a real decision guide.
Command: `$impeccable distill`

**[P3] No anchor/CTA buttons in navbar.**
Mintlify supports `"anchors"` for CTAs. Linear and Vercel docs have them prominently. Strait nav has only `Website` and `Dashboard` plain links.
Fix: Add `"anchors": [{ "name": "GitHub", "icon": "github", "url": "https://github.com/strait-dev/strait" }, { "name": "Start for free", "url": "https://app.strait.dev", "icon": "rocket" }]`.
Command: `$impeccable polish`

## Persona Red Flags

**Power Evaluator (5-min skim):** Sidebar — Introduction, Choose Your Path, Quick Start, 5 Use Cases, 8 Guides, then Core Concepts. Clicks Authentication hoping to understand auth model — finds full guide with 4 methods and no summary of which one an evaluator picks. Won't find Architecture without scrolling past 11 Advanced items. After 5 minutes knows Strait works with Postgres + Redis but has no clear mental model relative to Temporal, Inngest, or bare SQS. Compare section exists but requires clicking — not surfaced in flow.

**New Implementer (signed up, building first job):** Follows quickstart on Cloud. Reads `choose-your-path` — gets `client.triggerJob()` snippet. Clicks through to quickstart — first thing is `git clone` + `docker compose up` which they don't need. Note "skip the local setup steps" is buried in prose, not a callout. Scrolls past Docker steps, tries step 4 with `http://localhost:8080` (what quickstart says), fails with connection refused, re-reads, swaps to `https://api.strait.dev`, tries again. Quickstart succeeds; "Next" card → "Create a Production Job" which starts with "Create an Endpoint" — they have no live URL. Guide doesn't bridge the gap.

## Minor Observations

- `introduction.mdx:36-49` and `quickstart.mdx:124-137` both end with a 4-card CardGroup with 3 overlapping cards. Repetition weakens both. Quickstart footer should advance the reader, not reiterate the starting options.
- `api-reference/overview.mdx` has two error envelope sections (line 33-35 and line 123-155). First is redundant.
- `guides/authentication.mdx` uses horizontal rules `---` between sections — unusual; not in other guides. Adopt site-wide or remove.
- `tutorials/webhook-processor.mdx:8` "What you'll build" vs `quickstart.mdx:14` "What You Will Do" — pick one.
- `Monitoring` sidebar group mixes monitoring (observability, log-drains, ClickHouse, notifications, audit-logging) with outbound webhooks/webhook-subscriptions/real-time. Outbound webhooks aren't monitoring.
- `background-jobs.mdx` is 48 lines — placeholder, not a real scenario.
- `changelog` under Advanced rather than its own tab/anchor. Changelogs are trust signals for actively-evaluated products.

## Questions to Consider

1. Is `choose-your-path` worth the friction it creates? Could the decision live as a tab inside quickstart?
2. What should Use Cases pages actually do? Currently 50-line stubs. For an evaluator, use cases are where credibility is built. Should they include a realistic before/after?
3. How should docs handle the Community/Cloud edition split? Currently no docs marks any feature as Cloud-only.
4. Should Getting Started end with a "next steps" page summarizing what you just did and where to go next, or stay as a CardGroup?
5. Is the font omission a tradeoff or a blocker? If Mintlify's `luma` theme can't serve Geist, brand alignment needs to come from somewhere else (color, code-first layouts, diagrams).
